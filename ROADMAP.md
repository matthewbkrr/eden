# ROADMAP — eden

Приватный мессенджер на Elixir/Phoenix. Текст + фото, realtime через Phoenix
Channels / LiveView, PostgreSQL. Аудитория на старте ~20 человек, проект
с прицелом на расширение.

> Этот файл — план для кодового агента (Claude Code). Работай строго по фазам,
> сверху вниз. Не перескакивай вперёд и не реализуй фичи будущих фаз заранее.

---

## Контекст и ключевые решения (приняты, не пересматривать без спроса)

- **Имя проекта:** `eden` (OTP-app `:eden`, корневой модуль `Eden`).
- **Хостинг:** VPS за рубежом, аудитория преимущественно из РФ. Следствие:
  трансграничное WebSocket-соединение — узкое место по надёжности (см. Фазу 5).
- **Клиент:** веб (LiveView). Десктоп-обёртка (Tauri) — за рамками MVP, не делать.
- **Аутентификация:** по **invite-ссылкам**, без email-верификации.
  Это осознанный отказ от SMTP: доставка писем с зарубежного VPS на
  российские ящики ненадёжна. Никакого mailer в проекте быть не должно.
- **Группы с первого дня:** диалог 1:1 — это частный случай беседы на двух
  участников. Схемы строятся вокруг `Conversation` / `Membership` / `Message`.
- **Точки абстракции (порты/адаптеры)** только там, где переезд реально вероятен:
  **хранилище файлов** (локально → S3-совместимое) и **аутентификация**.
  Остальное — без преждевременных абстракций.
- **Очереди:** Oban подключается с самого начала (фоновые задачи появятся быстро).
- **Наблюдаемость:** структурное логирование + Telemetry с старта.
- **Роутинг:** через домен, не голый IP.

## Фаза 0 — Фундамент и инженерная обвязка
**Цель:** пустой, но «промышленно одетый» каркас. Веха: `mix phx.server`
поднимается, `mix check` и CI зелёные.

- [ ] `mix phx.new eden --database postgres` (без mailer). Каркас компилируется,
      `mix test` проходит.
- [ ] Dev/test-зависимости в `mix.exs`: `credo`, `sobelow`, `dialyxir`,
      `mix_audit`. Алиас `mix check` гоняет последовательно:
      `format --check-formatted`, `compile --warnings-as-errors`,
      `credo --strict`, `sobelow --config`, `deps.audit`, `test`.
- [ ] Конфиги инструментов: `.credo.exs` (strict), `.sobelow-conf`,
      `.dialyzer_ignore.exs` (пустой задел).
- [ ] `.github/workflows/ci.yml`: джобы (параллельно где можно) format-check,
      compile-warnings-as-errors, credo, sobelow, deps-audit, dialyzer (кеш PLT),
      test (с сервисом postgres). `actions/cache` для deps, `_build`, PLT.
      Плюс gitleaks (secret scan) и dependency-review на pull_request.
- [ ] Шаблоны: `.github/ISSUE_TEMPLATE/feature.yml`,
      `.github/ISSUE_TEMPLATE/bug.yml`, `.github/PULL_REQUEST_TEMPLATE.md`
      (что меняется, как тестировалось, миграции да/нет, влияние на
      безопасность, связанный issue, чеклист).
- [ ] `CLAUDE.md`: стек и версии; архитектура (контексты Accounts / Chat /
      Storage; беседы как первоклассная сущность; адаптер хранилища);
      команды (`mix setup`, `mix test`, `mix check`); Definition of Done;
      рубрика код-ревью P0–P3.
- [ ] `docker-compose.yml` для локалки (Postgres).
- [ ] Oban + базовая Telemetry-обвязка и структурный логгер.

**Done:** сервер стартует, CI зелёный, агент-ревью настроено на будущие PR.

## Фаза 1 — Аутентификация по invite-ссылкам
**Цель:** закрытый вход без email. Веха: по одноразовой ссылке создаётся
аккаунт и сессия; без валидной ссылки регистрация невозможна.

- [ ] Контекст `Accounts`: схема `User` (без email-верификации; логин/пароль
      или passwordless — на твой выбор, но без mailer). С первого дня — поле
      `display_name` (никнейм): это отображаемая идентичность в чатах, нужна
      сразу. Описание (bio) и аватар — НЕ здесь (см. бэклог): аватар ложится
      поверх адаптера хранилища (Фаза 3), bio — дешёвая доп. колонка позже.
- [ ] Схема `Invite`: токен, кто пригласил, срок жизни, лимит использований,
      статус. Генерация криптостойкого токена.
- [ ] LiveView/контроллер: страница приглашения `/invite/:token` → создание
      аккаунта → сессия. Невалидный/просроченный/исчерпанный токен — отказ.
- [ ] Сессии и базовый `on_mount` для защищённых LiveView.
- [ ] **Адаптер аутентификации** как поведение (behaviour): текущая реализация
      локальная, но интерфейс позволяет заменить позже.
- [ ] Тесты: валидная ссылка, просрочка, повторное использование, лимит.

**Done:** новый пользователь заходит только по приглашению, сессия живёт.

## Фаза 2 — Ядро: текстовые сообщения в реальном времени
**Цель:** двое (или больше) переписываются текстом, история сохраняется.

- [ ] Контекст `Chat`. Схемы `Conversation`, `Membership`, `Message`
      (группы поддержаны конструктивно с первого дня).
- [ ] **Шов идентичности (заложить сразу):** `Message` ссылается на автора по
      связи (`belongs_to :sender`), а имя/аватар рисуются из записи `User`
      через preload — НИКОГДА не копировать имя/аватар в строку сообщения.
      Тогда профиль (никнейм, аватар, bio) и его изменения добавляются потом
      без боли. (Состояние прочтения уже заложено как `last_read` в `Membership`.)
- [ ] `Phoenix.PubSub` + Channel/LiveView: список бесед, окно чата,
      отправка/приём в реальном времени.
- [ ] История из БД с пагинацией (подгрузка старых сообщений).
- [ ] `Phoenix.Presence`: онлайн/офлайн участников.
- [ ] Валидация и нормализация ввода (changeset'ы), защита от пустых/огромных
      сообщений.
- [ ] Тесты контекста и LiveView (включая realtime-доставку).

**Done:** реальное время работает, история переживает перезагрузку.

## Фаза 3 — Обмен фотографиями
**Цель:** отправка изображений в беседу. Веха: фото грузится, отдаётся
участникам, переживает рестарт.

- [ ] **Адаптер хранилища** (behaviour `Eden.Storage`): реализации `Local`
      (для dev) и `S3` (совместимое). Переключение — через конфиг.
- [ ] Загрузка через LiveView uploads: ограничение типов и размера,
      генерация превью/миниатюр (через Oban-задачу).
- [ ] Схема вложения, связь с `Message`. Отдача файлов с правильными заголовками.
- [ ] Базовая защита: проверка MIME, лимиты, безопасные имена объектов.
- [ ] Тесты: загрузка, ограничения, отдача, фоновая обработка превью.

**Done:** фото ходят в беседах, хранилище меняется одной строкой конфига.

## Фаза 4 — Надёжность соединения (критично для трансграничного канала)
**Цель:** соединение РФ↔зарубежный VPS не теряет сообщения при обрывах.
Это отдельная фаза именно из-за хостинга за границей — не пропускать.

- [ ] Переотправка/идемпотентность сообщений: клиентский temp-id + серверная
      дедупликация, чтобы повтор после реконнекта не двоил сообщения.
- [ ] Корректное восстановление LiveView/Channel после разрыва: догрузка
      пропущенного, индикация «переподключение».
- [ ] Очередь исходящих на клиенте на время оффлайна.
- [ ] Нагрузочная/хаос-проверка: рвём соединение, проверяем целостность истории.

**Done:** при обрывах нет потерь и дублей, UX деградирует мягко.
