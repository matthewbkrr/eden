# ROADMAP — eden

Приватный мессенджер на Elixir/Phoenix. Текст + фото, realtime через Phoenix
Channels / LiveView, PostgreSQL. Аудитория на старте ~20 человек, проект
с прицелом на расширение.

> Этот файл — план для кодового агента (Claude Code). Работай строго по фазам,
> сверху вниз. Не перескакивай вперёд и не реализуй фичи будущих фаз заранее.

---

## Контекст и ключевые решения (приняты, не пересматривать без спроса)

- **Имя проекта:** `eden` (OTP-app `:eden`, корневой модуль `Eden`).
- **Хостинг:** VPS за рубежом, аудитория преимущественно из РФ. Следствие:
  трансграничное WebSocket-соединение — узкое место по надёжности (см. Фазу 5).
- **Клиент:** веб (LiveView). Десктоп-обёртка (Tauri) — за рамками MVP, не делать.
- **Аутентификация:** по **invite-ссылкам**, без email-верификации.
  Это осознанный отказ от SMTP: доставка писем с зарубежного VPS на
  российские ящики ненадёжна. Никакого mailer в проекте быть не должно.
- **Группы с первого дня:** диалог 1:1 — это частный случай беседы на двух
  участников. Схемы строятся вокруг `Conversation` / `Membership` / `Message`.
- **Точки абстракции (порты/адаптеры)** только там, где переезд реально вероятен:
  **хранилище файлов** (локально → S3-совместимое) и **аутентификация**.
  Остальное — без преждевременных абстракций.
- **Очереди:** Oban подключается с самого начала (фоновые задачи появятся быстро).
- **Наблюдаемость:** структурное логирование + Telemetry с старта.
- **Роутинг:** через домен, не голый IP.

## Фаза 1 — Аутентификация по invite-ссылкам
**Цель:** закрытый вход без email. Веха: по одноразовой ссылке создаётся
аккаунт и сессия; без валидной ссылки регистрация невозможна.

- [x] Контекст `Accounts`: схема `User` (без email-верификации; **логин/пароль**,
      bcrypt). С первого дня — поле `display_name` (никнейм) + `username`
      (уникальный логин). Описание (bio) и аватар — НЕ здесь (см. бэклог).
- [x] Схема `Invite`: хранится **хеш** токена (сырой только в URL); срок жизни,
      лимит использований, отзыв (`revoked_at`); валидность выводится, без
      мутабельного статуса. Приём — транзакция с `FOR UPDATE` на инвайте.
- [x] LiveView/контроллер: `/invite/:token` (InviteLive + InviteController) →
      создание аккаунта → сессия. Невалидный/просроченный/исчерпанный токен — отказ.
- [x] Сессии (ревокабельные токены в БД) и `on_mount` (`current_scope`,
      Phoenix 1.8 Scope) для защищённых LiveView; `/login`, logout, `/app`.
      Bootstrap первого инвайта — `mix eden.invite`.
- [ ] **Адаптер аутентификации** как поведение (behaviour). **Отложено**
      (YAGNI): пока один способ входа. Контекст `Accounts` уже изолирует логику —
      ввести behaviour легко, когда появится 2-й метод (OAuth/passkey).
- [x] Тесты: валидная ссылка, просрочка, повторное использование, лимит,
      дубликат username, вход/выход, защита маршрутов.

**Done (по сути):** новый пользователь заходит только по приглашению, сессия
живёт. Открытый пункт — auth-адаптер (отложен).

## Фаза 2 — Ядро: текстовые сообщения в реальном времени
**Цель:** двое (или больше) переписываются текстом, история сохраняется.

- [ ] Контекст `Chat`. Схемы `Conversation`, `Membership`, `Message`
      (группы поддержаны конструктивно с первого дня).
- [ ] **Шов идентичности (заложить сразу):** `Message` ссылается на автора по
      связи (`belongs_to :sender`), а имя/аватар рисуются из записи `User`
      через preload — НИКОГДА не копировать имя/аватар в строку сообщения.
      Тогда профиль (никнейм, аватар, bio) и его изменения добавляются потом
      без боли. (Состояние прочтения уже заложено как `last_read` в `Membership`.)
- [ ] `Phoenix.PubSub` + Channel/LiveView: список бесед, окно чата,
      отправка/приём в реальном времени.
- [ ] История из БД с пагинацией (подгрузка старых сообщений).
- [ ] `Phoenix.Presence`: онлайн/офлайн участников.
- [ ] Валидация и нормализация ввода (changeset'ы), защита от пустых/огромных
      сообщений.
- [ ] Тесты контекста и LiveView (включая realtime-доставку).

**Done:** реальное время работает, история переживает перезагрузку.
