1. Обзор системы

ihichat — корпоративный realtime-мессенджер (текст, фото, видео, файлы), разработанный и эксплуатируемый компанией самостоятельно. Пользовательский бренд — «ihichat»; кодовая база, модули и репозиторий носят имя eden.

Ключевые проектные решения:

  • Веб-приложение без клиентов. Единственный клиент — браузер (десктоп и мобильный). UI обновляется в реальном времени по WebSocket (Phoenix LiveView), без REST/JSON-API и без отдельного фронтенд-фреймворка.
  • Закрытый периметр. Регистрация только по одноразовым ссылкам-приглашениям. Открытой самостоятельной регистрации нет — это осознанное продуктовое и юридическое решение (см. ADR-0002): система остаётся в рамках существующей позиции компании как оператора ПДн.
  • Никакой электронной почты. В системе нет почтовой подсистемы вообще (доставка с зарубежного VPS в российские ящики ненадёжна). Все «почтовые» сценарии — приглашения, сброс пароля — реализованы одноразовыми ссылками, которые администратор передаёт лично.
  • Один сервер. Прод — один VPS: Docker Compose (Caddy + приложение + PostgreSQL). Основное сетевое ограничение — трансграничный RTT ~160 мс, а не ресурсы сервера.
  • Два слоя общения. Личный мессенджер (диалоги и группы, пузырьковый UI в духе Telegram) и корпоративный слой (каналы → комнаты → треды, плоский UI в духе Mattermost) — на одной и той же механике сообщений.

3. Архитектура приложения

Ограниченные контексты

Домен разбит на контексты в lib/eden/; веб-слой — lib/eden_web/. Инварианты зависимостей: веб-слой вызывает контексты, контексты никогда не вызывают веб-слой и не лезут друг в друга напрямую (сквозные сценарии оркестрируются задачей уровня приложения, например воркером стирания данных).

КонтекстОтветственность
Eden.AccountsПользователи, сессии, приглашения, пароли и их сброс, TOTP-2FA, платформенные роли (member/admin/super_admin), управляемые «кадровые» поля, деактивация и необратимое удаление (анонимизация).
Eden.ChatВся механика сообщений: диалоги/группы (Conversation + Membership), сообщения с вложениями-альбомами, ответы-цитаты, реакции, редактирование, удаление («у себя»/«у всех»), пересылка, треды комнат, папки, непрочитанные, поиск (trigram), системные сообщения.
Eden.ChannelsКорпоративный слой: каналы, роли участников (owner/admin/member), комнаты (это Conversation с channel_id), доступ в приватные комнаты («постучаться»), ссылки-приглашения в канал, передача владения.
Eden.StorageФайлы за интерфейсом-поведением (behaviour): ключ + метаданные у вызывающего, реализация — Local (атомарная запись) или S3 (ручной SigV4, range-запросы для перемотки видео). Смена — одна строка конфигурации.
Eden.NotificationsДоставка уведомлений за интерфейсом-адаптером (ADR-0001). «Кому» решает Chat (mute/DND/фокус/подписки на треды/деактивация — в одном месте), адаптеры только доставляют. Транспорты: Web (внутривкладочный, всегда) + мобильные пуши APNs/FCM — включаются переменными окружения; HTTP к Apple/Google уходит из фоновой Oban-очереди :push, не из пути отправки сообщения. Устройства регистрируются в notification_targets через POST /devices по обычной cookie-сессии.

Realtime

Каждое открытое LiveView подписано на PubSub-топики: беседа (conversation:<id>), пользовательские топики (user:<id>:chat — синхронизация сайдбара, user:<id>:notify — уведомления, отдельно, чтобы не флудить), канал (channel:<id>). Отправка сообщения широковещает событие, и все клиенты обновляются без опроса. Оптимистичная отправка: сообщение появляется мгновенно, статус «отправляется/отправлено» честный, при обрыве связи очередь досылается после переподключения.

Фоновая обработка

  • :media — миниатюры изображений, постеры и длительность видео (ffmpeg). Габариты в пикселях читаются синхронно при отправке, чтобы вёрстка не «прыгала».
  • :default — durable-воркер стирания следов удалённого пользователя (транзакционный outbox: задача ставится внутри транзакции анонимизации).
  • Cron — ежедневная очистка истёкших токенов.

4. Модель данных

Основные таблицы (все — PostgreSQL, миграции в priv/repo/migrations):

ТаблицаНазначение
usersАккаунты: логин-«тег» (@username, переименовываемый), профиль, аватар (avatar_key), платформенная роль, presence-статус, TOTP-поля (секрет зашифрован), active, deleted_at, управляемые поля (corp_email, должность и т.д.).

message_deletions, message_reactionsПерсональные скрытия сообщений; реакции (уникальность «участник+эмодзи», закрытый набор).
chat_folders, chat_folder_memberships, chat_folder_prefsЛичные папки-вкладки, членство чатов в папках, персональные предпочтения (позиция «Все чаты», быстрые реакции, настройки уведомлений).
channels, channel_memberships, channel_invitesКаналы, роли и «последняя комната», приглашения в канал/комнату.

oban_jobsОчередь фоновых задач (Oban).

Заметные решения: удаление пользователя — анонимизация, а не каскад (строка остаётся, переписка коллег не «дырявится», подпись становится «Удалённый аккаунт»); блобы вложений разделяются при пересылке и убираются только когда не осталось ссылок; поиск по телу сообщений — триграммный индекс GIN (подстрока + устойчивость к опечаткам).

5. Аутентификация и безопасность

Вход в периметр

  • Регистрация только по приглашению. Токены приглашений хранятся хешированными, имеют срок (по умолчанию 30 минут) и лимит использований; погашение транзакционно.
  • Пароли — bcrypt. Смена пароля отзывает все сессии. «Выйти везде» доступно пользователю.
  • Сброс пароля — только через администратора: одноразовая короткоживущая ссылка (та же токен-механика), выдаётся из /admin. Почты в системе нет.
  • TOTP-2FA (RFC 6238): пользователь включает сам (QR + резервные коды), при входе — отдельный второй шаг до выдачи сессии; защита от повтора кода. Секреты шифруются AES-256-GCM (Eden.Vault, ключ EDEN_VAULT_KEY — отдельный секрет, не в БД). Для администраторов 2FA обязательна: без неё админ-панель недоступна. Восстановление при утере устройства — сброс фактора администратором.

Роли и авторизация

  • Платформенные роли: member → admin → super_admin. Назначает только super_admin; последнего работоспособного super_admin понизить/удалить нельзя (блокировка FOR UPDATE) — админ-доступ не может быть потерян.
  • Роли канала: owner / admin / member — отдельная ось, матрица «кто кого может удалить» строгая, владелец уходит только передав канал.
  • Все функции контекстов принимают %Scope{} (текущий пользователь) и сами проверяют доступ: не-участнику возвращается :not_found — существование чужих сущностей не раскрывается. Авторизация живёт в контекстах, а не в шаблонах.
  • Деактивация (обратима) и необратимое удаление-анонимизация: обе мгновенно отзывают сессии; вход деактивированного неотличим от неверного пароля (состояние аккаунта не утекает).

Защита веб-слоя

  • Rate limiting по IP на анонимных POST: вход — 10/5 мин, приглашение — 30/5 мин (реальный IP восстанавливается из X-Forwarded-For, который перезаписывает Caddy). Неудачные входы логируются (логин + IP, без пароля).
  • CSP с per-request nonce (скрипты только с nonce); CSRF-защита Phoenix; безопасные заголовки браузера.
  • Контент. Тип вложения определяется по магическим байтам, а не по заявленному Content-Type; лимиты: фото 8 МБ, видео 50 МБ, файл 25 МБ, до 10 в альбоме; защита от «декомпрессионных бомб». Markdown рендерится по whitelist-подмножеству (HTML-инъекции исключены на уровне генерации).
  • Раздача файлов — только через контроллер с проверкой членства в беседе; прямых публичных URL к блобам нет.

6. Администрирование (панель /admin)

Панель https://chat.ihi.ru/admin доступна ролям admin/super_admin (обязательно с включённой 2FA). Ссылка на неё видна в настройках только администраторам.

7. Развёртывание

Топология прода

Один хост (Debian 12) с Docker Compose (/opt/eden): контейнеры caddy (реверс-прокси, автоматический TLS Let’s Encrypt), app (OTP-релиз в многостадийном образе), db (PostgreSQL 16). Медиа — локальный volume uploads; переключение на S3/R2 — переменными окружения, без изменения кода.

браузер → Caddy (:80/:443, TLS) → app (:4000) → PostgreSQL
                                      └── медиа: volume uploads | S3/R2

CD

GitHub Actions, workflow Deploy (ручной запуск): сборка образа → публикация в GHCR → по SSH на сервер: docker compose pull, миграции bin/migrate, docker compose up -d. Миграции должны быть обратно-совместимыми (expand/contract): старый контейнер обслуживает трафик до перезапуска.

Переменные окружения (/opt/eden/.env)

Проверка живости

GET /healthz → ok — отвечает до роутера, без обращения к БД; ходит по HTTP без редиректа (TLS терминируется на Caddy).

Первичный запуск («go-live», уже выполняется по чек-листу deploy/README.md)

  1. Бэкап текущего состояния (./backup.sh).
  2. Сброс данных: Eden.Release.reset!() — TRUNCATE всех таблиц + очистка uploads; защищён точным значением EDEN_ALLOW_RESET=yes-wipe-everything.
  3. Первый администратор: Eden.Release.bootstrap_super_admin() (логин/пароль через env, пароль не попадает в историю шелла).
  4. Вход → /admin → выпуск приглашений реальным сотрудникам.

8. Эксплуатация: бэкапы, мониторинг, обновления

Шпаргалка (на сервере, из /opt/eden)

docker compose ps                     # статус контейнеров
docker compose logs -f app            # логи приложения
docker compose logs -f caddy          # логи TLS/прокси
docker compose run --rm app /app/bin/migrate    # миграции вручную
docker compose exec app /app/bin/eden remote    # IEx в работающий релиз
docker compose up -d                  # применить .env / перезапуск

Бэкапы

  • backup.sh по cron (ежедневно 03:30): pg_dump в /opt/eden/backups/, хранится 14 последних; каждый дамп зеркалируется в отдельный S3/R2-bucket (rclone в одноразовом контейнере). Сбой отгрузки не удаляет локальный дамп и попадает в алерты.
  • Восстановление: gunzip -c backups/eden-<stamp>.sql.gz | docker compose exec -T db psql -U <user> <db> (при потере хоста — сначала забрать дамп из bucket).
  • Медиа-файлы в дампы БД не входят: это volume uploads (или S3-bucket) — учитывать при переносе сервера.

Мониторинг и алерты

  • monitor.sh по cron каждые 10 минут: заполнение диска (порог 80%), здоровье контейнера приложения; алерты в Telegram (почты в проекте нет), повтор не чаще раза в 6 часов.
  • Dead-man switch: пинг внешнего сервиса (healthchecks.io) — падение всего хоста замечает внешняя сторона.

Обновление и откат

  • Обновление = merge в main → запуск workflow Deploy.
  • Откат приложения: указать в .env предыдущий образ APP_IMAGE=ghcr.io/…:<sha> и docker compose up -d.
Миграции — только вперёд. Часть ранних миграций не обратима по данным (например, room_access_general_only удаляет строки членства). Проблемы чинить новой миграцией «вперёд», а не откатом схемы.

9. Разработка и контроль качества

Локальная среда

mix setup            # зависимости + БД + ассеты
mix phx.server       # запуск (http://localhost:4000)
mix test             # тесты
mix check            # ЕДИНЫЙ гейт: format → compile --warnings-as-errors
                     #   → credo --strict → sobelow → deps.audit → test
mix dialyzer         # типовой анализ (обязателен, отдельно от check)

Definition of Done: зелёные mix check и mix dialyzer, тесты на новое поведение, заполненный чек-лист PR (обратимость миграций, отсутствие секретов, оценка влияния на безопасность, обновление документации при изменении архитектуры).

CI/CD

  • CI на PR: полный гейт + dialyzer; job release-smoke собирает прод-релиз и гоняет миграции через него — прод-специфичные регрессии ловятся до деплоя.
  • E2E (Playwright, мульти-пользовательский харнес в test/e2e/) — запуск вручную (workflow «E2E (manual)»); не является PR-гейтом.
  • Деплой — только ручной запуск workflow Deploy (см. §7).

10. Известные ограничения и планы

  • Уведомления в браузере — только в открытой вкладке. Web-адаптер внутривкладочный; браузерных push при закрытой вкладке нет. Мобильные транспорты APNs (iOS) и FCM (Android) реализованы на шве ADR-0001 и включаются переменными окружения (см. §7) — до выпуска ключей пуш выключен и поведение не меняется. FCM бесплатен, но на Android без Google-сервисов не доставляется — фолбэк RuStore/VK запланирован тем же адаптером.
  • Электронной почты нет и не планируется — все сценарии восстановления идут через администратора (осознанное ограничение, см. §1).
  • Аудио-вложения классифицируются как обычные файлы (голосовые сообщения — отложены).
  • Поиск — подстрочный/триграммный; полнотекстовый поиск со стеммингом и ранжированием — зафиксированный follow-up.
  • Латентность для дальних пользователей ограничена RTT; заготовлен план Cloudflare (edge-TLS + кэш статики/медиа) — активация требует доступа к DNS-зоне домена.
  • Мульти-организации (тенант-модель над каналами), внешние IdP (VK ID / Яндекс ID / корп-OIDC) — проработаны в RFC и ADR-0002 как будущие шаги; в текущей версии один периметр и парольный вход с 2FA.
