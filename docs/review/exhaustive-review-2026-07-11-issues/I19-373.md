# Storage/медиа: разрулить мёртвый kind «audio», усечь длинные имена файлов и закрыть контракт image-пайплайна тестами

**Приоритет:** P2 — мёртвый UI и молчаливый провал отправки по длинному имени бьют по продукту/UX русскоязычной аудитории, но не приводят к крашу или потере данных; сопутствующие находки — поддерживаемость (дрейф контракта) и покрытие тестами.
**Лейблы:** `enhancement` `P2: medium` `backend` `UI/UX` `ready-for-agent`
**Находки ревью:** R039, R040, R154, R158, R210

## Контекст
Все пять находок — про **пайплайн вложений и обработки изображений** (зона `storage-media`): классификация загрузок по магическим байтам (`Eden.Chat.sniff/2`), правила схемы `Attachment`, и image-процессинг (`Eden.Images`). Общий корень у трёх из них — расхождение между **заявленным контрактом** (moduledoc/CLAUDE.md обещают `kind ∈ image|video|file|audio` и «EXIF stripped») и **фактической реализацией**: аудио физически недостижимо (R039/R154), а гарантия стрипа метаданных нигде не проверяется (R210). Плюс два локальных UX/качество-дефекта в том же слое: длинное имя файла валит отправку generic-ошибкой (R040) и `square_avatar` может пропустить наружу сырую vix-ошибку мимо задокументированного набора атомов (R158). Сводим в один issue, потому что это один слой и часть шагов пересекается (решение по `audio` затрагивает и `sniff`, и UI, и caps).

## Находки и доказательства

### R039 · P2/product · `lib/eden/chat.ex:4208` — kind «audio» никогда не присваивается; вкладка «Audio» мертва, m4a едет как видео
- **Проблема:** `sniff/2` не содержит ни одной аудио-сигнатуры (mp3/ID3, OggS, fLaC, RIFF/WAVE). Все аудиофайлы получают `kind="file"`. Хуже — `.m4a`/`.m4b` (ISO-контейнер, brand `M4A `/`M4B `) попадает в **video-ветку** ftyp и хранится как `video/mp4`, потому что аудио-брэнды не в HEIC-списке, а всё-не-HEIC по умолчанию `video`.
- **Доказательство в коде** (`sniff/2`, актуальные строки 4208–4236):
  ```elixir
  defp sniff(<<_::binary-size(4), "ftyp", brand::binary-size(4), _::binary>>, _f) do
    if brand in ~w(heic heix heim heis hevc hevx hevm hevs mif1 msf1 heif),
      do: {"image", "image/heic", "heic"},
      else: {"video", "video/mp4", "mp4"}   # ← M4A/M4B попадают сюда
  end
  # ...ни одной аудио-клаузы...
  defp sniff(_other, f), do: {"file", "application/octet-stream", file_ext(f) || "bin"}
  ```
  При этом `kind="audio"` объявлен и подпёрт по всему коду как «живой»: `@kinds` (`attachment.ex:15`), cap `max_attachment_bytes("audio")` (`chat.ex:120`, `@max_audio_bytes = 25MB` — `chat.ex:57`), иконки (`chat_live.ex:9311 kind_icon("audio")`, `12151 gallery_empty_icon("audio")`), лейбл нотификации (`notify_hook.ex:111`, `chat_live.ex:9971 media_label("audio")`) и **вкладка галереи** (`chat_live.ex:12134-12141 gallery_tabs`). Инлайн-плеера нет: `attachment_view/1` для всего, кроме image/video, — это download-anchor (`chat_live.ex:11377-11395`), `<audio controls>` в шаблоне отсутствует (grep по `<audio` пуст).
- **Влияние:** вкладка «Audio» всегда пуста («No audio in this chat yet») — мёртвый, вводящий в заблуждение UI. Аудиофайлы качаются как документы; m4a-голосовушка рендерится тегом `<video>` с пустым чёрным боксом. Moduledoc `Attachment` и CLAUDE.md («kind classified by magic bytes: image|video|file|audio») не соответствуют коду.

### R154 · P3/quality · `lib/eden/chat.ex:4208` — kind «audio» недостижим: sniff не знает ни одного аудиоформата
- **Проблема:** дубль-подтверждение R039 под углом «мёртвый код»: ветки `@max_audio_bytes`/`max_attachment_bytes("audio")`/`audio` в `@kinds` и его UI объявлены, но недостижимы, т.к. классификатор не выдаёт `"audio"` ни для одного входа.
- **Доказательство в коде:** те же строки, что в R039 (`chat.ex:57`, `:120`, `attachment.ex:15`). `check_size/2` считает cap через `max_attachment_bytes(kind)` (`chat.ex:4251-4260`) — клауза `"audio"` никогда не выбирается, т.к. `classify/1` не возвращает `"audio"`.
- **Влияние:** объявленный лимит 25MB для audio неприменим; читателю кода неочевидно, что целая ветвь недостижима. Продуктовый гэп + дрейф кода. (Это та же проблема, что R039 — фикс общий.)

### R040 · P2/ux · `lib/eden/chat/attachment.ex:61` — имя файла длиннее 255 байт валит всю отправку generic-ошибкой
- **Проблема:** `validate_length(:filename, max: 255, count: :bytes)` **отклоняет** changeset вместо усечения. Кириллица — 2 байта/символ → предел ~127 символов. Ошибка всплывает на `insert` (в `persist_album`), уже **после** завершённого аплоада, мапится в `attachment_error(_other)` → «Couldn't send that file.» без причины.
- **Доказательство в коде** (`attachment.ex:60-79`):
  ```elixir
  |> update_change(:filename, &sanitize_filename/1)
  |> validate_length(:filename, max: 255, count: :bytes)   # ← отклоняет, не усекает
  ...
  defp sanitize_filename(name) when is_binary(name) do
    name |> Path.basename() |> String.replace(~r/[\x00-\x1f"\\\/]/u, "") |> String.trim() ...
  end
  ```
  На стороне `persist_album/4` (`chat.ex:2469-2473`) `{:error, changeset}` откатывает блобы и уходит в `resolve_duplicate`, а web-слой в итоге показывает generic-текст:
  ```elixir
  # chat_live.ex:14768
  defp attachment_error(_other), do: gettext("Couldn't send that file.")
  ```
- **Влияние:** для аудитории ihichat файл с длинным осмысленным именем («Протокол совещания по …….docx» >127 символов) молча не отправляется — пользователь прождал загрузку и получил невнятную ошибку. Имя — не контент; его безопасно усечь (так делают Telegram и почтовые клиенты).

### R158 · P3/quality · `lib/eden/images.ex:44` — square_avatar: сырые vix-ошибки просачиваются мимо контракта
- **Проблема:** `@doc` обещает `{:error, :too_large | :unprocessable}`, но `rescue` в `square_avatar/1` ловит только **исключения**. Если `thumbnail_buffer` или `Image.write` **вернут** `{:error, "vips-строка"}` (напр. обрезанный JPEG прошёл header-парс `from_binary`, но падает на полном декоде), с `with` без `else` этот `{:error, <строка>}` уходит вызывающему как есть.
- **Доказательство в коде** (`images.ex:33-48`, `with` без `else`):
  ```elixir
  def square_avatar(path) do
    with {:ok, bytes} <- File.read(path),
         :ok <- check_size(bytes),
         {:ok, image} <- Image.from_binary(bytes),
         :ok <- check_pixels(Image.width(image), Image.height(image)),
         {:ok, square} <- Vix.Vips.Operation.thumbnail_buffer(bytes, @avatar_size, ...) do
      Image.write(square, :memory, suffix: ".jpg", quality: 82, strip_metadata: true)
    end     # ← нет else: {:error, "vips string"} выходит наружу как есть
  rescue
    _ -> {:error, :unprocessable}
  end
  ```
  `compress_photo/2` от этого спасён своим `else -> :keep` (`images.ex:80-82`). Эталон нормализации — уже применённый в `heic_to_jpeg/1` после ревью #123 B1 (`chat.ex:4315-4320`): `else _ -> {:error, :unprocessable}`.
- **Влияние:** вызывающие (`Accounts.set_avatar`, `Channels`, `Chat.set_group_avatar`) и их UI-мапперы рассчитывают на закрытый набор атомов; web-слой сегодня выживает только за счёт catch-all (`group_avatar_error(_other)`). Любой новый вызывающий без catch-all получит неожиданный терм. **Confidence: medium** (путь редкий — нужен файл, проходящий header-парс, но падающий на полном декоде).

### R210 · P3/test-gap · `test/eden/images_test.exs:18` — стрип EXIF/GPS нигде не проверяется тестом
- **Проблема:** `images_test.exs` покрывает только pixel-cap и happy-path; ни один тест в репозитории не проверяет, что `square_avatar`/`compress_photo` действительно вырезают EXIF (в т.ч. GPS). `grep` по `exif`/`strip` в `test/` пуст.
- **Доказательство в коде** (`images_test.exs:18-41` — только `describe "pixel cap (#231)"`): три теста — reject 40MP, degrade-to-keep, happy-path; ни одного ассерта на метаданные. При этом код заявляет гарантию: `strip_metadata: true` в `square_avatar` (`images.ex:44`) и `compress_photo` (`images.ex:73`), moduledoc `Eden.Images` («EXIF stripped») и CLAUDE.md («metadata stripped»).
- **Влияние:** приватность-гарантия пайплайна не защищена регресс-тестом; её тихая поломка (напр. при смене опций `Image.write`) утечёт геолокацию отправителей фото. **Confidence: medium** (нужна фикстура JPEG с EXIF/GPS).

## План реализации

Порядок: сначала продуктовое решение по audio (R039/R154 — развилка, см. «Заметки»), затем механические фиксы.

- [ ] **Решить и реализовать судьбу kind «audio» (R039 + R154 — один фикс).** Развилка для человека (см. «Заметки»):
  - **Вариант A (внедрить аудио):** добавить в `sniff/2` (`lib/eden/chat.ex:4208-4236`) аудио-сигнатуры **перед** generic-клаузой: `"ID3"` и `<<0xFF, 0xFB, _>>`/`<<0xFF, 0xF3>>`/`<<0xFF, 0xF2>>` → `{"audio","audio/mpeg","mp3"}`; `"OggS"` → `{"audio","audio/ogg","ogg"}`; `"fLaC"` → `{"audio","audio/flac","flac"}`; `<<"RIFF", _::binary-size(4), "WAVE", _>>` → `{"audio","audio/wav","wav"}` (осторожно: WEBP тоже RIFF — WAVE-клауза должна отличаться по маркеру, WEBP-клауза уже на `4208+5`). В ftyp-ветке (`chat.ex:4221-4225`) выделить аудио-брэнды `~w(M4A M4B  mp42)`-подобные в `{"audio","audio/mp4","m4a"}` **до** `else video`. В UI — новая клауза `attachment_view/1` для `kind: "audio"` c `<audio controls preload="metadata">` (`chat_live.ex:11377`, добавить ветку выше generic-download). Обновить `needs_media_processing?/1` (`chat.ex:4264-4266`) при желании тянуть длительность через ffprobe (опционально).
  - **Вариант B (отложить аудио явно):** удалить мёртвые ветки и зафиксировать решение комментарием: убрать `{"audio", ...}` из `gallery_tabs/0` (`chat_live.ex:12134-12141`), клаузы `gallery_empty_text("audio")`/`gallery_empty_icon("audio")` (`:12146`, `:12151`); удалить `max_attachment_bytes("audio")` (`chat.ex:120`) и `@max_audio_bytes` (`chat.ex:57`); убрать `"audio"` из `@kinds` (`attachment.ex:15`) **только если** миграции/данных с таким kind нет (иначе оставить и пометить комментарием); поправить moduledoc `Attachment` (`attachment.ex:1-11`) и запись в CLAUDE.md, оставив комментарий-маркер в `sniff/2`, что audio сознательно не классифицируется. (m4a при этом остаётся видео — при желании добавить только распознавание `M4A ` как `file`, чтобы не показывать чёрный `<video>`.)
- [ ] **Усечь `filename` до 255 байт с сохранением расширения (R040).** В `sanitize_filename/1` (`lib/eden/chat/attachment.ex:70-79`) после очистки усекать basename до `255 - byte_size(ext)` байт **по границам графем** (не рвать UTF-8): выделить `ext = Path.extname(cleaned)`, `base = Path.rootname(cleaned)`, накапливать графемы `String.graphemes(base)` пока `byte_size(acc <> g) <= limit`, собрать `truncated_base <> ext`. `validate_length(:filename, max: 255, count: :bytes)` (`attachment.ex:61`) **оставить** как страховку. Проверить, что усечение не сносит расширение, если оно само >255 байт (краевой случай — тогда усечь и его).
- [ ] **Закрыть контракт `square_avatar/1` (R158).** Добавить в `with` (`lib/eden/images.ex:33-45`) `else`-ветку по образцу `heic_to_jpeg/1` (`chat.ex:4315-4320`): `{:error, :too_large} -> {:error, :too_large}; _ -> {:error, :unprocessable}`, чтобы наружу выходили только задокументированные атомы даже при `{:error, "vips-строка"}`. `compress_photo/2` (`images.ex:60-85`) уже безопасен (`else -> :keep`) — не трогать.

## Критерии приёмки
- Принято продуктовое решение по audio (A или B) и реализовано целостно: нет объявленного, но недостижимого `kind="audio"` (Вариант B) **или** аудио реально классифицируется и играет инлайн (Вариант A). m4a больше не рендерится пустым `<video>`.
- Файл с именем >255 байт (в т.ч. кириллица) отправляется успешно — имя усечено, расширение сохранено; generic-ошибка «Couldn't send that file.» на длинном имени больше не воспроизводится.
- `square_avatar/1` при любом входе возвращает только `{:ok, binary}` | `{:error, :too_large}` | `{:error, :unprocessable}` — сырых vix-строк наружу нет.
- Есть регресс-тест, падающий при отключении `strip_metadata`.
- Moduledoc `Attachment`/`Eden.Images` и CLAUDE.md согласованы с фактической реализацией.
- `mix check` и `mix dialyzer` зелёные; регрессий в существующих тестах attachment/images нет.

## Заметки
- **Развилка для человека (R039/R154):** внедрять ли аудио как первоклассный тип (Вариант A: сигнатуры + инлайн-плеер + возможно ffprobe-длительность — это заметный объём) или отложить и убрать мёртвый UI/caps (Вариант B — механический). Это продуктовое решение (нужен ли голос/аудио в ihichat сейчас), поэтому issue помечен `ready-for-agent` для остальных четырёх шагов, но по audio агент должен реализовывать **выбранный человеком** вариант; если решение не принято — стоит завести отдельный `ready-for-human`-подтикет только на audio-продукт-вопрос, а R040/R158/R210 закрыть здесь сразу.
- Все пять находок статуса **advisory** (не проходили независимую верификацию верификатором — `verdict_note` пуст), но при чтении кода **все пять подтвердились** на актуальных строках: `sniff/2` без audio (4208-4236), ftyp→video для m4a (4221-4225), `validate_length` отклоняет вместо усечения (attachment.ex:61), `square_avatar` без `else` (images.ex:33-45), тест-файл без EXIF-ассертов (images_test.exs:18-41). Confidence: R039/R040/R154 — high; R158/R210 — medium (редкий путь / нужна фикстура).
- R039 и R154 — одна и та же проблема с двух ракурсов (product-gap vs dead-code); фикс общий (первый шаг плана), оба id оставлены для трассируемости.
- Связь: паттерн нормализации ошибок для R158 уже есть в репо (`heic_to_jpeg`, ревью #123 B1) — просто перенести на `square_avatar`.
