# Убрать избыточную работу на горячих путях: рассылка/пересчёт сайдбара, пер-ключевые blob-запросы и лишние UPDATE при доставке

**Приоритет:** P2 — поддерживаемость и масштабируемость: сейчас код функционально корректен (бейджи сходятся, блобы чистятся, доставка работает), но на самых частых путях (доставка каждого сообщения, каждый рефреш сайдбара, каждое delete-for-both) выполняется лишняя работа — дублирующие агрегатные запросы, N round-trip'ов на альбом, безусловные UPDATE и рассылки. Это не краш и не потеря данных, но линейно растущая паразитная нагрузка на БД/сокет single-VPS прода за ~160 ms RTT.
**Лейблы:** `enhancement` `P2: medium` `backend`
**Находки ревью:** R047, R051, R059, R118, R119, R132, R133, R148, R149, R152

## Контекст

Все находки — про **производительность горячих путей** мессенджера: путь доставки сообщения (`Eden.Chat.deliver/2` → `notify_members`/`notify_new`/`resurface_direct`), путь рефреша сайдбара (`Chat.list_conversations/2` + его `preload`/`last_message_previews`), жизненный цикл `ChatLive` (mount/handle_params и PubSub-обработчики), пути удаления (`delete_message_for_both`, `delete_messages_for_both`, `unshared_blob_keys`) и первичный рендер `/admin`. Общий корень — **работа делается безусловно и/или поштучно там, где её можно дедуплицировать, обусловить или сбатчить**: одно и то же событие пересчитывает бейджи дважды, один UPDATE/EXISTS выполняется на каждое сообщение/ключ, а один mount-стрим тут же выбрасывается следующим `handle_params`. По отдельности каждая мелочь, но все они — на путях, которые исполняются на каждое сообщение / каждый заход / каждый рефреш у каждого участника.

Три находки (R047, R051, R059) — **P2** (масштабируемость/лишний запрос на заход), остальные семь — **P3** (advisory, не верифицированы независимо; см. отметки в доказательствах). Все проверены чтением актуального кода — номера строк ниже сверены.

### R051 · P2/perf · `lib/eden_web/live/chat_live.ex:272` — mount-стрим сайдбара сразу перезаписывается handle_params

- **Проблема:** `mount` безусловно грузит и стримит `Chat.list_conversations(scope)`, но каждый не-канальный `handle_params` немедленно пере-стримит сайдбар с `reset: true` — mount-результат заведомо выбрасывается.
- **Доказательство в коде:** verified. `chat_live.ex:272-275` (в mount, вне `if connected?`):
  ```elixir
  |> then(fn s ->
    convos = Chat.list_conversations(scope)
    s |> assign(sidebar_top: top_conv_id(convos)) |> stream(:conversations, convos)
  end)
  ```
  Все не-канальные ветки `handle_params` тут же пере-стримивают: `_params → refresh_sidebar` (`chat_live.ex:453`); `%{"id"}`/`%{"id","message_id"}` → `select_conversation` → `refresh_sidebar`. `refresh_sidebar/1` (`chat_live.ex:12364`) = `stream_conversations(socket, reset: true)`, а `stream_conversations/2` (`chat_live.ex:12369-12382`) снова зовёт `Chat.list_conversations(scope, folder_id)`. При mount `folder_id: nil` (`chat_live.ex:176`), т.е. mount-вызов эквивалентен handle_params-вызову. На connected-заходе mount+handle_params исполняются дважды (disconnected+connected) → `list_conversations` прогоняется ~4 раза за загрузку.
- **Влияние:** лишний DB-запрос + лишний reset-стрим на каждый заход любого маршрута. Не краш/некорректность. *Уточнение границы:* в **канальном** режиме (`handle_params` @ `chat_live.ex:382`) `stream_conversations` — no-op (`chat_live.ex:12370`, `socket.assigns.channel` truthy), т.е. mount-стрим там не «перезаписывается», а просто не рендерится (контейнер под `:if={is_nil(@channel)}`). Формулировка «всегда перезаписывается» неточна для канального режима, но вывод «mount-стрим = впустую» верен везде.

### R119 · P3/perf · `lib/eden/chat.ex:210` — last_message_previews: DISTINCT ON без поддерживающего индекса

- **Проблема:** запрос превью использует `DISTINCT ON (conversation_id)` c `ORDER BY conversation_id ASC, id DESC`, но на `messages` нет индекса `(conversation_id, id DESC)` — единственный композит `(conversation_id, inserted_at)` порядок по `id` не отдаёт, поэтому Postgres фетчит все видимые сообщения всех бесед сайдбара и сортирует их.
- **Доказательство в коде:** advisory (не верифицировано независимо; сверено чтением). `chat.ex:210-211`:
  ```elixir
  distinct: m.conversation_id,
  order_by: [asc: m.conversation_id, desc: m.id],
  ```
  Индексы на `messages` (миграции): `[:conversation_id, :inserted_at]` (`20260606130148_create_chat.exs:33`), `[:root_id]`, `[:reply_to_id]`, `[:forwarded_from_id]`, `[:conversation_id, :group_id] WHERE group_id IS NOT NULL`, trigram GIN на `body`, `unique(:sender_id, :client_id)`. **Индекса `(conversation_id, id DESC)` нет.** Запрос выполняется на каждый `list_conversations`, т.е. на каждый `{:conversation_activity}`-пинг каждого участника.
- **Влияние:** стоимость растёт линейно с суммарной историей всех чатов пользователя; на single-VPS проде это самый горячий путь сайдбара.

### R148 · P3/perf · `lib/eden/chat.ex:3471` — unshared_blob_keys: один EXISTS-запрос на каждый ключ (до ~20 на альбом)

- **Проблема:** `unshared_blob_keys/1` гоняет `blob_shared_outside?` отдельным `Repo.exists?` на каждый `storage_key`/`thumbnail_key` (альбом из 10 фото с тумбами = до 20 запросов), тогда как рядом `delete_unreferenced_blobs/1` решает ту же задачу одним запросом `where key in ^keys`.
- **Доказательство в коде:** advisory (confidence high, но не верифицировано независимо; сверено чтением). `chat.ex:3461-3476`:
  ```elixir
  defp unshared_blob_keys(attachments) do
    ids = Enum.map(attachments, & &1.id)
    attachments
    |> Enum.flat_map(&[&1.storage_key, &1.thumbnail_key])
    |> Enum.reject(&is_nil/1)
    |> Enum.uniq()
    |> Enum.reject(&blob_shared_outside?(&1, ids))
  end

  defp blob_shared_outside?(key, exclude_ids) do
    Repo.exists?(
      from a in Attachment,
        where: a.id not in ^exclude_ids and (a.storage_key == ^key or a.thumbnail_key == ^key)
    )
  end
  ```
  Батчевый образец рядом — `delete_unreferenced_blobs/1` (`chat.ex:3487-3502`): один `where: a.storage_key in ^keys or a.thumbnail_key in ^keys`.
- **Влияние:** лишние round-trip'ы на каждом delete-for-both/edit-media; к тому же результат всё равно пост-коммитно перепроверяется `delete_unreferenced_blobs` — двойная работа.

## План реализации

Сгруппировано по путям. Все шаги независимы; порядок — по риску (сначала точечные дедупы/гварды, потом индекс, потом батч, в конце — фичевая пагинация).

- [ ] **R059 — дедуп двойного пересчёта badge в открытой комнате.** Сломать один из двух путей. Предпочтительно: в self-ветке `{:read}` (`chat_live.ex:2531-2541`) НЕ звать `refresh_folders`/`refresh_rail` — тот же пересчёт уже сделан `{:conversation_activity}` (`chat_live.ex:2568-2576`) на той же операции доставки; оставить только `put_sidebar_conversation`. Проверить, что путь «открытие чата» (`mark_read` при `select_conversation`, `chat_live.ex:12223`) не полагается на этот пересчёт — там `{:conversation_activity}` не приходит, значит убрать пересчёт из self-`{:read}` **нельзя без замены**; вместо этого лучше **не слать `{:read}` самому себе при авто-mark_read**. Развилка на человека — см. «Заметки».
- [ ] **R132 — пропускать `notify_members` для тред-реплаев.** В `delete_message_for_both/2` (`chat.ex:2679`) обусловить: `if is_nil(tombstone.root_id), do: notify_members(message.conversation_id)`. `{:thread_updated}` уже покрывает тред-обновление; сайдбар/фолдеры реплаи не трогают.
- [ ] **R148 — переписать `unshared_blob_keys/1` одним запросом.** Заменить `Enum.reject(&blob_shared_outside?/2)` (`chat.ex:3461-3476`) на один запрос по образцу `delete_unreferenced_blobs/1`: собрать `keys`, одним `from a in Attachment, where: a.id not in ^ids and (a.storage_key in ^keys or a.thumbnail_key in ^keys), select: [a.storage_key, a.thumbnail_key]` получить множество «shared» и вычесть. Удалить `blob_shared_outside?/2`.
- [ ] **R152 — обусловить `resurface_direct/1`.** Прокинуть тип беседы в `deliver/2` (`conv` и так грузится в `notify_new`, `chat.ex:3765`) и звать `resurface_direct` только для не-групповых не-канальных бесед, ИЛИ слить его с `touch_conversation` в один UPDATE. Порядок: не менять контракт `deliver/2` до того, как `notify_new` отдаёт `conv` наверх (или загрузить `conv` в `deliver` один раз и передать вниз).
- [ ] **R118 — облегчить preload сайдбара.** В `list_conversations/2` (`chat.ex:144`) прелоадить только активные membership'ы: `preload(memberships: ^from(m in Membership, where: is_nil(m.left_at)) )` (или именованный query-fun) с вложенным `:user`. Свериться, что `active_members/1` (`chat_live.ex:13104`) и все читатели `memberships` в веб-слое остаются корректны при отсутствии ушедших строк (1:1-«собеседник ушёл» кейс — peer с `left_at` уже фильтруется, проверить peer-логику).
- [ ] **R119 — индекс под DISTINCT ON превью.** Новая миграция: `create index(:messages, [:conversation_id, "id DESC"], where: "deleted_at IS NULL AND root_id IS NULL AND kind != 'system'")` (частичный, под предикаты `last_message_previews`, `chat.ex:207-211`). Индекс большой на write — оценить; альтернатива — переписать `last_message_previews/2` через `JOIN LATERAL (... ORDER BY id DESC LIMIT 1)`. Выбор — «Заметки».

- Поведение неизменно во всех путях: бейджи rail/folders сходятся, блобы чистятся forward-safe, доставка/resurface/мьюты работают как раньше, `/admin` показывает и правит людей.
