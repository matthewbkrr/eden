# Закрыть локаут админки через деактивированного super_admin и достроить deleted/normalize-гварды в Accounts

**Приоритет:** P1 — `set_role_guarded`/`anonymize_locked`/`bootstrap_super_admin` считают деактивированного super_admin «живым», из-за чего двумя легитимными действиями можно свести число АКТИВНЫХ super_admin к нулю и полностью потерять доступ в `/admin` (нарушение заявленного инварианта «never ZERO super_admins», восстановление только прямым SQL) — это нарушение корректности/инварианта, а не краш/потеря данных.
**Лейблы:** `bug` `P1: critical` `backend`
**Находки ревью:** R002, R017, R018, R019, R048, R104, R113, R115

## Контекст

Все находки лежат в оффбординг-подсистеме `Eden.Accounts` (роли, деактивация #251, анонимизация/right-to-erasure #303) и её тестах. Общий корень — **непоследовательная трактовка «непригодных» аккаунтов**: guard «последнего super_admin» и bootstrap считают роль по колонке `role`, игнорируя `active`/`deleted_at`, а часть админ-мутаций пишет по in-memory struct'у без `FOR UPDATE`-перечитки и без `deleted?`-гварда — при том что соседние мутации (`reactivate_locked`, `mint_reset_locked`, `anonymize_locked`) эту идиому уже соблюдают. К этому примыкают два самостоятельных дефекта нормализации/чистоты (`display_name` без `normalize_text`; `broadcast_invites_changed(nil)`), пробелы в документации `set_user_role/3` и незакреплённые тестами инварианты вокруг deactivated/deleted — именно те края, где найдены дефекты.

## Находки и доказательства

### R002 · P1/bug · `lib/eden/accounts.ex:96-112`, `:296-310`, `:136` — guard «последнего super_admin» считает деактивированных → полный локаут админки

- **Проблема:** оба запроса `super_ids` фильтруют только по `role == "super_admin"`, без `active`/`deleted_at`; `deactivate_user/2` вообще не имеет счётчика «последний активный super». Сценарий из двух легитимных шагов приводит к нулю активных super_admin.
- **Доказательство в коде:** `set_role_guarded/2` (accounts.ex:96-107):
  ```elixir
  defp set_role_guarded(%User{} = target, role) do
    Repo.transact(fn ->
      super_ids =
        from(u in User, where: u.role == "super_admin", lock: "FOR UPDATE", select: u.id)
        |> Repo.all()

      if role != "super_admin" and target.id in super_ids and length(super_ids) <= 1 do
        {:error, :last_super_admin}
  ```
  Тот же запрос в `anonymize_locked/3` (accounts.ex:297-299) и exists-проверка bootstrap (accounts.ex:136): `Repo.exists?(from(u in User, where: u.role == "super_admin"))` — все по `role` без `active`/`deleted_at`. `deactivate_writes/1` (accounts.ex:176-183) роль НЕ снимает (флипается только `active`), а `deactivate_user/2` (accounts.ex:162) не проверяет число активных super. `super_admin?/1` (user.ex:85) и `can_reset_password?/2` (accounts.ex:683-684) смотрят только на `role`, не на `active`. Трасса: super A и B активны → A деактивирует B (`can_reset_password?` true, счётчика нет) → A демоутит себя (`super_ids=[A,B]`, `length ≤ 1` ложно, guard молчит) → **ноль активных super_admin**. B в `/admin` не вернётся (`get_user_by_session_token` отбивает inactive — тест accounts_test.exs:547-553), а `bootstrap_super_admin` найдёт деактивированного B по `role` → `{:error, :already_bootstrapped}`.
- **Влияние:** никто не может войти в `/admin`, реактивировать B или выдать reset-ссылку — восстановление только прямым SQL на проде. Инвариант «the platform must never reach ZERO super_admins» (комментарий accounts.ex:89-95) нарушается обычными действиями, без гонки. Confidence high, verified.

### R017 · P2/quality · `lib/eden/accounts.ex:162-183` — `deactivate_user` авторизует по in-memory struct без `FOR UPDATE` re-read

- **Проблема:** единственная админ-мутация, авторизующая и пишущая по переданному struct'у, а не по перечитанной под блокировкой свежей строке.
- **Доказательство в коде:** `deactivate_user/2` (accounts.ex:162-171) вызывает `authorize_activation(actor, target)` и `deactivate_writes(target)` по одному и тому же переданному struct'у; `deactivate_writes/1` пишет `set_active(target, false)`. Для контраста, `reactivate_locked` (accounts.ex:212-226) перечитывает `Repo.one(from u in User, ..., lock: "FOR UPDATE")` и повторяет `authorize_activation` + `User.deleted?` на свежей строке; так же устроены `mint_reset_locked` (:653-680) и `anonymize_locked` (:296-310). Confidence medium, **advisory** (не подтверждено независимым верификатором — рассуждение по асимметрии кода).
- **Влияние:** обычный админ может деактивировать цель, только что повышенную до super_admin (стейл `role` в панели до прихода `{:user_updated}`); деактивация параллельно удалённого аккаунта проходит молча. Несогласованность с собственной идиомой #263 кодовой базы.

### R018 · P2/bug · `lib/eden/accounts.ex:443-451`, `:96-112` — `apply_managed_fields` и `set_user_role` не отклоняют удалённый аккаунт

- **Проблема:** обе функции не проверяют `User.deleted?(target)` и не перечитывают цель под `FOR UPDATE`, в отличие от `reactivate_locked`/`mint_reset_locked`/`anonymize_locked`.
- **Доказательство в коде:** `apply_managed_fields/3` (accounts.ex:443-451):
  ```elixir
  def apply_managed_fields(%Scope{user: %User{} = actor}, %User{} = target, attrs) do
    if User.admin?(actor) do
      with {:ok, updated} <- target |> User.managed_changeset(attrs) |> Repo.update() do
  ```
  — только `admin?(actor)`, `Repo.update()` по переданному struct'у, без `deleted?`. `set_role_guarded/2` (accounts.ex:96-107) лочит `FOR UPDATE` только набор super_admin (id), но НЕ перечитывает target и не проверяет `deleted?` — обновляет устаревший struct напрямую. `managed_changeset` (user.ex:170) и `role_changeset` (user.ex:97) не фильтруют по `deleted_at`. Асимметрия реальна: `reactivate_locked` (:221), `mint_reset_locked` (:660), `anonymize_locked` (:306), `authorize_deletion` (:287) содержат явный `User.deleted?`-guard. Confidence medium, **verified**.
- **Влияние:** при гонке двух админ-сессий (A держит устаревший `selected`, B делает `delete_user_permanently` того же target; обе транзакции сериализуются на `FOR UPDATE` super-набора, delete коммитится первым) апдейт пройдёт по анонимизированной строке: `corp_email`/`position` перепишут вычищенный PII, `set_user_role` навесит `role=admin` на `deleted_at`-строку. Ломает гарантии #303 «a deleted account keeps no privilege» и right-to-erasure. Privilege-escalation нет (логин гейтится `active?`), поэтому P2, а не P1.

### R019 · P2/bug · `lib/eden/accounts/user.ex:230-234` — `display_name` не проходит `normalize_text`: NUL-байт валит запись (500), пробелы проходят в профиле

- **Проблема:** `validate_display_name/1` делает только `validate_required` + `validate_length`, без `normalize_text` — в отличие от `bio`/managed-полей.
- **Доказательство в коде:** `validate_display_name/1` (user.ex:230-234):
  ```elixir
  defp validate_display_name(changeset) do
    changeset
    |> validate_required([:display_name])
    |> validate_length(:display_name, min: 1, max: 50)
  end
  ```
  `profile_changeset` (user.ex:142) применяет `update_change(:bio, &normalize_text/1)`, но не к `display_name`; `registration_changeset` (user.ex:111-122) — тоже. `normalize_text/1` (user.ex:200-207) ровно для этого: `String.replace("\0", "") |> String.trim()`. `display_name = "\0"` проходит `validate_required`/`validate_length` (1 графема) и уходит сырым в `Repo.update` (accounts.ex:405) / `Repo.insert` (accounts.ex:1070) → `Postgrex.Error` (Postgres отвергает NUL) — краш LiveView настроек / 500 на POST инвайта. Confidence high, **verified**.
- **Влияние:** краш вместо ошибки валидации на пользовательском пути (профиль, регистрация). **Уточнение границы (из верификатора):** whitespace-only `"   "` сохраняется пустым только в `update_profile` — там `cast(..., empty_values: [])` (user.ex:140) отключает trim-в-пусто; в `registration_changeset` дефолтный `empty_values=[""]` с trim нилит `"   "` и честно роняет на `validate_required`, т.е. whitespace-only при регистрации НЕ сохраняется и НЕ крашит (заявление находки про пробелы на инвайте over-broad). NUL-краш требует crafted-payload, масштаб self-scoped → P2 (NUL-часть тяготеет к P1). Тот же класс, что уже зафиксирован для `title` в #267.

### R048 · P2/quality · `lib/eden_web/live/admin_live.ex:204-205` — пост-коммитная зачистка при удалении невозобновляема

- **Проблема:** cross-context cleanup выполняется вне транзакции; сбой между коммитом анонимизации и scrub'ом оставляет PII навсегда без продуктового пути дочистки.
- **Доказательство в коде:** `handle_event("delete_account", ...)` (admin_live.ex:196-219): после `{:ok, updated} <- Accounts.delete_user_permanently(...)` вызываются `Chat.scrub_deleted_user_content(updated.id)` и `Channels.revoke_invites_by(updated.id)` (admin_live.ex:204-205) — вне транзакции анонимизации. Повторный клик «удалить» вернёт `{:error, :already_deleted}` (accounts.ex:287/306), ветка со scrub'ом (admin_live.ex:198-211) не выполнится. Confidence medium, **advisory** (не подтверждено независимым верификатором).
- **Влияние:** right-to-erasure #303 частично не выполняется: имя остаётся в denormalized `meta` системных сообщений (knock/add/remove), приватные папки живут, дочистить может только консоль.

### R104 · P2/test-gap · `test/eden/accounts_test.exs:482` — не закреплены инварианты вокруг deactivated/deleted

- **Проблема:** отсутствуют тесты именно тех краёв, где найдены дефекты выше.
- **Доказательство в коде:** тесты `set_user_role` (accounts_test.exs:482-508) проверяют только role-путь: «cannot remove the last super_admin», «step down while another remains», «demote a peer then last is protected» — деактивированный super не покрыт. Deleted-кейсы (accounts_test.exs:659-724) покрывают только `reactivate`/reset/`delete`, но не `set_user_role`/`apply_managed_fields`/`deactivate_user` на удалённой цели. Bootstrap (accounts_test.exs:1013-1016) тестирует `:already_bootstrapped` только при **активном** super. `list_other_users` (accounts.ex:45-52) фильтрует `deleted_at`, но НЕ `active` — деактивированные пользователи всё ещё предлагаются в new-conversation picker; поведение не закреплено (тест :717 проверяет только deleted). Confidence high, **advisory**.
- **Влияние:** без пиннинга любое исправление guard'ов или будущий IdP-sync (авто-деактивация, ADR-0002) молча изменит семантику. Ядро оффбординга — core-логика по рубрике проекта.

### R113 · P3/quality · `lib/eden/accounts.ex:75-79` — `@doc set_user_role/3` противоречит реализации

- **Проблема:** док утверждает «An actor can't change their own role» и не перечисляет `{:error, :last_super_admin}`.
- **Доказательство в коде:** `@doc` (accounts.ex:75-79): «An actor can't change their own role … Returns `{:ok, user}` | `{:error, :forbidden}` | `{:error, changeset}`». Реализация намеренно разрешает самодемоушен при втором супере (комментарий set_role_guarded accounts.ex:89-95, тест accounts_test.exs:491-497) и возвращает недокументированный `{:error, :last_super_admin}` (accounts.ex:103). Confidence high, **advisory**.
- **Влияние:** первоисточник для будущих вызовов лжёт: caller не обработает `:last_super_admin` и будет считать self-change невозможным.

### R115 · P3/quality · `lib/eden/accounts.ex:1037` — `broadcast_invites_changed(nil)` вместо 0-арности

- **Проблема:** вызов 1-арной формы с `nil` вместо предназначенной для «просто оповестить» 0-арности.
- **Доказательство в коде:** `register_user_with_invite/2` (accounts.ex:1037) вызывает `broadcast_invites_changed(nil)`. 1-арная форма (accounts.ex:992-995) возвращает переданный invite для пайплайна; 0-арная (accounts.ex:999-1002) — ровно для «просто оповестить». Confidence high, **advisory**.
- **Влияние:** шум для читателя; будущий pattern-match на `%Invite{}` в 1-арности сломает этот вызов.

## План реализации

Порядок: сперва контекст-гварды (R002/R018/R017) как основа, затем changeset-нормализация (R019), затем идемпотентность зачистки (R048), затем тесты (R104) и косметика (R113/R115).

- [ ] **R002 — гварды по пригодным super_admin.** В обоих запросах `super_ids` — `set_role_guarded/2` (accounts.ex:98-99) и `anonymize_locked/3` (accounts.ex:297-298) — и в exists-проверке `bootstrap_super_admin` (accounts.ex:136) добавить `and u.active == true and is_nil(u.deleted_at)`, чтобы считался только пригодный к использованию super_admin. Выделить общий фрагмент запроса (напр. приватный `active_super_admin_ids/0` или `usable_super_admin_query/0`), чтобы три места не разъехались.
- [ ] **R002 — счётчик «последний активный super» в `deactivate_user`.** В `deactivate_user/2` (accounts.ex:162) добавить отказ `{:error, :last_super_admin}`, если target — единственный активный super_admin (по тому же общему запросу под `FOR UPDATE`), симметрично `delete_user_permanently`. Обновить `@doc` (accounts.ex:152-160) — добавить `:last_super_admin` в возвраты, а в AdminLive добавить flash-ветку на `{:error, :last_super_admin}` для пути деактивации (см. admin_live.ex:213).
- [ ] **R017 + R018 + R002 — перевести `deactivate_user/2` на locked-паттерн.** По образцу `reactivate_locked` (accounts.ex:212-226): внутри `Repo.transact` перечитать target `FOR UPDATE`, повторить `authorize_activation`, добавить отказ на `User.deleted?(target)` (`{:error, :already_deleted}`) и на «последний активный super» (`{:error, :last_super_admin}`), затем `set_active(fresh, false)` + удаления токенов/reset/инвайтов. Это закрывает и TOCTOU R017, и deleted-гвард R018 для деактивации в одном месте.
- [ ] **R018 — deleted-гвард в `set_role_guarded/2`.** В транзакции (accounts.ex:96-107) перечитывать target `FOR UPDATE` (по образцу `mint_reset_locked` accounts.ex:653-655) и возвращать `{:error, :already_deleted}` при `User.deleted?(target)` до `role_changeset`.
- [ ] **R018 — deleted-гвард в `apply_managed_fields/3`.** В `apply_managed_fields/3` (accounts.ex:443-451) обернуть запись в `Repo.transact`, перечитать target `FOR UPDATE`, вернуть `{:error, :already_deleted}` при `User.deleted?(target)` до `managed_changeset`. Обновить `@doc` (accounts.ex:441) — добавить `:already_deleted` в возвраты.
- [ ] **R019 — нормализовать `display_name`.** В `validate_display_name/1` (user.ex:230-234) добавить `update_change(:display_name, &normalize_text/1)` перед `validate_required` — NUL вычищается, а `"   "` становится `nil` и честно падает на `validate_required`. Применяется к обоим путям через `profile_changeset` (user.ex:141) и `registration_changeset` (user.ex:115). Проверить, что `normalize_text` не конфликтует с `empty_values: []` в `profile_changeset` (там change приходит уже без empty-обработки — normalize сам сведёт `"   "` к `nil`).
- [ ] **R048 — идемпотентная зачистка при удалении.** Сделать пост-коммитную чистку повторяемой: предпочтительно ставить Oban-джобу (очередь `:media`/новая уже есть в supervision tree) внутри `delete_user_permanently` после коммита — тогда падение LiveView-процесса не теряет чистку; либо в AdminLive на `{:error, :already_deleted}` всё равно прогонять идемпотентные `Chat.scrub_deleted_user_content/1` + `Channels.revoke_invites_by/1` (admin_live.ex:204-205, 216-217). Требует продуктового выбора между Oban и повтором-на-already_deleted (см. Заметки).
- [ ] **R113 — переписать `@doc set_user_role/3`.** В accounts.ex:75-79: актор МОЖЕТ менять и свою роль; отклоняется только снятие роли с последнего super_admin; добавить `{:error, :last_super_admin}` в список возвратов.
- [ ] **R115 — заменить `broadcast_invites_changed(nil)` на `broadcast_invites_changed()`** (accounts.ex:1037).

## Тесты

- [ ] **R002 (локаут):** в `describe "set_user_role"` — super A и B активны; `deactivate_user(A, B)`; затем `set_user_role(A, A, "member")` → `{:error, :last_super_admin}` (демоушен себя при деактивированном коллеге отклоняется).
- [ ] **R002 (deactivate-счётчик):** super A и B; `deactivate_user(A, B)`; затем `deactivate_user(A, A)`… (self отбивается `authorize_activation`) — проверить, что последний активный super нельзя деактивировать чужой рукой: A и B активны, A деактивирует B, второй актор C-super нет → эмулировать «последний активный super не деактивируется» через прямой сценарий с двумя суперами, один уже деактивирован.
- [ ] **R002 (bootstrap):** `bootstrap_super_admin` при существующем, но **деактивированном** super_admin → должен по-прежнему `{:error, :already_bootstrapped}` НЕ выдаваться, если решено считать только активных (закрепить выбранную семантику; см. - [ ] **R018:** в `describe "delete/anonymize"` — на удалённой цели `set_user_role(actor, deleted, "admin")` → `{:error, :already_deleted}`; `apply_managed_fields(actor, deleted, %{...})` → `{:error, :already_deleted}`; `deactivate_user(actor, deleted)` → отказ. Проверить, что PII на deleted-строке не перезаписан.

- [ ] **R048:** повторный вызов пути удаления (или Oban-джоба) дочищает `Chat.scrub_deleted_user_content`/папки — assert, что `meta` системных сообщений и приватные папки очищены даже при повторном прогоне.

## Критерии приёмки

- Двумя легитимными действиями (деактивация коллеги + самодемоушен) невозможно свести число активных super_admin к нулю: `set_user_role` и `deactivate_user` возвращают `{:error, :last_super_admin}`, когда затрагивается последний **активный** super_admin.
- `set_user_role`/`apply_managed_fields`/`deactivate_user` на удалённой (`deleted_at ≠ nil`) цели возвращают `{:error, :already_deleted}` и не пишут по анонимизированной строке.
- `display_name` с NUL не роняет `update_profile`/регистрацию (нет `Postgrex.Error`); whitespace-only отклоняется валидацией на обоих путях.
- Пост-коммитная зачистка при удалении возобновляема (Oban-джоба или повтор на `:already_deleted`).
- `@doc set_user_role/3` соответствует реализации; `broadcast_invites_changed()` без `nil`.

## Заметки

- **Развилка для человека (R048):** выбор между Oban-джобой для зачистки и повтором `Chat.scrub_deleted_user_content`/`revoke_invites_by` на `{:error, :already_deleted}` — продуктовое/архитектурное решение. Oban уже в supervision tree (`Eden.Chat.ThumbnailWorker`, `Eden.Accounts.TokenPruner`), так что worker предпочтителен и согласован с правилом CLAUDE.md «prefer Oban worker for background work». Если выбрать Oban — джоба должна быть строго идемпотентной (scrub по `user_id` уже идемпотентен по смыслу).
- **Развилка для человека (R104, `list_other_users`):** сейчас `list_other_users` (accounts.ex:45-52) фильтрует `deleted_at`, но НЕ `active` — деактивированный пользователь всё ещё предлагается в new-conversation picker. Нужно продуктовое решение: исключать ли deactivated из picker'а (тогда добавить `and u.active == true`). Тест должен закрепить выбранное поведение, а не менять его молча.
- **Развилка для человека (R002, bootstrap-семантика):** после сужения exists-проверки до активных super, `bootstrap_super_admin` перестанет считать деактивированного super «уже забутстрапленным». Это правильно для сценария восстановления (единственный super деактивирован → bootstrap может восстановить доступ), но меняет семантику go-live-бутстрапа — подтвердить, что это желаемое поведение (вероятно да: это и есть аварийный путь восстановления).
- **Уточнение R019 (не выкидывать, но сузить):** заявление находки «строка из пробелов проходит валидацию на инвайте» — over-broad: в `registration_changeset` дефолтный `empty_values=[""]` уже нилит `"   "` и роняет на `validate_required`. Whitespace-only сохраняется пустым ТОЛЬКО в `update_profile` (там `empty_values: []`). Фикс через `normalize_text` покрывает оба, но тест на whitespace для регистрации проверяет уже-существующее поведение.
- **Статусы находок:** R002, R018, R019 — verified (высокая/средняя уверенность, подтверждены трассировкой). R017, R048, R104, R113, R115 — advisory (independent verifier не приложил `verdict_note`), но при чтении кода все подтвердились по цитатам выше; R017 и R048 держатся на асимметрии с идиомой #263, R104/R113/R115 — фактические расхождения дока/тестов с кодом.
