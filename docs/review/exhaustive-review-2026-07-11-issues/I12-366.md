# Довести доступность до паритета: клавиатура/тач для реордера папок, aria-состояния меню и полей, единообразие QR/danger в 2FA

**Приоритет:** P2 — набор дефектов поддерживаемости/доступности и UX: клавиатурно-/тач-недоступный реордер папок (WCAG 2.1.1), нехватка aria-состояний/меток и мелкое расхождение токенов; данные не теряются и приложение не падает, но ключевая операция недоступна части аудитории.
**Лейблы:** `enhancement` `P2: medium` `UI/UX` `ready-for-agent`
**Находки ревью:** R093, R094, R174, R178, R181, R197

## Контекст
Пачка находок из ux-аудита LiveView-слоя (`settings_live.ex` + `chat_live.ex`): они не связаны одной причиной, но все относятся к одной подсистеме — **доступность и единообразие интерактивных контролов**. Три оси: (1) переупорядочивание папок держится только на нативном HTML5 drag-and-drop — ни клавиатуры, ни тача (R093/R094), хотя PRODUCT.md говорит «mostly from phones»; (2) aria-семантика/имена контролов неполны или неединообразны (меню канала R178, инпут композера R181, QR в 2FA R197); (3) один локальный баг корректности — `toggle_follow_thread` игнорирует ошибку сервера (R174). Общий приём фикса — переиспользовать уже существующие в кодовой базе паттерны: `.ContextMenu`-хук (aria-expanded + arrow-nav + Escape), `aria-label`, `aria-hidden`, `var(--ed-danger-strong)`, обработку `{:error, _}` как в `send_thread_reply_text`.

## Находки и доказательства

### R093 · P2/ux · `lib/eden_web/live/settings_live.ex:1043` — реордер папок без клавиатурной альтернативы
- **Проблема:** список папок сортируется исключительно нативным HTML5 drag-and-drop; ручка-«грипп» помечена `aria-hidden="true"`, кнопок «вверх/вниз» нет, клавиатурного пути к `reorder_folders` не существует.
- **Доказательство в коде:** `#folder-list` навешивает только `.Sortable`, ручки декоративны:
  ```elixir
  <ul id="folder-list" phx-hook=".Sortable" class="space-y-1.5">
    ...
    <span class="ed-folder-row__handle ed-folder-row__handle--grab" aria-hidden="true">
      <.icon name="hero-bars-3-micro" class="size-4" />
    </span>
  ```
  Хук `.Sortable` (settings_live.ex:1201-1250) шлёт `reorder_folders` только из `commit()`, вызываемого из `dragend` — никакого другого триггера нет.
- **Влияние:** нарушение WCAG 2.1.1 (Keyboard) — пользователи только с клавиатурой/скринридером не могут изменить порядок папок, включая позицию виртуального «All Chats». (confidence high, status advisory — не верифицировано вторым проходом, но подтверждено чтением кода.)

### R094 · P2/ux · `lib/eden_web/live/settings_live.ex:1201` — реордер только мышью, нет touch-пути
- **Проблема:** хук `.Sortable` слушает только `dragstart`/`dragend`/`dragover` — нативный HTML5 DnD, который на тач-устройствах не срабатывает. При этом копирайт (стр. 1039) обещает перетаскивание всем.
- **Доказательство в коде:** обработчики хука — исключительно HTML5 drag-события:
  ```javascript
  item.addEventListener("dragstart", (e) => { ... })
  item.addEventListener("dragend", () => { ... this.commit() })
  this.el.addEventListener("dragover", (e) => { e.preventDefault(); ... })
  ```
  Ни `touchstart`/`touchmove`, ни Pointer Events, ни long-press. Подсказка обещает: `"Group your chats. Drag to reorder — \"All Chats\" can be moved but not deleted."` (settings_live.ex:1038-1040).
- **Влияние:** PRODUCT.md прямо говорит, что пользуются «mostly from phones»; ключевая операция управления папками недоступна на мобильных — функция, обещанная копирайтом, не работает у основной аудитории. (confidence medium.)

### R174 · P3/ux · `lib/eden_web/live/chat_live.ex:2052` — `toggle_follow_thread` игнорирует ошибку follow/unfollow
- **Проблема:** результаты `Chat.unfollow_thread`/`follow_thread` отбрасываются; при `{:error, :deleted}` (рут затумбстоунен, пока панель открыта) или `:not_found` колокольчик всё равно перещёлкивается локально и `thread_unreads` правится, хотя на сервере ничего не изменилось.
- **Доказательство в коде:** возврат функций не матчится:
  ```elixir
  {following, unreads} =
    if socket.assigns.thread_following do
      Chat.unfollow_thread(scope, root.id)              # результат отброшен
      {false, Map.delete(socket.assigns.thread_unreads, root.id)}
    else
      Chat.follow_thread(scope, root.id)                # результат отброшен
      {true, Map.put_new(socket.assigns.thread_unreads, root.id, 0)}
    end
  ```
  При этом обе функции возвращают `{:ok, _}` либо `{:error, :not_found | :deleted | :not_a_root | :not_threaded}` из `with` (chat.ex:2895-2917). Эталон обработки ошибки уже есть — `send_thread_reply_text` (chat_live.ex:14711-14715): на `{:error, _}` он ставит `thread_root: nil` и `put_flash(:error, gettext("Thread not found."))`.
- **Влияние:** рассинхрон UI и сервера до следующего sync — пользователь думает, что подписался/отписался, а состояние не применилось. (confidence high; единственная находка category=bug в связке.)

### R178 · P3/ux · `lib/eden_web/live/chat_live.ex:2927` — меню канала не синхронизирует aria-expanded и без стрелочной навигации
- **Проблема:** кнопка меню канала имеет `aria-haspopup="menu"`, но `aria-expanded` не переключается при открытии/закрытии; меню открывается через `JS.toggle`, а не `.ContextMenu`-хук, поэтому нет arrow-навигации и переноса фокуса в пункты.
- **Доказательство в коде:**
  ```elixir
  <button type="button" class="ed-btn--icon"
    phx-click={JS.toggle(to: "#channel-menu")}
    aria-haspopup="menu"
    aria-label={gettext("Channel menu")}>   <%!-- нет aria-expanded --%>
  ...
  <div id="channel-menu" class="ed-menu ed-menu--anchored" role="menu" style="display: none;">
  ```
  Click-away/Escape тоже лишь `JS.hide` (chat_live.ex:2923-2925) — атрибут не трогают. Для сравнения, общий `.ContextMenu`-хук уже делает всё правильно: `trigger.setAttribute("aria-expanded", "true"/"false")` (chat_live.ex:5426/5439), фокусирует первый `[role=menuitem]` (5428-5429) и восстанавливает фокус на Escape (5422-5423).
- **Влияние:** скринридер не сообщает состояние «раскрыто/свёрнуто»; пункты (Members, Add members, Invite link, Edit channel, New room, Delete) достижимы обычным Tab без семантики меню. Второстепенно, но неединообразно с остальными меню. (confidence medium.)

### R181 · P3/ux · `lib/eden_web/live/chat_live.ex:3728` — основной инпут композера без aria-label
- **Проблема:** поле ввода сообщения `#composer-body` — `<input type="text">` с `placeholder`, но без `aria-label` и без связанного `<label>`; доступное имя вычисляется из placeholder, что не гарантировано во всех AT и исчезает при вводе текста.
- **Доказательство в коде:**
  ```elixir
  <input type="text" id="composer-body" name="message[body]"
    value={@composer[:body].value}
    class="ed-input"
    placeholder={gettext("Message")}   <%!-- единственный источник имени --%>
    autocomplete="off"
    phx-hook=".PasteUpload" />
  ```
  Никакого `aria-label` и никакого `<label for="composer-body">` рядом. (Ср. поле переименования папки, где `aria-label={gettext("Folder name")}` есть — settings_live.ex:1091.)
- **Влияние:** placeholder — ненадёжный источник доступного имени (WCAG рекомендует явный label/aria-label); центральный элемент интерфейса стоит подписать явно. (confidence medium.)

### R197 · P3/ux · `lib/eden_web/live/settings_live.ex:675` — QR TOTP в Settings не помечен aria-hidden, и danger-токен расходится с welcome
- **Проблема:** контейнер QR (`Phoenix.HTML.raw(@totp_setup.qr)`, SVG) в settings не имеет `aria-hidden="true"`, тогда как идентичный блок в welcome помечен; плюс ошибки TOTP в settings красятся `var(--ed-danger)`, а в welcome — `var(--ed-danger-strong)`.
- **Доказательство в коде:** settings — без `aria-hidden`:
  ```elixir
  <div class="rounded-[var(--ed-radius)] p-2" style="background:#fff; width:184px;">
    {Phoenix.HTML.raw(@totp_setup.qr)}   <%!-- сырой SVG, читается AT --%>
  </div>
  ```
  welcome — с `aria-hidden="true"` (welcome_totp_live.ex:140-146). TOTP-ошибки settings: `style="color: var(--ed-danger); ..."` (settings_live.ex:649 и 717); welcome: `style="color: var(--ed-danger-strong); ..."` (welcome_totp_live.ex:176). Комментарий в welcome явно фиксирует намерение: «The QR is decorative … hide the SVG from assistive tech» (welcome_totp_live.ex:134-137).
- **Влияние:** QR декоративен (доступный путь — ручной ключ ниже), поэтому должен скрываться от AT единообразно с welcome-экраном; расхождение по danger-токену — тот же контраст-разнобой. (confidence high.)

## План реализации
- [ ] **R174 (bug первым — корректность):** в `handle_event("toggle_follow_thread", ...)` (chat_live.ex:2052-2077) заматчить результат `Chat.follow_thread`/`unfollow_thread`. На `{:ok, _}` — текущее поведение; на `{:error, _}` — не менять локальное состояние, а поставить `thread_root: nil` (или закрыть панель) + `put_flash(:error, gettext("Thread not found."))`, как в `send_thread_reply_text` (chat_live.ex:14711-14715). Перенести вызов из `if` наружу в `case`, чтобы `following`/`unreads` вычислялись только на успехе.
- [ ] **R197 (два одностр. правки):** в `settings_live.ex` добавить `aria-hidden="true"` на QR-контейнер (settings_live.ex:671-676) и заменить `var(--ed-danger)` → `var(--ed-danger-strong)` в обоих TOTP-error-`<p>` (settings_live.ex:649 и 717) — привести к welcome_totp_live.
- [ ] **R181 (одностр. правка):** добавить `aria-label={gettext("Message")}` на `#composer-body` (chat_live.ex:3728-3737); placeholder оставить как подсказку. Проверить, нет ли второго (thread-)композера с той же проблемой и подписать его аналогично, если да.
- [ ] **R178:** синхронизировать `aria-expanded` кнопки меню канала. Минимальный вариант — добавить `aria-expanded="false"` на кнопку (chat_live.ex:2927-2935) и переключать через `JS.set_attribute`/`JS.remove_attribute` в `phx-click` toggle и в click-away/escape `JS.hide` (2923-2925). Предпочтительно — перевести `#channel-menu` на общий `.ContextMenu`-хук (у него уже есть aria-expanded + фокус первого пункта + Escape + arrow-nav, chat_live.ex:5122+/5426/5439), пометив кнопку `data-menu-trigger` и меню как `[role=menuitem]`-контейнер; тогда семантика станет единообразной с остальными меню.
- [ ] **R093 + R094 (свести в один клавиатурно-И-тач-доступный путь reorder):** добавить видимые кнопки «Переместить вверх / вниз» на каждую строку `#folder-list` (settings_live.ex:1043-1115), с `aria-label` (напр. `gettext("Move up")`/`gettext("Move down")`), пушащие `reorder_folders` с уже пересобранным порядком id (переиспользовать существующий серверный обработчик — сериализация порядка уже есть). Это одновременно закрывает клавиатурный барьер (R093) и тач-барьер (R094): кнопки работают и тапом, и с клавиатуры, и с ручкой DnD как есть. Дизейблить «вверх» на первой строке и «вниз» на последней; учесть виртуальный `:all` в перестановке. Обновить копирайт-подсказку (settings_live.ex:1038-1040), чтобы не обещать только «Drag» (напр. «Drag or use the arrows to reorder»).

## Критерии приёмки
- Порядок папок (включая позицию «All Chats») можно изменить **без мыши** — с клавиатуры и тапом на мобильном (R093/R094).
- `toggle_follow_thread` при серверной ошибке не рассинхронизирует UI: локальное состояние не меняется, показан flash, панель закрыта (R174).
- `#composer-body` имеет явное доступное имя `aria-label` (R181); QR в settings скрыт от AT `aria-hidden="true"`, TOTP-ошибки используют `var(--ed-danger-strong)` — паритет с welcome (R197); кнопка меню канала синхронизирует `aria-expanded` (R178).
- `mix check` зелёный, `mix dialyzer` зелёный, регрессий в существующих тестах папок/тредов/2FA нет.

## Заметки
- Все находки status=advisory (не прошли второй верификаторский проход), но **каждая подтверждена чтением актуального кода** — номера строк совпали (R093/R094 хук на 1201-1250; R174 на 2052-2077; R178 на 2927-2941; R181 на 3728-3737; R197 QR на 671-676, ошибки на 649/717).
- Развилка для человека по R178: минимальный `JS.set_attribute` быстрее и локальнее; перевод на `.ContextMenu`-хук даёт полную клавиатурную семантику меню, но требует привести разметку `#channel-menu` под контракт хука (`data-menu-trigger`, позиционирование fixed) — это уже небольшой рефактор. По умолчанию для агента — минимальный вариант; полный перевод оставить на усмотрение ревьюера.
- R093/R094 намеренно сведены в один пункт плана: кнопки «вверх/вниз» — это тот самый второй способ менять уже сериализуемый порядок, закрывающий обе оси (клавиатура + тач) без второго механизма.
