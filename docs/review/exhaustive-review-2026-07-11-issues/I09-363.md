# Уведомления: применять смену настроек вживую, озвучивать «стук» в приватную комнату, добить gating (деактивированные, media-баннер) и закрыть тесты

**Приоритет:** P2 — самый серьёзный дефект здесь (R096) — корректностный UX-баг настроек: выключение звука/десктопа/смена пресета не применяются до перезагрузки страницы (не потеря данных и не обход авторизации); остальное — продуктовые дыры gating-матрицы, будущая утечка на push-адаптерах и пробелы в тестах.
**Лейблы:** `bug` `P2: medium` `backend` `ready-for-agent`
**Находки ревью:** R029, R096, R108, R120, R144, R150, R164, R165, R166, R202, R203

## Контекст

Подсистема уведомлений (эпик #212–#218): `Eden.Chat` решает, **кто** слышит (гейтинг `common_gates`/`notify_recipient_ids` + payload через `notify_payload`), `Eden.Notifications.deliver/2` фанится по адаптерам (сейчас единственный — `Eden.Notifications.Web`), а `EdenWeb.NotifyHook` + компонент `EdenWeb.Notifier` (colocated-хук) рендерят chime / OS-баннер на каждой authed-странице. Находки бьют по трём слоям одного контракта: (1) настройки, которые пользователь меняет, не доезжают до рендерера в живой сессии (R096); (2) в gating-матрице есть тихие дыры — беззвучный «стук» в приватную комнату (R029/R120), отсутствие уведомления о реакции (R144), деактивированные/удалённые не отфильтрованы перед будущими push-адаптерами (R150); (3) баннер медиа-сообщения теряет медиа-маркер и тащит лишние поля клиенту (R202/R203); плюс серия мелких cross-tab/Firefox-нюансов рендерера (R164/R165/R166) и пробелы в тестах ядра «кто слышит» (R108). Общий корень — контракт payload/prefs между контекстом и рендерером ещё не «затянут» по краям.

## Находки и доказательства

### R096 · P2/bug · `lib/eden_web/live/settings_live.ex:1527-1576` — тумблеры уведомлений не обновляют `@notify_prefs`

- **Проблема:** обработчики `set_notify_sound` / `set_notify_sound_name` / `set_notify_desktop` пишут в БД, но присваивают ТОЛЬКО локальные ассайны `notify_sound`/`notify_sound_name`/`notify_desktop` (для отрисовки тумблеров, заданы в `assign_notifications/1`, settings_live.ex:92-112). Компонент `<.notifier :if={@notify_prefs} prefs={@notify_prefs} />` (settings_live.ex:201; тот же ассайн читают admin_live и chat_live) читает ОТДЕЛЬНЫЙ `@notify_prefs`, который `NotifyHook` задаёт ЕДИНОЖДЫ через `assign_new` и больше не пересчитывает. Бродкаста prefs на другие вкладки тоже нет.
- **Доказательство в коде:**
  ```elixir
  # settings_live.ex — пишем только локальный ассайн
  def handle_event("set_notify_sound", _params, socket) do
    %{sound: cur} = Chat.notification_prefs(socket.assigns.current_scope)
    {:ok, on} = Chat.set_notify_sound(socket.assigns.current_scope, not cur)
    {:noreply, assign(socket, notify_sound: on)}          # ← notify_prefs не тронут
  end
  ```
  ```elixir
  # notify_hook.ex:50 — задаётся один раз, дальше не пересчитывается
  socket = assign_new(socket, :notify_prefs, fn -> Chat.notification_prefs(scope) end)
  ```
  ```heex
  <!-- notifier.ex:19-21 — читает СТАРЫЙ @notify_prefs -->
  data-sound={to_string(@prefs.sound)}
  data-sound-name={@prefs.sound_name}
  data-desktop={to_string(@prefs.desktop)}
  ```
  На `/settings` `NotifyHook` доставляет `{:notify}` (`focused?` всегда false вне ChatLive), а `onNotify` при фокусе читает `this.el.dataset.sound` (notifier.ex:69) → chime играет по устаревшему значению. `play()` читает `this.el.dataset.soundName` (notifier.ex:127), поэтому комментарий notifier.ex:125-126 «is kept fresh by the server render» описывает несуществующее поведение — сервер `data-sound-name` не переприсваивает. Подтверждено верификатором (confidence high, verified).
- **Влияние:** пользователь выключает звук/десктоп или меняет пресет, остаётся на `/settings` (или держит открытую вкладку чата) — и chime/баннер продолжают срабатывать по старым настройкам до перезагрузки. В параллельных вкладках изменение не применяется вовсе (нет бродкаста).

### R029 · P2/product · `lib/eden/chat.ex:1063-1084`, `lib/eden/channels.ex:401-432` — «стук» в приватную комнату полностью беззвучен

- **Проблема:** `create_system_message/2` делает только `broadcast(conversation_id, {:new_message, message})` и НЕ вызывает `notify_new`. Единственное actionable системное сообщение — join-request (`request_room_join` → `post_join_request` → `create_system_message` с `action: "join_request"`) — не даёт админам/владельцу ни chime, ни OS-баннера.
- **Доказательство в коде:**
  ```elixir
  # chat.ex:1076-1083 — только broadcast, без notify_new
  |> case do
    {:ok, message} ->
      broadcast(conversation_id, {:new_message, message})   # ← нет notify_new(message)
      {:ok, message}
    error -> error
  end
  ```
  ```elixir
  # channels.ex:422 — knock кладётся именно как system-message
  Chat.create_system_message(room_id, %{
    "action" => "join_request",
    "requester_id" => user.id,
    "requester_name" => user.display_name,
    "status" => "pending"
  })
  ```
  Дополнительно `notify_payload` жёстко требует `%{sender: %User{}}` (chat.ex:3872) — у системного сообщения `sender_id` = nil, так что «в лоб» звать `notify_new` на нём нельзя, нужен отдельный путь адресатов + payload. confidence medium, advisory.
- **Влияние:** админ приватной комнаты узнаёт о заявке, только если сам откроет комнату; постучавшийся может ждать одобрения часами/днями — прямой провал ключевого флоу корпоративного слоя (#41).

### R120 · P3/product · `lib/eden/chat.ex:255, 277, 1280` — «стук» не даёт unread-сигнала админам

- **Проблема:** все unread-запросы отсекают системные сообщения через `m.sender_id != ^user.id` (у system-сообщения `sender_id` = NULL, NULL-сравнение не проходит), поэтому knock не бампает ни бейдж комнаты, ни рейл канала; `create_system_message` также не трогает сортировку и не пингует `notify_members`.
- **Доказательство в коде:**
  ```elixir
  # chat.ex:255 (channel_unread_counts), :277 (unread_counts), :1280 (folder badges)
  where: m.sender_id != ^user.id and is_nil(m.deleted_at) and is_nil(m.root_id),
  ```
  Тот же корень, что у R029: knock — system-сообщение, а вся сигнализация (unread + notify) построена вокруг `%User{}`-отправителя. confidence medium, advisory. Это оборотная (badge) сторона R029; фиксить их стоит вместе.
- **Влияние:** заявка на вступление висит незамеченной, пока админ случайно не откроет комнату — провал сценария «доступ по запросу».

### R150 · P3/quality · `lib/eden/chat.ex:3832-3838` — gating не исключает деактивированных/удалённых

- **Проблема:** `common_gates/2` фильтрует sender/left/mute/DND, но НЕ проверяет `u.active` / `is_nil(u.deleted_at)`, хотя join на `User` (`as: :user`) уже есть в обоих базовых запросах.
- **Доказательство в коде:**
  ```elixir
  # chat.ex:3833-3837 — u.active не проверяется
  from([membership: m, user: u] in query,
    where:
      m.user_id != ^message.sender_id and is_nil(m.left_at) and
        is_nil(m.muted_at) and u.presence_status != "dnd"
  )
  ```
  Поля есть: `field :active, :boolean, default: true` (user.ex:38), `field :deleted_at, :utc_datetime` (user.ex:45); `deleted_at` всегда идёт с `active=false` (см. `delete_user_permanently/2`), так что одного `u.active == true` достаточно. Сегодня безвредно (Web-адаптер: у деактивированного нет живой сессии, топик пуст), но по ADR-0001 recipient set — единственное место гейтинга. confidence high, advisory.
- **Влияние:** первый же push-транспорт (desktop/APNs/FCM) начнёт слать уведомления на устройства деактивированных и анонимизированных (#303) пользователей — нарушение смысла деактивации и right-to-erasure, чинить придётся именно здесь.

### R164 · P3/quality · `lib/eden_web/components/notifier.ex:67-73, 108-116` — сфокусированная вкладка частично обходит cross-tab дедуп

- **Проблема:** `onNotify` при `here=true` сразу зовёт `chime()`; `chime()` делает localStorage-дедуп (`ed:lastChime`, 1500ms), так что двойного звона нет, НО фокусная вкладка может звенеть, пока фоновая для того же сообщения показывает OS-баннер (со своим системным звуком) — двойной сигнал.
- **Доказательство в коде:**

- **Влияние:** при нескольких вкладках пользователь может услышать и chime, и системный звук на одно сообщение.

## План реализации

Порядок: сначала changeset/context-гейты и контракт payload (R150, R203, R202), затем notify-путь для knock (R029) вместе с его unread-стороной (R120), затем live-apply настроек (R096); в конце тесты (R108) и мелкие JS-нюансы.

- [ ] **R150 — исключить деактивированных/удалённых в `common_gates`.** В `chat.ex:3832-3838` добавить в `where` условие `u.active == true` (одного флага достаточно — `deleted_at` всегда при `active=false`). Одно изменение, покрывает оба базовых запроса (DM/group и thread).
- [ ] **R203 — не слать клиенту сырой `preview` и `avatar_key`.** В `notify_hook.ex:92-95` после формирования `body`/`avatar_url` вернуть `payload |> Map.merge(%{body: ..., avatar_url: ...}) |> Map.drop([:preview, :avatar_key])` перед `push_event`.
- [ ] **R202 — медиа-маркер в баннере при медиа+подписи.** В `notify_hook.ex:85-90` при `payload.media_kind` собирать `body` как `media_label(kind) <> ", " <> display_preview(preview)`, когда `preview` непустой; иначе — как сейчас. Метка локализуется на стороне получателя.
- [ ] **R029 + R120 — озвучить и посчитать «стук» (общий фикс).** Решить путь адресатов для system-сообщения `action=join_request`:
  - Уведомление: собрать админов/владельца канала комнаты (минус mute/DND через существующий `common_gates`-совместимый путь; **осторожно** — `notify_payload` требует `%User{}`-sender, у knock его нет, поэтому нужен отдельный payload-конструктор, например `kind: "knock"` с локализуемым маркером-preview) и вызвать `Notifications.deliver/2`. Точка вызова — в `Channels.post_join_request/2` (channels.ex:421) после успешного `create_system_message`, а не внутри самого `create_system_message` (он трудовой утилитарный, не должен знать про knock). Так фикс не задевает прочие system-сообщения (member add/remove).
  - Unread/бейдж (R120): либо считать pending `join_request` в отдельный индикатор «N запросов» на комнате/канале для админов, либо явно НЕ бампать общий unread (решить — см. «Заметки»). Не переписывать `m.sender_id != ^user.id` в unread-запросах (chat.ex:255/277/1280) под system-сообщения — это зашумит бейджи всем.
  - Заодно решить, уведомлять ли одобренного requester об одобрении (`approve_room_join`).
- [ ] **R096 — применять смену настроек вживую.** В `settings_live.ex` в трёх обработчиках (`set_notify_sound` :1527, `set_notify_sound_name` :1535, `set_notify_desktop` :1546) после записи в БД переприсваивать `:notify_prefs` (`assign(socket, notify_prefs: Chat.notification_prefs(scope))`), чтобы `<.notifier>` перерендерил `data-*`. Для мультитаба — бродкастить лёгкое `{:notify_prefs_changed}` на `user:<id>:notify` (или отдельный prefs-топик) и в `NotifyHook` обновлять `:notify_prefs` по этому сообщению. Поправить устаревший комментарий notifier.ex:125-126.
- [ ] **R166 — Firefox/Safari: не глушить chime, когда `renotify` не работает.** В `notifier.ex` feature-detect (`"renotify" in Notification.prototype`); при отсутствии — не подавлять chime при показанном баннере (либо не ставить `tag`, ценой стопки баннеров). Взвесить: минимально — только снять подавление chime.
- [ ] **R165 — (по решению) cross-tab focus-suppression.** Аналогично chime-дедупу (#273): фокусная вкладка при focus-suppress пишет `ed:lastFocused-<conv_id>` в localStorage, фоновая перед `new Notification` проверяет свежую метку. ## Критерии приёмки

- Выключение звука/десктопа и смена пресета применяются в текущей вкладке без reload; при `{:notify_prefs_changed}` — и в параллельных (R096).
- «Стук» в приватную комнату даёт админам сигнал (уведомление и/или индикатор «N запросов»); прочие system-сообщения (member add/remove) уведомлений НЕ порождают (R029/R120).
- `common_gates` не включает деактивированных/удалённых (R150); тест это фиксирует.
- Баннер медиа+подписи показывает медиа-маркер (R202); клиентский payload не содержит `:preview`/`:avatar_key` (R203).
- Все новые тесты из раздела выше проходят; сюита ловит регрессии в left/forward/system/media/focus-кейсах (R108).
- `mix check` и `mix dialyzer` зелёные; существующие notify-тесты не сломаны.

## Заметки

- **Развилки для человека (не для AFK-агента без решения):** R144 (уведомлять ли о реакции — продуктовое) и R165 (cross-tab focus-suppression vs дождаться presence-based из ADR-0001). Из-за них общий триаж — `ready-for-agent` для механической части (R096/R150/R202/R203/R108/R166), но эти два пункта помечены «по решению» и их можно вынести/отложить.
- **R029 и R120 — один корень** (knock как system-сообщение вне `%User{}`-sender-контракта), фиксить вместе; unread-сторону (R120) НЕ решать переписыванием `m.sender_id != ^user.id` в общих unread-запросах — это зашумит бейджи для всех, нужен отдельный индикатор.
- **R164 подтверждён как «не баг»** (accepted minor, confidence low) — в issue присутствует только для трассируемости, правки не требует.
- Payload-контракт документирован один раз в moduledoc `Eden.Notifications` (notifications.ex:24-36) — при добавлении `kind: "knock"`/`"reaction"` или удалении полей из клиентского события обновить его.
- Осторожно с `notify_payload/1` (chat.ex:3872): match `%{sender: %User{}}` — жёсткий контракт, любой новый notify-путь для системных/реакционных событий должен идти через отдельный конструктор payload, иначе краш на NotLoaded/nil sender.
