# Клиентская отправка: сохранять failed-ноды при смене чата, in-flight ноды при смене треда и снимать body.overflow при навигации

**Приоритет:** P2 — не крашей и не потери доставленных данных, а UX-краевые случаи: пользователь на флаки-линке теряет ●!/Resend и обратную связь по идущей отправке, а оверлей-синглтон может залипить скролл страницы.
**Лейблы:** `bug` `P2: medium` `UI/UX` `ready-for-agent`
**Находки ревью:** R064, R066, R187

## Контекст

Все три находки — в клиентских хуках отправки/оверлеев `lib/eden_web/live/chat_live.ex` (colocated hooks `.SendQueue`, `.ThreadSendQueue`, `.Lightbox`, `.VideoExpand`). Общий корень — **живучесть DOM-состояния, привязанного к хуку, при server-driven навигации/переключении**: optimistic-ноды и оверлеи создаются вне «своей» LiveView-области (в `#pending-messages` / `#thread-pending` / на `document.body`) и переживают навигацию, а хук-владелец их состояние не восстанавливает и не подчищает.

- R064/R066 — обе про **потерю визуальной ноды отправки при переключении** (чата / треда), где очистка слишком «слепая».
- R187 — про **оверлей-синглтон, переживающий навигацию** с застрявшим `body.overflow:hidden`.

Механизм «media-нода переживает switch, а text-нода — нет» уже реализован для главной панели (#144, `wrapAndAppendOptimistic` тегирует `data-conv-id`, `SendQueue.updated()` прячет чужие/показывает свои). Эти находки закрывают три дыры в том же дизайне.

## Находки и доказательства

### R064 · P2/bug · `chat_live.ex:5974` (первопричина), `:6373`/`:7762` — failed text-ноды слепо удаляются при переключении чата

- **Проблема:** На переключении конверсации `SendQueue.updated()` итерирует по `#pending-messages` и **безусловно удаляет любую ноду без `data-conv-id`**. Текстовые optimistic-ноды (`addOptimistic`) намеренно НЕ тегированы `data-conv-id` (они queue/timer-bound к чату), поэтому текстовая **failed-нода** (с ●! и кнопкой Resend) молча удаляется при уходе из чата и обратно — вместе с `data-body` и аффордансом переотправки недоставленного сообщения.
- **Доказательство в коде:** `addOptimistic` не ставит `convId` (в отличие от `wrapAndAppendOptimistic:6874, row.dataset.convId = this.convId`):
  ```elixir
  // addOptimistic (chat_live.ex:6373) — тегирует clientId и body, но НЕ convId
  row.dataset.clientId = clientId
  row.dataset.body = body
  // …ни flat-, ни bubble-ветка convId не ставят…
  const pend = (this._sendTarget && this._sendTarget.pending) || this.pending
  pend.appendChild(row)
  ```
  Слепое удаление в `SendQueue.updated()` (5972-5986):
  ```elixir
  for (const node of [...this.pending.children]) {
    const conv = node.dataset.convId
    if (!conv) {
      node.remove()                         // ← failed text-нода (.ed-msg-failed) попадает сюда
    } else if (conv !== this.convId) {
      node.style.display = "none"
    } else if (node.dataset.clientId &&
               stream?.querySelector(`[data-client-id="${node.dataset.clientId}"]`)) {
      node.remove()
    } else {
      node.style.display = ""
    }
  }
  ```
  `markFailed` (7762) вешает `.ed-msg-failed` + кнопку `.ed-msg-failed__bang` (→ `openFailMenu`/`resendNode`) на эту же untagged-ноду; `resendNode:7803` восстановления после удаления не даёт. **Уточнение границы (из верификации):** после #351 text-нода рисуется уже на `send` (6280 `this.addOptimistic`), поэтому ветка re-materialize внутри `markFailed` (7768-7771) фактически мертва — нода untagged с момента отправки. Настоящий дефект — в blanket-удалении в `updated()`, а не в `addOptimistic`. Тест-гварды нет: `resilience.spec.js` покрывает лишь happy-path (offline→optimistic→reconnect), без switch и без `.ed-msg-failed`; `bg-upload.spec.js:32` прямо подтверждает, что без `data-conv-id` switch дропает ноду. Уверенность **medium**, статус **verified**.
- **Влияние:** Пользователь на нестабильном линке (где failed-ноды и живут) при переключении чата теряет и ●!, и кнопку Resend — недоставленное сообщение исчезает из UI без отправки.

### R066 · P2/bug · `chat_live.ex:7947` — смена корня треда затирает `#thread-pending` с in-flight optimistic-нодами

- **Проблема:** `ThreadSendQueue.updated()` при смене `data-thread-root` делает безусловный `this.pending.replaceChildren()` (стирает весь `#thread-pending`). Но thread-вложения отправляются владельцем `.SendQueue.threadComposeSend`, который кладёт optimistic-ноды именно в `#thread-pending` и держит их в `seqQueues`/`seqFeeding` (+ `item.file`). Открыв тред B в той же комнате во время незавершённой отправки в тред A, пользователь стирает ноды треда A, хотя загрузка на сервере продолжается.
- **Доказательство в коде:** `ThreadSendQueue.updated()` (7933-7948):
  ```elixir
  if (this.el.dataset.threadRoot !== this.threadRoot) {
    this.threadRoot = this.el.dataset.threadRoot
    this.queue = []
    // …
    if (this.pending) this.pending.replaceChildren()   // ← стирает ВСЁ в #thread-pending
  }
  ```
  где `this.pending = document.getElementById("thread-pending")` (7896). Ноды кладёт `threadComposeSend` (7312), таргетя тред-пейн:
  ```elixir
  this._sendTarget = {
    pending: document.getElementById("thread-pending"),
    scroller: document.getElementById("thread-scroll"),
  }
  // …addOptimisticMedia/addOptimisticFile + seqItems с file: stash.get(key)…
  ;(this.seqQueues = this.seqQueues || []).push({ queueId, items: seqItems })
  this.pumpSeq()
  ```
  При смене тред A→B в **той же комнате** conv id комнаты не меняется, поэтому conv-switch блок главного `.SendQueue.updated()` (5954) не срабатывает и эти ноды не прячет; главная панель по #144 их **прячет** (`display:none`, 5977) и восстанавливает — а тред-панель их **стирает**. Реальный ответ всё же прилетит по `{:thread_reply}`. Краша нет (`setRing`/`armSeqStall` null-safe), потери данных нет. `thread-attach-optimistic.spec.js` покрывает только один тред — сценарий двух тредов не тестируется. Уверенность **medium**, статус **verified**.
- **Влияние:** Пользователь теряет обратную связь (кольца прогресса/карточки) по всё ещё идущей отправке вложения; при возврате в тред A ноды не восстанавливаются — промежуток выглядит как «отправка молча пропала».

### R187 · P3/quality · `chat_live.ex:8330`, `:8477` — `body.style.overflow` может залипнуть при навигации с открытым оверлеем

- **Проблема:** `Lightbox.openLightbox()` и `VideoExpand.open()` ставят `document.body.style.overflow = "hidden"` и снимают его только в `close()` (по Esc / клику по фону). Оверлеи — синглтоны на `document.body` вне LiveView-корня. Если во время открытого оверлея произойдёт server-driven navigate/patch и хук-владелец будет уничтожен, `close()` не вызовется — `overflow:hidden` останется на body новой страницы.
- **Доказательство в коде:** Установка без парного сброса при `destroyed`:
  ```elixir
  // Lightbox.openLightbox (8330)
  document.body.style.overflow = "hidden"
  // VideoExpand.open (8477)
  document.body.style.overflow = "hidden"
  ```
  Сброс `document.body.style.overflow = ""` живёт только в замыкании `close()` каждого синглтона (Lightbox 8357, VideoExpand 8498), а `close` вызывается лишь по Esc/клику. `destroyed()` хуков сброс не делает: `Lightbox.destroyed` (8297-8299) чистит только `this._openT`; `VideoExpand.destroyed` (8519-8522) снимает только click/keydown-листенеры. Сами `box`/`modal` добавлены в `document.body` (8433, 8516) → переживают навигацию, а их открытое состояние — нет. `verdict_note` пуст, уверенность **medium**, статус **advisory** (не подтверждён независимым верификатором, но механизм в коде очевиден: сброс есть только в `close()`, а `close()` навигацией не вызывается).
- **Влияние:** Редкий залипающий скролл — страница перестаёт прокручиваться до следующего открытия/закрытия оверлея.

## План реализации

- [ ] **R064 — исключить failed-ноды из слепого удаления при переключении чата.** В `SendQueue.updated()` (`chat_live.ex:5972-5986`) в ветке `if (!conv)` пропускать ноды с `.ed-msg-failed` (не удалять их: `if (!conv) { if (!node.classList.contains("ed-msg-failed")) node.remove(); continue }` — или эквивалент). Так недоставленная нода с ●!/Resend переживёт переключение. Resend/flush привязаны к `this.convId`, поэтому Resend сработает при возврате в исходный чат; при желании — дополнительно тегировать failed text-ноду её `convId` на пути `markFailed` (7772+, `node.dataset.convId = this.convId`), чтобы она пряталась/показывалась как media-ноды, а не оставалась во всех чатах. Выбрать один из двух подходов (см. Заметки).
- [ ] **R066 — не стирать безусловно `#thread-pending` при смене треда.** В `ThreadSendQueue.updated()` (`chat_live.ex:7947`) заменить безусловный `this.pending.replaceChildren()` на выборочную очистку: удалять только failed-ноды и ноды других тредов, а ноды с активной отправкой (чей `client_id`/`albumCid` ещё числится в `window.__edSendQueue.seqQueues`/`seqFeeding`) — **прятать** (`display:none`) и восстанавливать при возврате, зеркаля дизайн главного `SendQueue.updated()` (#144). Для этого нужно уметь сопоставить ноду с тредом-владельцем: либо тегировать тред-ноды `data-thread-root` в `threadComposeSend` (7323-7326, при таргетинге `_sendTarget`), либо свериться со списком активных `client_id` из `window.__edSendQueue`. Минимальный вариант — не стирать ноды, чей `client_id` ещё в очереди феда.
- [ ] **R187 — снимать `body.overflow` при уходе с открытым оверлеем.** Добавить сброс `document.body.style.overflow = ""` (и закрытие оверлея, если открыт) при уничтожении/навигации. Варианты: (а) в `Lightbox.destroyed` (8297) и `VideoExpand.destroyed` (8519) проверять открыт ли синглтон (`#ed-lightbox.ed-lightbox--open` / `#ed-video-modal.ed-video-modal--open`) и звать его `close()`; (б) добавить один глобальный слушатель `phx:page-loading-start`, закрывающий оба оверлея; (в) как минимум — сбрасывать `overflow` в начале `openLightbox`/`open`, если предыдущий оверлей не закрылся. Предпочтителен (б) — один слушатель покрывает оба оверлея и любую будущую навигацию, не завися от того, какой конкретно хук уничтожается.

Порядок: пункты независимы; R064 и R066 логически парные (оба про «не стирать слепо ноду отправки») — делать их вместе; R187 — отдельно.

## Критерии приёмки

- Failed text-нода (●!/Resend) переживает переключение чата и остаётся переотправляемой (R064).
- Optimistic-ноды незавершённой отправки вложения в тред не стираются при переключении треда в той же комнате, восстанавливаются при возврате, реальный ответ приходит (R066).
- `document.body.style.overflow` никогда не остаётся `"hidden"` после навигации с открытым Lightbox/VideoExpand (R187).
- Новые e2e-кейсы зелёные; регрессий в существующих `resilience`/`thread-attach-optimistic`/`bg-upload` нет.
- `mix check` зелёный (изменения — в colocated hooks внутри HEEx `chat_live.ex`; убедиться, что компиляция и формат проходят).

## Заметки

- **Развилка по R064 (нужно решение исполнителя, не человека):** два эквивалентных фикса — (1) исключить `.ed-msg-failed` из удаления в `updated()` (минимальный, failed-нода будет висеть во всех чатах, пока не переотправлена — но она и так одна, редкий кейс); (2) тегировать failed text-ноду `data-conv-id` в `markFailed`, чтобы она пряталась/показывалась как media-ноды. Вариант (2) чище и консистентнее с #144-дизайном, но требует, чтобы `resend`/`flush` корректно отрабатывали только в «своём» чате (они уже привязаны к `this.convId`). Рекомендуется (1) как минимальный, либо (2) при желании единообразия — оба закрывают дефект.
- Все три находки подтверждены чтением актуального кода; **номера строк совпадают со входом, дрейфа нет**. R064 — реальный дефект в `updated()` (не в `addOptimistic`, как гласил исходный тезис находки: см. уточнение границы в верификации). R187 — статус advisory (не проходил независимую верификацию), но механизм однозначно виден в коде.
- Связано по духу с #144 (media-нода переживает switch) и #351 (optimistic на каждой поверхности) — тот же дизайн живучести optimistic-состояния; эти пункты доводят его до failed-нод и тред-панели.
