# Сгладить тихие/непоследовательные жесты в диалоге: скрытый 1:1, реакции, reply, dbl-click, свайп и задержка lightbox

**Приоритет:** P3 — набор UX-шероховатостей в слое взаимодействия с сообщениями (тихие no-op'ы, ложные срабатывания жестов, ощутимая задержка); ни одна находка не про краш/потерю данных/некорректность границ, поэтому по рубрике это P3 (стиль/мелочи UX).
**Лейблы:** `enhancement` `P3: low` `backend` `UI/UX` `ready-for-human`
**Находки ревью:** R126, R145, R175, R179, R184, R186

## Контекст
Все шесть находок живут в одном месте — слой взаимодействия с сообщениями в диалоге (`Eden.Chat` + хуки `ChatLive`: `.ContextMenu`, `.Lightbox`). Общая нить — **непоследовательный или тихий отклик на пользовательский жест**: скрытый после «удалить чат» 1:1 частично выпадает из UI (поиск, реакции), клик Reply в редкой гонке молча ничего не делает, а несколько жестов срабатывают там, где пользователь этого не хотел (двойной клик всегда реагирует, трекпад-свайп может ложно ответить, тап по фото «думает» четверть секунды). По отдельности каждая — мелочь P3; вместе они дают ощущение «жест сработал не так, как я ждал, и без объяснения». Две находки лежат в контексте (`lib/eden/chat.ex`), четыре — в JS-хуках `ChatLive`.

Корень для R126 + R145 один и тот же: контекст **сознательно пермиссивен к `left_at` для 1:1** в путях чтения/отправки (`access_query/2`, `get_conversation/2`), но **строг в двух других путях** (`search_*`, `ensure_active_member/2`) — рассинхрон предикатов.

## Находки и доказательства

### R126 · P3/ux · `lib/eden/chat.ex:1608,1636` — скрытый 1:1 выпадает из поиска
- **Проблема:** после «удалить чат» (1:1 помечается `left_at`) ни беседа, ни её сообщения, ни сам собеседник не находятся глобальным поиском — хотя открыть чат по URL/профилю и написать (resurface) можно.
- **Доказательство в коде:** оба поисковых запроса требуют `is_nil(my.left_at)` / `is_nil(mem.left_at)`:
  ```elixir
  # search_conversations/2 (строка 1608)
  join: my in Membership,
  on: my.conversation_id == c.id and my.user_id == ^user.id and is_nil(my.left_at),
  ```
  ```elixir
  # search_messages/2 (строка 1631)
  join: mem in Membership,
  on:
    mem.conversation_id == m.conversation_id and mem.user_id == ^user.id and
      is_nil(mem.left_at),
  ```
  При этом путь отправки/чтения намеренно пермиссивен к `left_at` для 1:1 (`access_query/2`, строки 3658–3665: `is_nil(m.left_at) or not (c.is_group and is_nil(c.channel_id))`), а `resurface_direct/1` (строка 4009) переоткрывает 1:1 при новой активности.
- **Влияние:** типовой сценарий «удалил чат → через неделю ищу человека по имени, чтобы написать снова» даёт пустой результат; пользователь вынужден догадаться про new-chat picker. В Telegram контакт остаётся находимым после удаления чата. Уверенность находки — medium, статус advisory (не верифицировано независимо).

### R145 · P3/ux · `lib/eden/chat.ex:3291` — в скрытом 1:1 реакции молча не работают до resurface
- **Проблема:** в открытом (через поиск/профиль) скрытом 1:1 отправка текста работает и resurface'ит чат, а тап по эмодзи-реакции молча ничего не делает — рассинхрон предикатов доступа.
- **Доказательство в коде:** `toggle_reaction/3` требует активного членства через `ensure_active_member/2`, который строг к `left_at`:
  ```elixir
  # ensure_active_member/2 (строка 3291) — строгий предикат
  defp ensure_active_member(%Scope{user: user}, conversation_id) do
    active? =
      Repo.exists?(
        from m in Membership,
          where:
            m.conversation_id == ^conversation_id and m.user_id == ^user.id and
              is_nil(m.left_at)
      )
    if active?, do: :ok, else: {:error, :not_found}
  end
  ```
  При скрытом 1:1 это возвращает `{:error, :not_found}`, а web-слой глотает его как no-op:
  ```elixir
  # chat_live.ex:2087–2091
  {:error, reason} ->
    Logger.debug("react rejected: #{inspect(reason)} (message #{inspect(id)})")
    {:noreply, socket}
  ```
  Отправка того же чата проходит, потому что `has_access?/2` → `access_query/2` (строки 3655–3665) пермиссивен к `left_at` для 1:1. Уверенность medium, статус advisory.
- **Влияние:** непоследовательный UX — в одном и том же открытом чате отправка работает, а реакция молча умирает; выглядит как баг.

### R175 · P3/quality · `lib/eden_web/live/chat_live.ex:2101` — Reply на исчезнувшее сообщение без фидбэка
- **Проблема:** в редкой гонке (меню/свайп открыты, сообщение удалено broadcast'ом) клик Reply приводит к тихому no-op: композер сфокусировался, а цитата не появилась и объяснения нет.
- **Доказательство в коде:** пункт меню Reply (строка 10823–10830) **не** сгейтчен по `deleted_at` — в отличие от Edit, который гейтит `is_nil(@message.deleted_at)` (строка 10845). А обработчик молча глотает `nil`:
  ```elixir
  # chat_live.ex:2101–2106
  def handle_event("reply", %{"id" => id}, socket) do
    case Chat.get_message(socket.assigns.current_scope, id) do
      nil -> {:noreply, socket}
      message -> {:noreply, assign(socket, reply_to: message)}
    end
  end
  ```
  `Chat.get_message/2` (chat.ex:2073) возвращает `nil` для удалённого/невидимого. В проекте уже есть готовый флеш ровно с нужной копирайт-строкой:
  ```elixir
  # chat_live.ex:2272–2273
  def handle_event("message_unavailable", _params, socket),
    do: {:noreply, put_flash(socket, :error, gettext("That message is unavailable."))}
  ```
  Практический риск мал (удалённая строка обычно уходит из DOM). Уверенность low, статус advisory.
- **Влияние:** редкий, но сбивающий с толку тихий провал — свайп/арроу/меню сфокусировали композер, а reply_to не выставился без всякого сообщения.

### R179 · P3/ux · `lib/eden/chat.ex:3155` — двойной клик по сообщению всегда реагирует, отключить нельзя
- **Проблема:** любой двойной клик по строке сообщения ставит реакцию — даже если пользователь ничего не настраивал; способа выключить поведение нет.
- **Доказательство в коде:** `dbl_click_reaction/1` **никогда** не возвращает nil — при отсутствии/невалидной настройке падает на `hd(quick_reactions(scope))`, а `quick_reactions/1` сам фолбэчит на непустой дефолтный набор:
  ```elixir
  # chat.ex:3155–3164
  def dbl_click_reaction(%Scope{user: user} = scope) do
    stored =
      Repo.one(from p in FolderPrefs, where: p.user_id == ^user.id, select: p.dbl_click_reaction)
    if is_binary(stored) and stored in MessageReaction.allowed() do
      stored
    else
      hd(quick_reactions(scope))          # всегда валидный эмодзи
    end
  end
  ```
  Значение всегда попадает в DOM (`chat_live.ex:3598` `data-dbl-react={@my_dbl}`), и хук всегда реагирует, если `emoji` непуст (`chat_live.ex:5176–5179`). В настройках (`settings_live.ex:884–900`) это `role="radiogroup"` из эмодзи — **пункта «выключено» нет**, можно только выбрать другой эмодзи. Зона двойного клика — вся строка `.ed-msg` (`chat_live.ex:5166` `this.el.closest(".ed-msg")`), т.е. двойной клик даже по пустому месту рядом со своим bubble реагирует. Уверенность medium, статус advisory.
- **Влияние:** двойной клик — частый непреднамеренный жест (особенно на тексте при выделении слова); пользователь неожиданно расставляет реакции и не может отключить фичу.

### R184 · P3/ux · `lib/eden_web/live/chat_live.ex:5294` — trackpad wheel-свайп может ложно ответить
- **Проблема:** горизонтально-доминантная прокрутка трекпадом над DM-бабблом может накопить достаточно смещения и ложно вызвать quote-reply.
- **Доказательство в коде:** passive `wheel`-листенер на `.ed-bubble` копит `wx` без порога направления/скорости и без сброса до тишины `SETTLE`:
  ```elixir
  # chat_live.ex:5294–5312
  this.el.addEventListener("wheel", (e) => {
    if (Math.abs(e.deltaX) <= Math.abs(e.deltaY)) return   // только dominant-X
    ...
    wx = Math.max(0, wx + e.deltaX)                          // накапливается без сброса
    this.el.style.transform = `translateX(${pull(-wx)}px)`
    if (!this._wheelFired && wx >= SWIPE) {                  // SWIPE = 56 (строка 5187)
      this._wheelFired = true
      fireReply()
      ...
    }
  }, { passive: true })
  ```
  Порог только `SWIPE=56` по накопленному `wx`; серия небольших горизонтальных дельт (инерция после диагонального флика, горизонтальный скролл внутри широкого медиа) складывается до 56 и стреляет. Уверенность low, статус advisory.
- **Влияние:** ложное срабатывание reply на трекпаде — пользователь просто скроллил, а открылся ответ на сообщение.

### R186 · P3/ux · `lib/eden_web/live/chat_live.ex:8284` — открытие фото отложено на 250 мс ради dbl-клика
- **Проблема:** любой первый тап/клик по фото в ленте открывает lightbox с задержкой в 250 мс, чтобы оставить шанс второму клику (react).
- **Доказательство в коде:** `.Lightbox` внутри сообщения откладывает `openLightbox` на `DBL_MS=250`:
  ```elixir
  // chat_live.ex:8284–8295
  const DBL_MS = 250
  ...
  const inMsg = !!this.el.closest(".ed-msg, .ed-flat")
  this.el.addEventListener("click", (e) => {
    ...
    if (!inMsg) return this.openLightbox()                 // профиль — мгновенно
    if (e.detail > 1) return clearTimeout(this._openT)     // dbl → react
    clearTimeout(this._openT)
    this._openT = setTimeout(() => this.openLightbox(), DBL_MS)
  })
  ```
  Это применяется к каждому одиночному тапу по фото в диалоге (`inMsg=true`). Компромисс задокументирован в комментарии. Уверенность low, статус advisory.
- **Влияние:** ощутимая вялость при открытии фото — каждый первый тап «думает» четверть секунды.

## План реализации
- [ ] **(Backend, R126) Смягчить видимость скрытого 1:1 в поиске.** Продуктовая развилка (см. «Заметки») — выбрать один вариант: (а) в `search_conversations/2` (`lib/eden/chat.ex:1602`) не фильтровать 1:1 по `left_at` (оставить фильтр `is_nil(my.left_at)` только для групп/комнат), симметрично поисковой пермиссивности `access_query/2`; **или** (б) добавить отдельную секцию «Люди» по всем расшаренным пользователям. `search_messages/2` (строка 1627) и группы фильтр по `left_at` **сохраняют** (удалённая история 1:1 не должна всплывать телом сообщений).
- [ ] **(Backend, R145) Устранить рассинхрон предикатов для 1:1-реакций.** В `toggle_reaction/3` для 1:1 смягчить `ensure_active_member/2` (`lib/eden/chat.ex:3291`) до предиката `access_query/2` (тот же, что у `has_access?/2`), чтобы реакция в скрытом 1:1 работала (и, как и отправка, могла resurface'ить чат). Для **групп и комнат** оставить строгую проверку `is_nil(left_at)` без изменений (реакция от ушедшего участника не должна улетать оставшимся — сохранить инвариант из комментария на строке 3287). Сделать этот шаг заодно с R126 — оба про один рассинхрон `left_at`.
- [ ] **(UI, R175) Явный фидбэк при Reply на исчезнувшее сообщение.** В `handle_event("reply", ...)` (`lib/eden_web/live/chat_live.ex:2101`) заменить `nil -> {:noreply, socket}` на флеш `gettext("That message is unavailable.")` (переиспользовать строку из `message_unavailable`, строка 2273 — msgid уже в PO, gettext.extract не тронет). Опционально дополнительно сгейтить пункт меню Reply (строка 10823) по `is_nil(@message.deleted_at)`, как уже сделано для Edit (строка 10845), — но серверный флеш обязателен как страховка от гонки.
- [ ] **(Backend + UI, R179) Дать возможность выключить double-click-to-react.** Ввести sentinel «выключено» в prefs: `dbl_click_reaction/1` (`lib/eden/chat.ex:3155`) должен уметь вернуть `nil` (или атом-сентинел), когда пользователь явно выбрал «off» — отличив «не настроено → дефолт» от «явно выключено» (потребует различимого хранения в `FolderPrefs.dbl_click_reaction`, напр. отдельный флаг или зарезервированное значение). При «выключено» `data-dbl-react` (`chat_live.ex:3598`) пуст → хук `.ContextMenu` (`chat_live.ex:5176–5177`) уже делает `if (!emoji) return`, так что клиент сам станет no-op. В `settings_live.ex:884–900` добавить в radiogroup пункт «Выкл.» (`set_dbl_reaction` с пустым/сентинел-значением). Опционально ограничить зону двойного клика самим `.ed-bubble`/содержимым вместо всей `.ed-msg` (`chat_live.ex:5166`).
- [ ] **(UI, R184) Ужесточить wheel-свайп-to-reply.** В wheel-листенере (`lib/eden_web/live/chat_live.ex:5294`) добавить минимальный порог старта жеста (первая дельта > небольшого порога, иначе не начинать накопление), сбрасывать `wx` при первом вертикально-доминантном событии, и игнорировать wheel над интерактивными потомками (`video`, ссылки). Проверить на реальном Firefox + трекпад (см. memory «Verify on Firefox»).
- [ ] **(UI, R186) Убрать задержку открытия lightbox по умолчанию.** В `.Lightbox` (`lib/eden_web/live/chat_live.ex:8284`) открывать lightbox сразу на первом клике/`pointerup` и закрывать при обнаружении второго клика (react выигрывает и закрывает lightbox), вместо предварительной задержки на все открытия — **или** применять `DBL_MS`-задержку только когда dbl-react реально включён (данные о состоянии появятся после R179: пустой `data-dbl-react` → задержка не нужна). Второй вариант естественно вытекает из R179, поэтому делать после него.

## Критерии приёмки
- Выбранные фиксы реализованы; `mix check` зелёный (format / compile --warnings-as-errors / credo --strict / sobelow / deps.audit / test), `mix dialyzer` зелёный.
- R126/R145: скрытый 1:1 ведёт себя последовательно — если пользователь может открыть/написать в него, то и найти собеседника, и поставить реакцию можно; групповые/комнатные инварианты `left_at` не нарушены.
- R175: Reply на исчезнувшее сообщение даёт явный флеш, а не тихий no-op.
- R179: двойной клик по сообщению можно отключить в настройках; при «выключено» жест — no-op.
- R184: горизонтальная трекпад-прокрутка над бабблом не вызывает ложный reply (проверено на Firefox).

- Нет регрессий в существующих сценариях реакций/reply/свайпа/lightbox.

## Заметки
- **Продуктовые развилки (почему `ready-for-human`):**
  - **R126** — «показывать ли скрытый 1:1 в поиске» и **в каком виде** (расфильтровать беседы vs отдельная секция «Люди») — продуктовое решение о приватности/ожиданиях; выбор влияет и на R145.
  - **R179** — вводить ли «выключено» и как хранить состояние (сентинел vs отдельный флаг в `FolderPrefs`) — дизайн-решение по данным + UX; плюс дефолт (dbl-react включён или выключен «из коробки»).
  - **R186** — менять ли поведение lightbox глобально или только когда dbl-react выключен — зависит от исхода R179.
- **Связка R126 ↔ R145 ↔ R179 ↔ R186:** R145 естественно закрывается тем же решением, что и R126 (единый предикат `left_at` для 1:1); R186 естественно упрощается после R179 (пустой `data-dbl-react` → задержка не нужна). Порядок в плане это учитывает.
- **Уверенность находок:** R175, R184, R186 — confidence **low**, все шесть — status **advisory** (не верифицированы независимым верификатором). При чтении кода все шесть подтвердились фактами (см. цитаты выше); ни одна не оказалась ложной. Практический риск R175 мал (удалённая строка обычно уходит из DOM до клика).
- **Область работ:** R126/R145/R179 затрагивают контекст `lib/eden/chat.ex` (+ настройки), R175/R179/R184/R186 — JS-хуки/шаблоны `ChatLive`; отсюда лейблы `backend` + `UI/UX`. JS-хуки в этом проекте — colocated-хуки внутри `chat_live.ex` (dev-сервер их не всегда пересобирает — см. memory «TG-attachments phase F»), проверять на реальном Firefox.
