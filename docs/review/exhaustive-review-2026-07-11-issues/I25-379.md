# Починить рассинхроны стрима сообщений: мульти-селект, слитые пузыри, сниппет поиска, rename в комнате, сохранение аватара

**Приоритет:** P2 — все шесть находок дают краевые UX-рассинхроны (фантомный счётчик выделения, разваливающийся слитый пузырь, нерелевантный сниппет, протухшее имя, «тихо» сохранённый аватар) без потери данных, краша или нарушения корректности persisted-состояния.
**Лейблы:** `bug` `P2: medium` `UI/UX` `backend` `ready-for-agent`
**Находки ревью:** R056, R058, R070, R071, R077, R095

## Контекст

Шесть дефектов вокруг **живой перерисовки стрима сообщений и связанного UI-состояния** в `EdenWeb.ChatLive` (плюс один в `SettingsLive`). Общий корень у большинства — расхождение между тем, что показывает уже отрисованный DOM/бар, и серверной истиной, когда строка ре-стримится по независимому событию (`{:read}`, `{:reaction_changed}`, `{:message_edited}`, `{:message_deleted}`, `{:user_updated}`): виртуальные render-флаги (`group_pos`, `compact`) и JS-навешенная подсветка выделения теряются, а серверные множества (`@selection`, `@pending_forward`) не прунятся. R070 (сниппет поиска) и R095 (порядок consume-аватара vs валидация) — родственные UI/state-несогласованности в тех же файлах, поэтому собраны в один issue. Потери данных ни в одном случае нет — источник истины на сервере корректен; чинится презентационный слой.

## Находки и доказательства

### R056 · P2/bug · `lib/eden_web/live/chat_live.ex:2432` — удаление/скрытие сообщения не прунит `@selection` и `@pending_forward`

- **Проблема:** `handle_info({:message_deleted, message})` (и близнец `{:message_hidden}` на 2465) удаляют строку из стрима и чистят `group_pos`/`compacts` через `forget_row/2`, но НЕ удаляют `message.id` из `@selection` (MapSet) и не прунят `@pending_forward`. Если сообщение, отмеченное в мульти-селекте, удаляется другим участником (delete-for-both → broadcast) или другой вкладкой (delete-for-me → user-topic), его id остаётся в множестве.
- **Доказательство в коде:**
  ```elixir
  # chat_live.ex:2432
  def handle_info({:message_deleted, message}, socket) do
    if open?(socket, message.conversation_id) do
      {:noreply,
       socket
       |> stream_delete_by_dom_id(:messages, "messages-#{message.id}")
       ...
       |> forget_row(message.id)          # чистит только group_pos/compacts (12733)
       |> ThreadPanel.refresh_thread_list()}
  ```
  Бар выделения пересчитывает счётчик из `@selection` на каждом ре-рендере — `selection_bar/1`, `chat_live.ex:10151`:
  ```elixir
  assigns = assign(assigns, :count, MapSet.size(assigns.selection))
  ```
  и отражает набор в `data-selected` (10159). Заголовок confirm-листа берёт число из `delete_prompt` (1955): `sel_delete: %{count: MapSet.size(sel), ...}`. Обе величины считают удалённый фантом.
- **Влияние:** пользователь видит «выбрано 3», хотя одно сообщение уже исчезло из ленты; «Delete N messages?» тоже завышен. Корректность сохранена: `for_all` в `delete_prompt` (1951-1953) честно пересчитывается через `Chat.get_messages`, `delete_selection` ре-фетчит и ре-чекает авторство, а `forward_message/4` имеет `ensure_not_deleted` — удалённый id пересылается безопасно. Дефект чисто UX-косметический. `status: verified`, `confidence: high`.

### R058 · P2/bug · `lib/eden_web/live/chat_live.ex:2527` — прочтение собеседником в DM разваливает слитый файловый пузырь (`group_pos` теряется)

- **Проблема:** DM-ветка `{:read}` (собеседник прочитал) пере-стримит сырой `Chat.list_messages` без восстановления `group_pos`. Это виртуальное поле (`Message`, default `nil`), которое ставит только `mark_group_pos`; `list_messages` его не заполняет. После пере-стрима у всех строк `group_pos = nil`, и шаблон DM-пузыря рисует слитый файловый блок как отдельные пузыри.
- **Доказательство в коде:**
  ```elixir
  # chat_live.ex:2516-2529 (ветка reader_id != scope.user.id)
  socket = assign(socket, other_read_at: read_at)
  if conversation.channel_id do
    {:noreply, socket}
  else
    {:ok, messages} = Chat.list_messages(scope, conversation.id, limit: @page)
    {:noreply, stream(socket, :messages, messages)}   # ← нет restore_group_pos
  end
  ```
  Сравните с корректными путями: `{:message_edited}` (2351) и `restream_message_in_place` (2830) прогоняют строку через `restore_group_pos/2` (12690):
  ```elixir
  defp restore_group_pos(socket, message),
    do: %{message | group_pos: Map.get(socket.assigns.group_pos, message.id, message.group_pos)}
  ```
  Комментарий рядом (2517-2521) осознанно оговаривает близнеца — потерю `compact` для комнат (#155) — но упускает `group_pos` для DM.
- **Влияние:** как только собеседник прочитывает диалог (а `{:read}` шлётся на весь топик при любом `mark_read`), у отправителя в загруженном окне слитый мульти-файловый пузырь распадается на отдельные пузыри — видимый косметический регресс. Само-лечится reload'ом или следующим restore-путём. `status: verified`, `confidence: high`.

### R070 · P2/bug · `lib/eden_web/live/chat_live.ex:9022` — сниппет и подсветка результата ломаются при typo-совпадении (trigram `<%`)

- **Проблема:** `snippet/2` строит окно вокруг ПЕРВОГО вхождения запроса. Но `Chat.body_match` матчит не только по `ILIKE`-подстроке, а и по word-similarity `term <% body` (типо-толерантность), где запрос НЕ является буквальной подстрокой тела. Тогда `String.split(body, q)` возвращает всё тело целиком → окно уезжает в конец сообщения.
- **Доказательство в коде:**
  ```elixir
  # chat_live.ex:9022
  defp snippet(body, query) do
    body = Markup.strip(body)
    q = String.downcase(String.trim(query))
    before = body |> String.downcase() |> String.split(q, parts: 2) |> hd()
    start = max(String.length(before) - 24, 0)   # при typo before == всё тело → хвост
    prefix = if start > 0, do: "…", else: ""
    prefix <> String.slice(body, start, 110)
  end
  ```
  `highlight_parts/2` (8995) делит по `Regex.escape(q)` — при опечатке ничего не находит, `<mark>` не проставляется. (Актуальная строка split — 9027; в находке цитировался вариант с отдельной переменной `body_down`, поведение идентично.)
- **Влияние:** для результатов, найденных по опечатке (ровно ради чего добавляли trigram-поиск), превью показывает нерелевантный хвост без подсветки. Сами результаты поиска корректны — дефект только в презентации превью на fuzzy-пути. `status: verified`, `confidence: high`.

### R071 · P2/bug · `lib/eden_web/live/chat_live.ex:10192` — выбранное сообщение теряет подсветку при live-перерисовке строки

- **Проблема:** `.SelectSync` отражает серверный `@selection` на строки стрима чисто JS-ом (класс `--selected` + `aria-pressed`) и делает это только в `mounted()/updated()` самого `#selbar`. `#selbar` перерисовывается лишь при изменении `@selection`/`@confirming`. Когда выбранная строка ре-стримится по независимому событию, morphdom пересоздаёт её DOM по серверной версии, где `select_overlay` всегда рендерит `aria-pressed="false"` (10122) и класс `--selected` отсутствует — подсветка слетает, а `sync()` не вызывается.
- **Доказательство в коде:**
  ```elixir
  # chat_live.ex:10192 (хук висит на #selbar, 10157, не на строках)
  updated() { this.sync() },
  ```
  Подсветка навешивается только внутри хука (10218-10219), в разметке строк (`.ed-flat`, `.ed-msg`) привязки `@selection → --selected` нет. Ре-стрим строки происходит из `{:reaction_changed}` (2496 → `restream_message_in_place` → `stream_insert(:messages)`), `{:message_edited}` (2358), `{:read}` DM (2528), `{:thumbnail_ready}` (2488) — при этом `@selection` не меняется (`toggle_select` меняет его лишь при явном выборе), так что `#selbar` не патчится и `sync()` не срабатывает. Прецедент фикса — `.DateRail`, которая вешает `MutationObserver` ровно для этой категории.
- **Влияние:** в активном чате (частые реакции, тики прочтения, догрузка превью) отмеченные сообщения визуально «снимаются» с выбора, хотя сервер их всё ещё считает выбранными. Forward/Delete отработают по фактическому набору (потери данных нет), но пользователь может по устаревшему виду перекликнуть строку. Формулировка «удалятся невыбранные» слегка преувеличена — набор реально серверный. `status: verified`, `confidence: high`.

### R077 · P2/bug · `lib/eden_web/live/chat_live.ex:12918` — переименование участника комнаты не обновляет имя в плоских строках открытой комнаты

- **Проблема:** `reload_selected/2` пере-стримит `:messages` только для группы (`is_group: true`); для комнаты срабатывает fall-through `{:ok, fresh}`, который делает лишь `assign(selected: fresh)` и НЕ пере-стримит. Плоские строки комнаты рисуют имя из `message.sender.display_name` (preload-снимок), поэтому уже отрисованные некомпактные строки держат старое имя до ре-маунта комнаты.
- **Доказательство в коде:**
  ```elixir
  # chat_live.ex:12910
  defp reload_selected(socket, conv) do
    case Chat.get_conversation(scope, conv.id) do
      {:ok, %{is_group: true} = fresh} ->
        {:ok, messages} = Chat.list_messages(scope, conv.id, limit: @page)
        socket |> assign(selected: fresh) |> stream(:messages, messages, reset: true)
      {:ok, fresh} ->
        assign(socket, selected: fresh)   # ← комната: имя в строках не обновляется
      {:error, _} -> socket
    end
  end
  ```
  Путь: `update_profile` → broadcast `{:user_updated}` (accounts.ex:517) → ChatLive подписан → `handle_info({:user_updated})` (2813) → `refresh_selected_for` → `reload_selected`. Для 1:1 DM имя обновляется через шапку (assign), для группы — через re-stream; комнаты выпали.
- **Влияние:** сменил участник комнаты `display_name` — у остальных с открытой комнатой некомпактные строки показывают старое имя до переоткрытия. Асимметрия с группами/DM. **Важно:** наивный фикс (сырой re-stream, как у групп) воспроизведёт #155 — потеряет виртуальные `compact`/`group_pos`. `status: verified`, `confidence: medium`.

### R095 · P2/bug · `lib/eden_web/live/settings_live.ex:1281` — аватар сохраняется, даже если сохранение профиля упало на валидации

- **Проблема:** в `save_profile` сначала безусловно вызывается `consume_avatar/1` (персистит новый `avatar_key` в БД и удаляет старый blob), и только потом `Accounts.update_profile/2`. Если форма профиля не проходит валидацию (например, очищено `display_name`), возвращается `{:error, changeset}`, рендерится ошибка формы — но аватар УЖЕ записан.
- **Доказательство в коде:**
  ```elixir
  # settings_live.ex:1281
  def handle_event("save_profile", %{"user" => params}, socket) do
    {user, avatar_error} = consume_avatar(socket)          # ← персистит аватар ДО валидации
    case Accounts.update_profile(user, params) do
      {:ok, updated} -> ...
      {:error, changeset} ->
        {:noreply, assign(socket, profile_form: to_form(changeset))}  # только ошибка формы
    end
  end
  ```
  `consume_avatar → Accounts.set_avatar` (accounts.ex:495-504) делает `Repo.update()` (пишет `avatar_key`) + best-effort удаляет старый blob + broadcast. `profile_changeset` (user.ex:136-144) содержит `validate_display_name` (required). Аватар и `display_name` — в одной форме `#profile-form`, так что один Save несёт оба.
- **Влияние:** пользователь видит «сохранение не удалось» (подсвечена ошибка `display_name`), но аватар молча поменялся в БД, а старый blob удалён; UI при этом показывает старый `avatar_key` из устаревшего `socket.profile_user` — расхождение UI/БД. `status: verified`, `confidence: high`.

## План реализации

- [ ] **R056 — прунить `@selection` и `@pending_forward` при удалении/скрытии сообщения.** В `handle_info({:message_deleted, message})` (`chat_live.ex:2432`) и `handle_info({:message_hidden, conversation_id, message_id, group_id})` (`chat_live.ex:2465`) при непустом `@selection` удалить id из MapSet и свернуть в `nil` при `size == 0`; аналогично отфильтровать удалённое из `@pending_forward` и вызвать `carry_clear` при пустом остатке. Вынести общий помощник (напр. `prune_removed_message(socket, message_id)`), т.к. логика одинакова в обоих обработчиках. Использовать `message.id` в `{:message_deleted}` и `message_id` в `{:message_hidden}`.

- [ ] **R058 — восстановить `group_pos` в DM-ветке `{:read}`.** В `chat_live.ex:2527` перед `stream/3` прогнать список через `restore_group_pos/2` (образец — `{:message_edited}`, 2351):
      `messages = Enum.map(messages, &restore_group_pos(socket, &1)); stream(socket, :messages, messages)`. Тем самым эта ветка встаёт в один паттерн с остальными re-stream-путями.

- [ ] **R070 — деградировать сниппет к началу тела при отсутствии буквального вхождения.** В `snippet/2` (`chat_live.ex:9022`): если `q` не является подстрокой `body` (typo-путь, `String.split(body_down, q)` вернул целиком), не уезжать в хвост — вернуть начало тела (`prefix = ""`, `String.slice(body, 0, 110)`). Подсветку для fuzzy-случая оставить отсутствующей (набранный терм буквально не встречается — определить корректный `<mark>` нельзя без fuzzy-anchor); в комментарии зафиксировать это решение. Опционально (человеко-развилка, см. Заметки) — вернуть anchor совпадения из БД.

- [ ] **R071 — ре-синхронизировать `.SelectSync` при перерисовке строк, а не только `#selbar`.** В хуке `.SelectSync` (`chat_live.ex:10170`) навесить `MutationObserver` на контейнер `this.c` (`#messages` / `#thread-replies`) и вызывать `sync()` при добавлении/замене узлов (образец — `.DateRail`, 5625); отключать observer в `destroyed()`. `sync()` уже идемпотентен и читает `data-selected` — дополнительного серверного изменения не нужно.

- [ ] **R077 — пере-стримить сообщения комнаты в `reload_selected` с восстановлением виртуальных флагов.** В `reload_selected/2` (`chat_live.ex:12918`) расширить ветку `{:ok, fresh}` для комнат (`channel_id != nil`): перезагрузить `list_messages`, прогнать через `mark_compact/2` + `mark_group_pos/1` и пере-стримить с `reset: true`, обновив `compacts`/`group_pos`/`last_flat`/`last_group` (как в `select_conversation`, 12225-12229) — чтобы имя в плоских строках обновилось, но не воспроизвести #155. Замечание: ветка групп (12914-12916) сейчас делает сырой re-stream без `mark_group_pos`; если она затрагивается, восстановить `group_pos` и там (см. Заметки).

- [ ] **R095 — консьюмить аватар только при валидном профиле.** В `save_profile/2` (`settings_live.ex:1281`) до `consume_avatar` проверить `Accounts.change_profile(user, params)` на `.valid?` (той же валидацией, что `update_profile`); при невалидном changeset рендерить ошибку формы БЕЗ вызова `consume_avatar` — аватар не тронут, blob не удалён, tray сохранён. На валидной ветке — прежний порядок (`consume_avatar` → `update_profile`). Порядок шагов: сперва этот changeset-гард, потом остальное.

## Критерии приёмки

- Все шесть находок закрыты по своим планам; поведение соответствует тест-кейсам выше.
- Счётчик выделения и confirm-заголовок совпадают с фактически видимыми выбранными строками; слитые пузыри держат форму после `{:read}`; сниппет fuzzy-результата начинается с начала тела; подсветка выделения переживает live-ре-стрим; имя в строках комнаты обновляется без потери `compact`/`group_pos`; аватар не сохраняется при провале валидации профиля.
- `mix check` зелёный, `mix dialyzer` зелёный, регрессов в существующих тестах (в т.ч. #155, `multi-select.spec.js`) нет.

## Заметки

- **Развилка R070 (human):** «правильная» подсветка совпадения по опечатке требует fuzzy-anchor. Полное решение — вернуть позицию/фрагмент из БД (`ts_headline` / word-similarity anchor) в `Chat.search`, что заметно шире. Минимальный безопасный фикс (сниппет от начала тела, без подсветки на fuzzy-пути) реализуем механически и покрывает жалобу; расширение до БД-anchor — отдельный follow-up.
- **R077 (confidence: medium)** — единственная находка не-high; проверено, что путь `{:user_updated}` → `reload_selected` реален и имя в комнате берётся только из строк. Ключевой риск фикса — регресс #155 (тест на 2704 требует НЕ терять `compact`/`group_pos` при re-stream комнаты), поэтому фикс обязан восстанавливать виртуальные флаги через `mark_compact`+`mark_group_pos`, а не делать сырой re-stream.
