# Закрыть параллельные дыры в контракте #255 и выровнять `kind == "user"`-гварды в Chat

**Приоритет:** P1 — обход авторизации: вышедший из личной группы участник читает сниппеты истории (`get_messages`) и инъецирует контент правкой старых сообщений (`edit_message`/`edit_message_media`) в группу, из которой удалён; корректность/нарушение инварианта #255, но блворадиус ограничен (свои сообщения / 120 символов на сообщение), поэтому не P0.
**Лейблы:** `bug` `P1: critical` `backend`
**Находки ревью:** R005, R006, R109, R127, R136, R137, R143, R146
**Триаж:** `ready-for-agent`

## Контекст

Все находки — про два соседних, недоприменённых инварианта авторизации в `Eden.Chat` (`lib/eden/chat.ex`):

1. **Гейт активного участия #255.** Вышедший/удалённый участник **личной группы** сохраняет свою membership-строку с проставленным `left_at` (по дизайну — чтобы forward-source/delete-своего работали), строка НЕ удаляется: `remove_group_member/3` → `leave_and_maybe_gc/2` (строки 462, 401-408) делает `set: [left_at: now()]`. Правильный контракт — гейт `has_access?/2` / `access_query/2` (строки 3655-3665), пермиссивный для 1:1 и room, но блокирующий вышедшего из группы. Он применён в `list_messages`, `create_message`, `create_album_message`, `mark_read`, `fetch_attachment`, forward-source/target и т.д. — но **пропущен** в `get_messages/2` (R005) и в edit-путях (R006). Приватный `fetch_message/2` (3404) авторизует только по «строка membership существует» — по комментарию (3433) он «намеренно пермиссивен для edit/delete-своего/reply», и edit-пути опираются лишь на него.

2. **Инвариант `kind == "user"`.** Системные сообщения (`kind == "system"`, `sender_id` nil, полезная нагрузка в `meta`, пустое `body`) не должны участвовать в user-операциях. Room-поиск это фильтрует (`m.kind == "user"` в `room_search_base`, строка 1533), но `search_messages` (DM-поиск, R127), `toggle_reaction` (R143), `forward_message` (R146) и тред-руты (`create_reply`/`create_album_reply`/`follow_thread`/forward-в-тред, R136/R137) — нет. Сегодня их спасают косвенные свойства данных (в DM нет system-сообщений и тредов), но это держится не на гварде.

R109 — недостающие тесты в describe #255 для edit-путей.

Все находки локализованы в `lib/eden/chat.ex`, лечатся однотипно (добавить гвард в `with`-цепочку) и тестируются в одном describe — поэтому сведены в один issue.

## Находки и доказательства

### R005 · P1/bug · `lib/eden/chat.ex:2087` — `get_messages/2` не применяет гейт #255: carry вытягивает сниппеты покинутой группы

- **Проблема:** `get_messages/2` джойнит `Membership` без `is_nil(mem.left_at)`, без джойна на `Conversation` и без `access_query` — в отличие от эталона `fetch_attachment` (3603-3608) и `access_query` (3658-3665).
- **Доказательство в коде:**
  ```elixir
  # lib/eden/chat.ex:2087-2100
  def get_messages(%Scope{user: user}, ids) when is_list(ids) do
    ids = ids |> Enum.map(&safe_id/1) |> Enum.filter(&is_integer/1)

    from(m in Message,
      join: mem in Membership,
      on: mem.conversation_id == m.conversation_id and mem.user_id == ^user.id,
      left_join: d in MessageDeletion,
      on: d.message_id == m.id and d.user_id == ^user.id,
      where: m.id in ^ids and is_nil(m.deleted_at) and is_nil(d.id),
      order_by: [asc: m.id],
      preload: [:sender, :attachments]
    )
    |> Repo.all()
  end
  ```
  Фильтруется только `is_nil(m.deleted_at)` и `is_nil(d.id)` — но membership вышедшего живёт (`left_at` set), поэтому джойн срабатывает и на сообщения, отправленные **после** удаления. Web-путь реален: `forward_rehydrate` (`chat_live.ex:1879`) берёт клиентский список `ids` из sessionStorage без доп-гейта → `carry/2` (`chat_live.ex:14167`) зовёт `Chat.get_messages` → для одиночного сообщения `forward_plaque_label([msg])` (`chat_live.ex:14159`) рендерит `reply_snippet` (`chat_live.ex:9954-9957`): `body |> Markup.strip() |> String.slice(0, 120)`.
- **Влияние:** Бывший участник группы, перебирая id по одному, читает по 120 символов тела каждого сообщения покинутой группы (включая посты после его удаления) через плашку forward. Тот же класс, что закрытая в #255 exfil через forward, но параллельным путём.

### R006 · P1/bug · `lib/eden/chat.ex:2530` — вышедший участник правит свои старые сообщения = инъекция контента в группу

- **Проблема:** `edit_message/3` (2530) и `edit_message_media/5` (2573) авторизуются лишь через `fetch_message` (3404, JOIN `Membership` без `left_at`) + `ensure_sender` + `ensure_editable` (блокирует только tombstone/system). Гейт `has_access?`/#255 не применяется.
- **Доказательство в коде:**
  ```elixir
  # lib/eden/chat.ex:2530-2533
  def edit_message(%Scope{user: user} = scope, message_id, body) do
    with {:ok, message} <- fetch_message(scope, message_id),
         :ok <- ensure_sender(message, user.id),
         :ok <- ensure_editable(message) do
      ...
      broadcast(message.conversation_id, {:message_edited, edited})   # 2550
  ```
  ```elixir
  # lib/eden/chat.ex:3404-3417 — fetch_message: НЕТ фильтра left_at
  Repo.one(
    from m in Message,
      join: mem in Membership,
      on: mem.conversation_id == m.conversation_id and mem.user_id == ^user.id,
      where: m.id == ^id
  )
  ```
  `broadcast/2` шлёт `{:message_edited, edited}` на топик `conversation:<id>` (доставляется оставшимся участникам), сообщение не tombstone → видно в `list_messages`. Показательно, что тот же риск для реакций **закрыт** отдельным гейтом с этим обоснованием:
  ```elixir
  # lib/eden/chat.ex:3287-3301 — комментарий: "reacting from a chat you've left
  # would broadcast to the remaining members — so reactions require live membership."
  defp ensure_active_member(%Scope{user: user}, conversation_id) do
    active? = Repo.exists?(from m in Membership,
      where: m.conversation_id == ^conversation_id and m.user_id == ^user.id and is_nil(m.left_at))
    if active?, do: :ok, else: {:error, :not_found}
  end
  ```
  Для edit — аналогичного гварда нет.
- **Влияние:** Удалённый из группы может переписать текст своего старого сообщения (до `max_body`) или заменить/добавить медиа; правка доставляется оставшимся участникам через `{:message_edited}`. Правка старого = отправка нового контента в группу, из которой человека удалили — прямое противоречие инварианту #255 («can't post text or media, and it never reaches the remaining members»).

### R136 · P3/bug · `lib/eden/chat.ex:2773` — тред-руты и `valid_reply_to_id` допускают system-сообщение

- **Проблема:** `create_reply` (2773), `create_album_reply` (2292) и forward-в-тред (`forward_message/4` с root_id, 3344) проверяют `ensure_root`/`ensure_threaded`/`ensure_not_deleted`, но не `kind` root'а — форженным event можно создать тред под системным «join_request»-сообщением (`reply_count` бампается на нём). `valid_reply_to_id` (2047) → `reply_target_visible?` (2056) не проверяет ни `kind` (цитата system рендерится пустым блоком), ни `is_nil(root_id)` (в главный поток комнаты можно процитировать thread-реплай).

       :ok <- ensure_root(root),          # только root_id == nil, kind не проверяется

- **Влияние:** Непредусмотренные состояния UI: тред-футер на системной плашке, пустые цитаты. Не покрыто тестами, штатным UI недостижимо, но данные становятся некорректными.

## Критерии приёмки

- Вышедший из личной группы участник получает `[]` от `get_messages` и `{:error, :not_found}` от `edit_message`/`edit_message_media`; ни `{:message_edited}`, ни сниппет истории не достигают оставшихся участников (R005/R006).
- Ни один user-путь (`get_messages`, `forward_message`, `toggle_reaction`, тред-руты, цитирование, DM-поиск) не принимает `kind == "system"`-сообщение (R127/R136/R137/R143/R146).
- 1:1-пермиссивность #255 не сломана (messaging back по-прежнему работает; edit своего сообщения в DM разрешён).
- Новые тесты в describe #255 и по kind-гвардам зелёные; регрессий в существующих тестах threads/reactions/forward/search нет.
