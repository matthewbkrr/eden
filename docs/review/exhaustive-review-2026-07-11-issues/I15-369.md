# Довести UX групп и пересылки: честный флеш частичного форварда, выход из группы, авто-переход после knock и гейтинг composer/модалок

**Приоритет:** P2 — это кластер UX-/консистентностных дефектов краевых случаев в мессенджере (частичный сбой пересылки, тупик owner-а при выходе из группы, «немой» онбординг после одобрения knock, молчаливая потеря ввода). Ни один не ведёт к краху или потере уже сохранённых данных, но каждый — заметная фрикция/сюрприз; наивысший приоритет среди находок — P2.
**Лейблы:** `bug` `P2: medium` `UI/UX` `ready-for-agent`
**Находки ревью:** R053, R069, R076, R083, R084, R176, R190

## Контекст
Все находки лежат в одном модуле — `EdenWeb.ChatLive` (`lib/eden_web/live/chat_live.ex`) — и касаются трёх пользовательских потоков поверх `Eden.Chat`: **пересылки** (carry-and-drop + multi-select), **групп** (создание/выход) и **входа в приватную комнату через knock**. Общий корень — обработчики событий и шаблоны, которые обрабатывают только «счастливый путь», а на краевых случаях либо молча теряют ввод/намерение (R053, R176), либо дают неточную/отсутствующую обратную связь (R083/R084, R076), либо оставляют пользователя в тупике/без нужной аффорданс-кнопки (R069, R190). R083 и R084 — одна и та же находка (частичный сбой `drop_forward/3`) с двух сторон и сводятся в один пункт плана.

## Находки и доказательства

### R084 · P2/bug · `lib/eden_web/live/chat_live.ex:14181-14200` — частичный сбой multi-select пересылки: общий флеш + потеря carry
### R083 · P2/quality · `lib/eden_web/live/chat_live.ex:14181-14200` — тот же `drop_forward`: частичный успех молча оставляет часть непересланными
- **Проблема:** `drop_forward/3` мапит `Chat.forward_message/4` по каждому id (независимые транзакции — частичный успех реален), но флеш бинарный: `Enum.all?({:ok,_})` → тишина, иначе один generic error. `pending_forward` (carry) сбрасывается всегда, независимо от результата, так что провалившиеся id нельзя повторить без пере-выбора.
- **Доказательство в коде** (актуальные строки совпадают с находкой):
```elixir
defp drop_forward(socket, conversation_id, root_id \\ nil) do
  %{current_scope: scope, pending_forward: messages} = socket.assigns

  results =
    Enum.map(messages, fn m -> Chat.forward_message(scope, m.id, conversation_id, root_id) end)

  socket =
    socket
    |> assign(pending_forward: nil)          # carry сбрасывается ВСЕГДА
    |> push_event("carry_clear", %{})

  socket =
    if Enum.all?(results, &match?({:ok, _}, &1)),
      do: socket,
      else: put_flash(socket, :error, gettext("Couldn't forward that message."))

  {:noreply, socket}
end
```
  Контраст — `delete_selection` (`:1970-1985`), где считается фактическое `deleted` и флеш честный: `"Deleted."` при `deleted > 0`, иначе ошибка. Пересылка этой честности лишена.
- **Влияние:** при multi-select пересылке нескольких сообщений, где одно с тех пор удалено/недоступно, пользователь видит общий провальный флеш, хотя остальные копии уже приземлились в стрим. Кажется, что не переслалось ничего → повторный клик → дубли. При частичном провале повторить именно неудавшиеся нельзя (carry уже сброшен). Не потеря данных (успешные копии сохранены и видны), а вводящая в заблуждение обратная связь. Верифицировано (`status: verified`, R084); `multi-select.spec.js` покрывает только полный успех, гварды/теста против частичного сбоя нет.

### R069 · P2/ux · `lib/eden_web/live/chat_live.ex:8808-8817` (+ `:1120-1135`, `:11873-11966`) — нет отдельного «Выйти из группы»; owner упирается в тупик
- **Проблема:** единственный путь покинуть группу — пункт «Delete chat» в контекстном меню строки сайдбара, с копией, написанной под DM-скрытие. Для группы это необратимый выход, а не re-surfaceable-скрытие. Owner подтверждает «удаление» и лишь потом узнаёт, что нельзя.
- **Доказательство в коде.** Пункт меню (одинаков для DM и группы — компонент `conversation_item`, `@conversation` несёт `is_group`, но ветвления нет):
```elixir
<button ... phx-click="delete_chat" phx-value-id={@conversation.id}
  data-confirm={gettext("Delete this chat? It will be removed from your list.")}>
  <.icon name="hero-trash-micro" class="size-4" /> {gettext("Delete chat")}
</button>
```
  Обработчик ловит owner-а уже ПОСЛЕ подтверждения:
```elixir
def handle_event("delete_chat", %{"id" => id}, socket) do
  case Chat.delete_conversation(socket.assigns.current_scope, id) do
    :ok -> {:noreply, socket}
    {:error, :owner} ->
      {:noreply, put_flash(socket, :error, gettext("Transfer ownership before leaving the group."))}
    ...
```
  Панель профиля группы (`:11873-11966`) даёт rename / add / per-member «Transfer ownership»/«Remove», но **своей** кнопки «Выйти из группы»/«Delete group» там нет — «Transfer ownership» доступен только как действие над *другим* участником. Контекстно (CLAUDE.md): выход из группы необратим (`resurface_direct/1` un-hide-ит только не-групповые треды), выход owner-а гейтится `ensure_not_group_owner/2` в `Chat.delete_conversation/2`.
- **Влияние:** три сюрприза разом — (1) копия не отличает необратимый выход из группы от временного скрытия DM; (2) owner узнаёт о блокировке только после подтверждения, а аффорданс «передать владение» — на другом экране (панель профиля); (3) сам выход спрятан под «Delete chat». `confidence: medium`, `advisory`.

### R076 · P2/ux · `lib/eden_web/live/chat_live.ex:12501-12511` (+ `:2615-2625`, `:488-497`) — после одобрения knock нет авто-перехода в комнату и позитивного подтверждения
- **Проблема:** после `request_join` UI показывает «Request sent.». Когда админ одобряет — прилетает `{:members_changed}`, `maybe_clear_knock/1` просто зануляет `knock_room`; `selected` остаётся `nil`, пользователь попадает на экран списка комнат/«about» канала, а не внутрь запрошенной комнаты. Тоста «Вам открыли доступ к #room» нет.
- **Доказательство в коде:**
```elixir
defp maybe_clear_knock(%{assigns: %{knock_room: %{id: id}}} = socket) do
  if Chat.room_member?(id, socket.assigns.current_scope.user.id) do
    assign(socket, knock_room: nil, knock_pending: false)   # selected остаётся nil
  else
    socket
  end
end
```
  Вызывается из `handle_info({:members_changed, channel_id}, ...)` (`:2615`) через `refresh_channel_access |> refresh_rooms |> maybe_clear_knock`. Тот же паттерн входа в комнату уже есть — `finish_open_room/3` (`:499-519`) делает `record_last_room` + `select_conversation` + опциональный `focus_message_target`; здесь это не переиспользуется. «Request sent.» рендерится в `:3813-3815`.
- **Влияние:** разрыв онбординга — экран «Request sent» молча исчезает, пользователь сам ищет и кликает новую комнату в сайдбаре; сигнала об успехе нет. `confidence: medium`, `advisory`.

### R176 · P3/ux · `lib/eden_web/live/chat_live.ex:2300-2302` — введённое имя группы молча теряется при выборе одного участника
- **Проблема:** в `handle_event("start")` `opts = [group: true, title]` формируется только при `length(ids) > 1`; при одном выбранном `opts = []`.
- **Доказательство в коде:**
```elixir
def handle_event("start", %{"member_ids" => ids} = params, socket) do
  scope = socket.assigns.current_scope
  opts = if length(List.wrap(ids)) > 1, do: [group: true, title: params["title"]], else: []
  case Chat.create_conversation(scope, List.wrap(ids), opts) do ...
```
  `Chat.create_conversation/3` (`lib/eden/chat.ex:359-364`): `group? = Keyword.get(opts, :group, length(other_ids) > 1)`; при одном участнике и пустых `opts` → `not group? and length == 1` → `find_or_create_direct/2`, `title` отбрасывается. Модалка при этом всегда показывает поле «Group name (optional)» (`chat_live.ex:11444-11450`).
- **Влияние:** пользователь вводит имя группы, отмечает одного человека, жмёт «Start» — получает безымянный 1:1 DM, имя тихо пропадает. Мелкий сюрприз/несогласованность формы. `confidence: medium`, `advisory`.

### R190 · P3/ux · `lib/eden_web/live/chat_live.ex:11473` — «Start» в модалке нового диалога не блокируется без выбранных участников
- **Проблема:** кнопка «Start» (submit) активна всегда, даже когда ни один чекбокс `member_ids[]` не отмечен. Сервер это ловит (`handle_event("start", _params, ...)` → flash «Pick at least one person»), но клиентского `disabled`-состояния нет.
- **Доказательство в коде:**
```elixir
<button class="ed-btn ed-btn--primary w-full" type="submit">{gettext("Start")}</button>
```
  Модалка использует нативные чекбоксы `name="member_ids[]"` без серверного трекинга (`:11456-11461`). В остальных модалках submit гейтится: `room_add` (`:9625`) `disabled={MapSet.size(@selected) == 0}`, forward-media (`:9295`), folders (`:10245/10258/10267`) `disabled={@count == 0}` — но там выбор трекается на сервере (`@selected`/`@count`); здесь такого счётчика нет.
- **Влияние:** можно нажать «Start» вхолостую и получить flash-ошибку вместо очевидного заблокированного действия — лишняя фрикция и несогласованность с остальными модалками. `confidence: high`, `advisory`.

### R053 · P2/ux · `lib/eden_web/live/chat_live.ex:922-930` — Forward при активном pending_forward «съедает» набранный текст и застейдженные вложения
- **Проблема:** `handle_event("send")` при `pending_forward && conversation` безусловно уходит в `drop_forward`, игнорируя `body` и любые застейдженные `:attachment`-вложения — они остаются в overlay, не отправляются и не отменяются.
- **Доказательство в коде:**
```elixir
def handle_event("send", %{"message" => %{"body" => body} = msg}, socket) do
  conversation = socket.assigns.selected
  if socket.assigns.pending_forward && conversation,
    do: drop_forward(socket, conversation.id),   # body + staged attachments отброшены
    else: send_dispatch(socket, body, msg)
end
```
  При этом composer остаётся живым при активном форварде: плашка `pending_forward` (`:3622-3645`) сосуществует с composer bar; input и attach не блокируются (`inert` вешается только на `live_entries(@uploads.attachment) != []`, `:3707`). Карри может пережить навигацию (`.ForwardCarry`), так что пользователь мог о нём забыть.
- **Влияние:** неожиданная потеря ввода — человек мог думать, что отправит и пересылку, и свой текст/фото одним Send; текст/медиа тихо не уходят. `confidence: low`, `status: advisory` — не верифицировано; ниже в «Заметках» развилка по продуктовому решению (гейтить vs. отправлять отдельным сообщением).

## План реализации
- [ ] **Честный флеш частичного форварда (R083 + R084)** в `drop_forward/3` (`lib/eden_web/live/chat_live.ex:14181-14200`): считать `{ok, fail}` из `results` (по образцу `delete_selection` `:1970-1985`). Полный успех — тишина (как сейчас); полный провал — текущий error; частичный — предупреждающий флеш вида `ngettext("Couldn't forward %{count} message.", "Couldn't forward %{count} messages.", fail, count: fail)` (или «Переслано N, не удалось M»). Опционально: при **полном** провале не сбрасывать `pending_forward`/не слать `carry_clear`, чтобы можно было повторить (при частичном carry всё равно сбрасываем, т.к. успешные уже ушли). Один пункт — обе находки указывают на одну функцию.
- [ ] **Разделить «Выйти из группы» и «Delete chat» (R069)** в `conversation_item` (`:8808-8817`): для `@conversation.is_group` показывать пункт «Выйти из группы» с копией, явно предупреждающей о необратимости (напр. `gettext("Leave this group? You can't undo this.")`), для DM оставить текущую копию про скрытие. Для owner-а группы — либо дизейблить пункт с подсказкой «Сначала передайте владение», либо в `data-confirm`/диалоге отсылать к передаче владения. Обработчик `delete_chat` (`:1120-1135`) уже даёт корректный `{:error, :owner}` флеш — менять его не обязательно, но текст можно уточнить. Добавить кнопку выхода и в панель профиля группы (`:11873-11966`, рядом с `@my_role`-действиями), переиспользуя тот же `phx-click="delete_chat"` с id `@conversation.id`.
- [ ] **Авто-переход + позитивный флеш после knock (R076)** в `maybe_clear_knock/1` (`:12503-12509`): когда `Chat.room_member?/2` стал `true`, вместо простого зануления — загрузить комнату и войти в неё, переиспользуя логику `finish_open_room/3` (`:499-519`) или `push_patch` на её route + `put_flash(:info, gettext("You've been given access to #%{room}", room: room.name))`. Сохранить и отдельную клаузу `maybe_clear_knock(socket)` для «knock_room не установлен».
- [ ] **Не терять имя группы при одном участнике (R176)** в `handle_event("start")` (`:2300-2302`): если `params["title"]` непусто, а выбран один участник — либо предупреждать флешем «Для группы выберите минимум двоих», либо (продуктовое решение) разрешить именованную группу из одного участника, передав `[group: true, title: ...]`. Как минимум — не отбрасывать введённое имя молча. См. развилку в «Заметках».
- [ ] **Гейтить submit «Start» без выбора (R190)** в `new_conversation_modal` (`:11443-11473`): дизейблить кнопку, пока не отмечен хотя бы один `member_ids[]`. Т.к. чекбоксы нативные (серверного `@selected` нет), либо (а) добавить лёгкий colocated `phx-hook`, который слушает `change` на чекбоксах и тоглит `disabled` на submit, либо (б) конвертировать выбор в серверно-трекаемый паттерн, как `room_add` (`disabled={MapSet.size(@selected) == 0}`, `:9625`). Согласовать с остальными модалками.
- [ ] **Гейтинг composer при активном форварде (R053)** — ТОЛЬКО после продуктового решения (см. «Заметки»). Вариант-минимум: при `@pending_forward` визуально гейтить/дизейблить composer body + attach (расширить условие `inert` в `:3705-3708`), чтобы намерение «Send = пересылка» было однозначным. Вариант-плюс: если при `pending_forward` есть непустой `body` ИЛИ `live_entries(@uploads.attachment) != []` — после `drop_forward` отправить текст/медиа отдельным сообщением, а не отбрасывать.

## Критерии приёмки
- Частичный сбой пересылки даёт точный флеш (N/M или «часть не удалась»), при полном успехе — тишина; успешные копии всегда видны в стриме (R083/R084).
- Для групп пункт выхода отличается копией от DM-скрытия и явно предупреждает о необратимости; owner не упирается в тупик после подтверждения; кнопка выхода есть и в панели профиля группы (R069).
- После одобрения knock проситель авто-попадает в комнату с позитивным подтверждением (R076).
- Имя группы не теряется молча при одном участнике (R176); submit «Start» гейтится по выбору (R190), согласованно с прочими модалками.
- `mix check` зелёный (format / warnings-as-errors / credo --strict / sobelow / deps.audit / test); `mix dialyzer` зелёный; строки в PO при новых `gettext`/`ngettext` добавлены; регрессий в отправке/пересылке/knock нет.

## Заметки
- **R053 — продуктовая развилка (требует решения человека).** Что должно происходить при `pending_forward` + непустой composer: (а) считать Send = только пересылка и *визуально гейтить* body/attach (минимум, механически); или (б) отправлять пересылку И текст/медиа как отдельные сообщения. Вариант (б) меняет семантику Send при карри — стоит согласовать с дизайном «carry-and-drop». Находка `confidence: low`/`advisory`, при чтении кода подтверждается механически (composer остаётся живым, `send` безусловно уходит в `drop_forward`), но приоритет/поведение — за продуктом. Реализовывать R053 после решения; остальные пункты (R069/R076/R083/R084/R176/R190) — механические и от него не зависят.
- **R176 — тоже мини-развилка:** «предупреждать» vs. «разрешить именованную группу из одного». `Chat.create_conversation/3` уже умеет `group: true` с одним участником (ветка `insert_conversation` при `group?`), так что технически возможны оба; выбор — продуктовый, но по умолчанию безопаснее предупреждать (не создавать «группу из двоих» неожиданно).
- **R083 и R084 — дубликат** (одна функция `drop_forward/3`, два id для трассируемости); сведены в один пункт плана.
- Номера строк сверены с актуальным кодом на момент ревью: `drop_forward` — 14181-14200 (логика флеша 14195-14197); `handle_event("start")` — 2300-2302; `maybe_clear_knock` — 12503-12511; пункт меню «Delete chat» — 8808-8817; submit «Start» — 11473; `send`-роутинг форварда — 922-930. Все совпадают с находками (мелкое уточнение к R084: заголовок указывает 14194, точная логика — 14195-14197).
