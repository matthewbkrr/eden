# Дать `meta` системных сообщений одного владельца в контексте и единый рендер

**Приоритет:** P1 — нарушение границы web↔context: web-слой знает внутреннюю stringly-typed форму jsonb `meta`, которую строят два разных контекста; не краш/потеря данных, но структурная утечка абстракции (сидит на стыке P1/P2 — вредит только при расширении, сегодня форма консистентна).
**Лейблы:** `enhancement` `P1: critical` `backend` `architecture` `ready-for-agent`
**Находки ревью:** R010, R147, R188, R189

## Контекст

Системные сообщения (`messages.kind = "system"`, полезная нагрузка в jsonb-поле `meta`) — это knock-запросы на вступление в приватную комнату (#41) и групповые «X добавлен / удалён» (#165). Форма `meta` (ключи `"action"`, `"name"`, `"requester_id"`, `"requester_name"`, `"status"`, `"user_id"`) — **неявный контракт без единого владельца**, размазанный по двум контекстам (`Eden.Chat`, `Eden.Channels`) и web-слою (`EdenWeb.ChatLive`). Схема `Message` не типизирует и не валидирует его: `field :meta, :map, default: %{}` (message.ex:71), а `create_system_message/2` (chat.ex:1063) — сквозной insert, принимающий любой `is_map(meta)`.

Четыре находки — грани одного корня. R010 (P1) — сама утечка формы и отсутствие владельца. R188/R189 (P3) — прямое следствие в рендере: системные сообщения рисуются **двумя несогласованными путями** (bubble vs flat), причём flat-путь безусловно предполагает join-request. R147 (P3) — родственная хрупкость в том же файле контекста (`forward_message/4` молча игнорирует `target_conversation_id`) — не по `meta`, но лечится тем же «не маскировать рассинхрон вызова» и удобно закрывается в одном PR.

## Находки и доказательства

### R010 · P1/quality · `lib/eden/chat.ex:1063`, `lib/eden_web/live/chat_live.ex:9473,10394` — stringly-typed `meta` без владельца, читаемый web-слоем

- **Проблема:** форма `meta` — неявный контракт, размазанный по 3 слоям без владельца. Пишется в `Eden.Chat` и `Eden.Channels`, читается сырыми `fragment(... ->> ...)` в `Eden.Chat`, и напрямую по строковым ключам рендерится в web-слое. Схема не защищает форму.
- **Доказательство в коде:** схема без типизации/билдера/валидатора:

  ```elixir
  # message.ex:71
  field :meta, :map, default: %{}

  # chat.ex:1063 — сквозной insert, принимает любой meta
  def create_system_message(conversation_id, meta) when is_map(meta) do
    %Message{}
    |> Ecto.Changeset.change(conversation_id: conversation_id, kind: "system", body: "", meta: meta)
    ...
  ```

  Писатели формы (три места, три набора ключей):

  ```elixir
  # chat.ex:477 — member_removed
  create_system_message(id, %{"action" => "member_removed", "name" => name, "user_id" => target_id})
  # chat.ex:576 — member_added
  create_system_message(id, %{"action" => "member_added", "name" => u.display_name, "user_id" => u.id})
  # channels.ex:422 — join_request
  Chat.create_system_message(room_id, %{"action" => "join_request",
    "requester_id" => user.id, "requester_name" => user.display_name, "status" => "pending"})
  ```

  Читатели формы сырыми jsonb-ключами внутри контекстов:

  ```elixir
  # chat.ex:1095 — pending_join_request/2
  fragment("? ->> 'action'", m.meta) == "join_request" and
    fragment("? ->> 'status'", m.meta) == "pending" and
    fragment("(? ->> 'requester_id')::bigint", m.meta) == ^requester_id
  # chat.ex:628 — scrub_meta_name (scrub_deleted_user_content)
  fragment("(?->>?)::bigint", m.meta, ^id_key) == ^user_id   # id_key ∈ {"requester_id","user_id"}
  # channels.ex:440 — approve_room_join матчит на карту
  %{meta: %{"action" => "join_request", "requester_id" => req_id} = meta} <- Chat.get_system_message(message_id)
  ```

  И, что нарушает границу — **web-слой** читает ту же сырую форму (chat_live.ex:9473, 10398-10422). Тесты закрепляют строковую форму (channels_test.exs, chat_test.exs), опровергающих гвардов нет.
- **Влияние:** добавление нового типа системного сообщения требует согласованных правок в ~4 местах без единого владельца; опечатка в ключе не ловится компилятором; локализация строк живёт в LiveView, но зависит от формы `meta`, которую строит `Channels`. Классическая протекающая абстракция и нарушение web↔context.

### R188 · P3/quality · `lib/eden_web/live/chat_live.ex:9473,10394,10589` — рендер системных сообщений раздвоен

- **Проблема:** bubble-путь (группы) и flat-путь (комнаты) рисуют один тип сущности двумя несогласованными способами, читая разные ключи `meta`. Статус — advisory (confidence medium).
- **Доказательство в коде:** bubble-путь роутит по `meta["action"]`:

  ```elixir
  # chat_live.ex:9473 — system_notice/1
  defp system_notice(%{"action" => "member_added", "name" => name}), do: gettext("%{name} was added to the group", name: name)
  defp system_notice(%{"action" => "member_removed", "name" => name}), do: gettext("%{name} was removed from the group", name: name)
  defp system_notice(_meta), do: ""
  # chat_live.ex:10589 — message_bubble system clause
  <div id={@id} class="ed-sysmsg"><span>{system_notice(@message.meta)}</span></div>
  ```

  flat-путь (комнаты) хардкодит knock-рендер инлайн по `meta["requester_name"]`/`meta["status"]` (chat_live.ex:10398-10422, см. R189). Две ветки, разные ключи.
- **Влияние:** системный тип, видимый и в группе, и в комнате, потребует правок в двух местах; read-логика дублируется, усиливая утечку из R010.

### R189 · P3/quality · `lib/eden_web/live/chat_live.ex:10394` — flat-системка всегда рендерит «requested to join»

- **Проблема:** клауза `flat_message` для `kind == "system"` безусловно выводит «X requested to join» и ветвит только по `meta["status"]`, игнорируя тип события. Статус — advisory (confidence medium).
- **Доказательство в коде:**

  ```elixir
  # chat_live.ex:10394
  defp flat_message(%{message: %{kind: "system"}} = assigns) do
    ~H"""
    <div id={@id} class="ed-sysmsg">
      <span>{gettext("%{name} requested to join", name: @message.meta["requester_name"])}</span>
      <button :if={@admin and @message.meta["status"] == "pending"} ... phx-click="approve_join">{gettext("Add")}</button>
      <button :if={@admin and @message.meta["status"] == "pending"} ... phx-click="decline_join">{gettext("Decline")}</button>
      <span :if={@message.meta["status"] == "accepted"} class="ed-sysmsg__done">{gettext("Added")}</span>
      <span :if={@message.meta["status"] == "declined"} class="ed-sysmsg__muted">{gettext("Declined")}</span>
    </div>
    """
  end
  ```

  Нет ветвления по `meta["action"]` — в отличие от `system_notice/1`. Сегодня в комнатах только join-request, поэтому корректно; но любой второй тип системного сообщения в комнате отрендерится как «X requested to join» с пустым именем.
- **Влияние:** хрупкая связка — новый системный тип в комнате даст неверную/пустую надпись без явной ошибки; групповая ветка уже делает это правильно через `system_notice/1`.

### R147 · P3/quality · `lib/eden/chat.ex:3344` — `forward_message/4` с `root_id` молча игнорирует `target_conversation_id`

- **Проблема:** клауза с `root_id` не использует параметр `target_conversation_id` (подчёркнут `_`); целью становится комната рута. Рассинхрон `target ≠ root.conversation_id` не сигнализируется. Статус — advisory (confidence medium).
- **Доказательство в коде:**

  ```elixir
  # chat.ex:3344
  def forward_message(%Scope{user: user} = scope, message_id, _target_conversation_id, root_id) do
    with {:ok, source} <- fetch_message(scope, message_id),
         :ok <- ensure_source_access(scope, source),
         :ok <- ensure_not_deleted(source),
         {:ok, root} <- fetch_message(scope, root_id),
         :ok <- ensure_not_deleted(root),
         :ok <- ensure_root(root),
         :ok <- ensure_threaded(root.conversation_id) do
      do_forward_reply(user, root, Repo.preload(source, :attachments))   # цель = root.conversation_id
    ...
  ```

  `do_forward_reply` вставляет в `root.conversation_id` (chat.ex:3393), параметр `_target_conversation_id` нигде не сверяется.
- **Влияние:** маскирует баги вызывающего кода — если web-слой передаст рассинхронизированную пару (панель треда пережила смену комнаты), форвард молча уедет в другую комнату. Сейчас гипотетично, но контракт `/4` двусмыслен.

## План реализации

- [ ] **Ввести владельца формы `meta` в контексте.** Новый модуль `Eden.Chat.SystemMessage` (или функции на `Chat.Message`) со строгими конструкторами: `join_request(requester)`, `member_added(user)`, `member_removed(user)` — единственный источник jsonb-карт; и read-функцией `describe/1`, возвращающей размеченный кортеж: `{:join_request, %{requester_id, requester_name, status}}` | `{:member_added, %{user_id, name}}` | `{:member_removed, %{user_id, name}}` | `:unknown`. Это шаг 0 — от него зависят остальные.
- [ ] **Перевести писателей на конструкторы.** Заменить голые карты на вызовы конструкторов: chat.ex:477 (member_removed), chat.ex:576 (member_added), channels.ex:422 (join_request), а также мутатор статуса channels.ex:1104 (`resolve_join_request`, `Map.put(message.meta, "status", status)` → именованный хелпер).
- [ ] **Инкапсулировать сырые fragment-читатели внутри контекста.** Свести доступ к jsonb в chat.ex:1095-1097 (`pending_join_request`), chat.ex:628 (`scrub_meta_name`, ключи `"requester_id"`/`"user_id"`) и матч channels.ex:440 (`approve_room_join`) к именованным ключам/хелперам одного владельца, чтобы имена ключей не были литералами, размазанными по запросам. Строковые ключи `"requester_id"`/`"user_id"` для scrub — единый источник констант.
- [ ] **Свести рендер к одному компоненту** `<.system_message message={..} admin={..} />`, матчащему на результат `describe/1` (атомы), а не на строковые ключи. Он заменяет обе ветки: `system_notice/1` + `message_bubble` system-клаузу (chat_live.ex:9473-9479, 10589) **и** `flat_message` system-клаузу (chat_live.ex:10394-10426). [R188]
- [ ] **Добавить catch-all по типу события в рендере комнат.** Внутри единого компонента ветвить join-request по `{:join_request, _}` и иметь клаузу для неизвестного типа (пустой/безопасный рендер), чтобы новый системный тип в комнате не выдавал себя за запрос на вступление. [R189]
- [ ] **Устранить двусмысленность `forward_message/4`.** Выбрать одно: (а) добавить в `with` проверку `true <- root.conversation_id == safe_id(target_conversation_id)` (иначе `{:error, :not_found}`), либо (б) честно убрать `target_conversation_id` из головы `/4` и задокументировать, что цель — комната рута. Обновить вызов(ы) в web-слое соответственно. [R147]

## Критерии приёмки

- Форма `meta` системных сообщений строится и читается ТОЛЬКО через владельца в `Eden.Chat`; ни один строковый jsonb-ключ системного сообщения не остаётся в web-слое (`chat_live.ex`).
- Системные сообщения (группы и комнаты) рендерятся одним компонентом по размеченному `describe/1`; неизвестный тип в комнате безопасен.
- `forward_message/4` больше не игнорирует молча рассинхрон `target ≠ root` (проверка или удалённый параметр).
- `mix check` и `mix dialyzer` зелёные; нет регрессий в knock-флоу (#41), групповых notices (#165) и scrub при удалении аккаунта (#303/#305).

## Заметки

- **Развилка для человека по R147:** проверять `target == root.conversation_id` или удалить параметр из `/4`. Проверка безопаснее (ловит будущий баг вызова), удаление — честнее по контракту. Мелкая, но продуктово-стилевая — предпочтительна проверка, т.к. дёшева и защитна.
- Все четыре находки при чтении кода **подтверждены**; смещений строк относительно входа не обнаружено (R010: 9473/10394 точны; R147: 3344 точна; R188/R189: 9473/10394/10589 точны).
- R010 верифицирована (high/verified); R147/R188/R189 — advisory (confidence medium), но код их подтверждает. Граница «мягкая»: сегодня нет краша/потери данных/некорректности, форма консистентна — стоимость проявляется при расширении. Приоритет issue = P1 по рубрике «нарушение границ контекстов» (наивысший среди находок).
- Пятое место чтения формы для полноты инвентаря — `scrub_deleted_user_content/1` (chat.ex:610) через `scrub_meta_name` (chat.ex:624), с документированным гэпом по сообщениям без `user_id` (до seam #305): их тоже стоит учесть при инкапсуляции ключей, но менять поведение gap не нужно.
