# Закрыть пробелы в failed-send/optimistic: тесты доставки, мёртвый движок, потеря подписи и thread-album

**Приоритет:** P1 — среди находок есть корректностные баги (R061 тихая пропажа thread-album, R081/R082 молчаливая потеря подписи и не-retriable-нода на серверной ошибке) и отсутствие тестов на core-путь доставки на «фляки»-линке (R009) — по рубрике «нет тестов на core-логику / нарушение корректности» = P1.
**Лейблы:** `bug` `P1: critical` `backend` `UI/UX`
**Находки ревью:** R009, R016, R061, R062, R063, R080, R081, R082, R085, R102, R103

## Контекст

Всё это — один куст вокруг **оптимистичной отправки и failed-send** (`SendQueue` / `ThreadSendQueue`, sequential media-движок `:attachment_seq`, серверные `send_dispatch`/`send_reply_dispatch`/`commit_album` в `lib/eden_web/live/chat_live.ex`). Это самое хрупкое место продукта: главное обещание PRODUCT.md — доставка на нестабильном RU↔overseas-линке, и класс «молча потерянное сообщение без индикации» здесь уже чинили в #142/#158/#309/#310. Находки бьются на три группы: (1) **дыры в тестах** core-пути доставки (R009, R016, R062, R080, R102, R103); (2) **реальные корректностные баги** на error/edge-путях (R061, R081, R082); (3) **техдолг**, повышающий риск регрессий — мёртвый параллельный движок (R085) и дублирование двух огромных JS-хуков (R063). Общий корень: два параллельных движка отправки (concurrent `media_sending` → sequential `:attachment_seq`) и дубль `SendQueue`/`ThreadSendQueue` живут бок о бок, а покрытие тестами не поспевает за переходом на sequential-движок.

## Находки и доказательства

### R009 · P1/test-gap · `lib/eden_web/live/chat_live.ex:7762` — нет ни одного теста на failed-send текста (●!, Resend/Delete, «Resend N», watchdog)

- **Проблема:** Клиентская логика failed-send текстовых сообщений — `markFailed`, `openFailMenu`, `resendNode`, «Resend N», `armSendWatchdog` (20s online / 3s offline) — не покрыта ни одним тестом. `grep 'nack'` по всему `test/` даёт **0 попаданий**; ни один LiveView-тест не триггерит nack-ответ.
- **Доказательство в коде:** Клиентский путь на месте (`markFailed` — 7762, `resendNode` — 7803, `openFailMenu` — 7811, «Resend N» — 7829-7833), и он единственный на happy-path для комнат/групп, где оптимистичной ноды нет:
  ```javascript
  // chat_live.ex:7762
  markFailed(clientId, body) {
    let node = this.findNode(clientId)
    // Rooms/groups draw no optimistic node on the happy path (#130/#142), so a
    // rejected send has nothing to mark — materialize it now (faded), then flag it failed.
    if (!node && body != null) { this.addOptimistic(clientId, body); node = ... }
    ...
    node.classList.add("ed-msg-failed")
  ```
  Watchdog по таймауту → `markFailed` (6354-6360), офлайн-грейс 3s (6370-6372), приём nack → `markFailed` (6348). Сервер шлёт `%{"nack" => client_id}` (14231-14232) на слишком длинный body (`send_text`, 14220-14223) и на невалидный reply (`send_thread_reply_text`, 14708-14715). Дубль всей failed-логики есть и в `.ThreadSendQueue` (7868+). **Существующие** failed-card тесты (`chat_live_test.exs:1256-1400`) — про ДРУГОЙ путь (`:attachment_retry`/`media_send_reset`, стол аплоада медиа), не про text-nack/watchdog. `resilience.spec.js` покрывает только happy-path (disconnect→optimistic→reconnect→доставка, тесты на строках 13 и 37), не проверяет `.ed-msg-failed`/●!/меню/Resend/Resend N и не даёт watchdog сработать.
- **Влияние:** Регрессия в самом хрупком клиентском пути (таймеры, ветвление online/offline, материализация ноды для комнат/групп) = молча потерянные сообщения без индикации — ровно тот класс, что чинили в #142/#309/#310. Статус: **verified, confidence high**.

### R061 · P2/bug · `lib/eden_web/live/chat_live.ex:14120` — `send_reply_dispatch` читает сырые `.entries`, cancelled-призрак роняет album-reply в текст

- **Проблема:** В `send_reply_dispatch` берётся сырой список энтри, а не `live_entries`. Отменённый мид-батч энтри Phoenix держит в `entries` как `cancelled?: true` до завершения канала — это делает `entries != []` истиной, но `Enum.all?(&.done?)` — ложью, и отправка проваливается в текстовую ветку, **молча НЕ отправив реально застейдженные вложения**.
- **Доказательство в коде:**
  ```elixir
  # chat_live.ex:14124 — СЫРОЙ список (в отличие от главного композера)
  entries = socket.assigns.uploads.thread_attachment.entries
  ...
  # 14145 — album-ветка:
  entries != [] and Enum.all?(entries, & &1.done?) ->
    send_thread_album(socket, root, body, reply_to_id)
  ```
  Главный композер `send_dispatch` (14054) наоборот делает `entries = live_entries(socket.assigns.uploads.attachment)` (14061) именно ради этого класса багов (коммент 14059-14060, #158), а `live_entries` (13294) отбрасывает `cancelled?`-призраки, живучесть которых описана в 13288-13293. Итог — непоследовательность точно того класса #158, что уже ловили в main-пути.
- **Влияние:** При частичной отмене вложения в thread-reply остальные застейдженные вложения тихо не уйдут. Путь **оборонительный** (обычная thread-медиа идёт через `.ThreadSendQueue.onSubmit`→`threadComposeSend`→фидер `:attachment_seq`, а `:thread_attachment` энтри отменяются на 7391), достижим лишь при деградированном JS/крафченом событии + частичной отмене альбома. Без краха/порчи данных. Статус: **verified, confidence low** (низкая частота, но реальная непоследовательность).

### R081 · P2/bug · `lib/eden_web/live/chat_live.ex:13499` — подпись теряется, если первый альбом падает при коммите

- **Проблема:** `commit_album` помечает `caption_used=true`, как только берёт подпись, и пишет это в очередь **безусловно** — даже в ветке `{:error, reason}`. Если альбом падает, подпись уже «использована», само сообщение-альбом не создано, а `maybe_trailing_caption` не всплывёт (условие `queue.albums == %{}`). Итог: текст подписи молча пропадает.
- **Доказательство в коде:**
  ```elixir
  # chat_live.ex:13499
  {caption, caption_used} =
    if not queue.caption_used and queue.caption != "", do: {queue.caption, true}, else: {"", queue.caption_used}
  ...
  result = Chat.create_attachments(scope, queue.conv_id, spec.sources, opts)
  socket = case result do
    {:ok, _} -> socket
    {:error, reason} -> socket |> put_flash(...) |> push_media_failed(acid)  # 13517
  end
  # 13520 — caption_used пишется в очередь БЕЗУСЛОВНО, и в {:error} тоже:
  {socket, %{queue | albums: Map.delete(queue.albums, acid), caption_used: caption_used}}
  ```
  `maybe_trailing_caption` (13525-13527) фильтрует `not queue.caption_used and queue.albums == %{}` — при `caption_used=true` в смешанном send'е подпись трейлингом не всплывёт. Границы: путь **полной** отмены альбома (`drop_album_photo`, `new_expected<=0`) НЕ вызывает `commit_album`, там `caption_used` остаётся `false` — баг только про `{:error}` коммита.
- **Влияние:** Пользователь ввёл подпись к фото, отправка упала на сервере — текст подписи не сохранён нигде. Потеря пользовательского контента, редкий error-путь. Статус: **verified, confidence medium**.

### R082 · P2/ux · `lib/eden_web/live/chat_live.ex:13517` — серверная ошибка seq-медиа молча удаляет оптимистичную ноду вместо retriable-состояния

- **Проблема:** В `commit_album`/`seq_settle_file` при `{:error, reason}` вызывается `push_media_failed` → клиентский `media_failed` → `dropPending(id)` → `node.remove()`: оптимистичная нода **удаляется целиком**, только транзиентный flash. Комментарий на 13493-13494 утверждает «a failed album marks its optimistic node failed (retriable)», но retriable-состояния (красный !, Resend/Delete) не появляется. Это расходится с клиентским столлом (`armStall`→`markUploadFailed`), который наоборот оставляет карточку с Resend.
- **Доказательство в коде:**
  ```javascript
  // chat_live.ex:5860 — серверная ошибка просто удаляет ноду:
  this.handleEvent("media_failed", ({ id }) => { this.dropPending(id) })
  // 7030:
  dropPending(id) { const node = this.findNode(id); if (!node) return; ...; node.remove() }
  // vs. стол (armStall → markUploadFailed → markMediaFailed), который КЕЙПИТ ноду с Resend (7076-7079)
  ```
  Коммент `commit_album` (13493-13494) обещает retriable, но `push_media_failed` (13517, 14521-14522) → `media_failed` → `dropPending`; Resend доступен только для клиентского stall-watchdog, не для серверного nack.
- **Влияние:** При редкой серверной ошибке (валидация/декомпресс-бомба/сбой стораджа) пользователь видит, как только что показанные фото/файл исчезают, флеш быстро уходит — send выглядит как молчаливая потеря без пере-отправки из UI. Плюс комментарий вводит будущего разработчика в заблуждение. Статус: **advisory, confidence medium**.

### R016 · P2/test-gap · `assets/js/send_store.js:163` — durable send-queue (IndexedDB resume) не покрыт e2e

### R063 · P2/quality · `lib/eden_web/live/chat_live.ex:5752` — дублирование optimistic-send: `.SendQueue` (~2116 строк) vs `.ThreadSendQueue`

- **Проблема:** Два огромных colocated-JS-хука в `render/1` — `.SendQueue` (5752-7867) и `.ThreadSendQueue` (7868-8110) — реализуют почти одну и ту же оптимистичную отправку (client_id, очередь, watchdog, failed-состояние, Resend/Delete-меню). Комментарий у `.ThreadSendQueue` прямо признаёт причину дубля. Плюс раскладка альбома зеркалится из Elixir `AlbumLayout` в JS без синхронизирующего тест-вектора.
- **Доказательство в коде:**
  ```javascript

  // A focused, self-contained hook (colocated hooks can't share .SendQueue's helpers): ...
  ```
  Дубляж раскладки реален: `chat_live/album_layout.ex` moduledoc (стр. 7-9) фиксирует, что JS зеркалит `rows`/`aspect`/`strip_photo?` как `balanceRows`/`albumAspect`/`isStrip` (chat_live.ex:6493-6734), а тест-вектора JS↔Elixir нет (`album_layout_test.exs` проверяет только Elixir). - **Влияние:** Массивные встроенные хуки непереиспользуемы и нетестируемы; два экземпляра расходятся при правках (уже разошлись). Уточнение: хуки НЕ «почти одно и то же» целиком — thread album/file идёт через `.SendQueue.threadComposeSend` (7312), а `.ThreadSendQueue` отвечает лишь за failed-send текстовых reply; отсутствие клока/галочек в нём — осознанный дизайн (треды стримятся через `{:thread_reply}`), не деградация. Это maintainability-риск, не краш. Статус: **verified, confidence high** (понижено до P2).

## План реализации

## Заметки

- **Продуктовая развилка (R082):** «retriable-нода на серверной ошибке» vs «drop + честный коммент». Для консистентности с seq-столлами (`armStall`→`markUploadFailed` кейпит карточку с Resend) предпочтительно первое, но это UX-решение — при decompression-bomb/валидации повторная отправка того же файла всё равно упадёт, так что «Resend» может путать. Требует решения человека.
- **Приоритеты внутри issue разнесены:** R009/R061/R081/R082 — корректность/потеря контента (двигают issue в P1); остальные — техдолг/тесты (P2). Механические фиксы (R061, R081, R103) и удаление R085 — `ready-for-agent`; крупный рефактор R063 и продуктовая развилка R082 — ближе к `ready-for-human`. Общий триаж выставлен `ready-for-agent`, т.к. большинство пунктов локализованы и специфицированы; R063/R082 стоит выделить отдельными задачами.
- **Смещения строк относительно входа:** находки R061/R081/R082/R085/R080 фиксировались с координатами `~+10000` (входные `line` из старого снапшота); в актуальном файле (14852 строки) реальные позиции — те, что процитированы выше (send_reply_dispatch — 14120, commit_album — 13498, media_sending handler — 606). R062 указывал 5122 — фактический swipe-блок ~5183-5290.
- **Все находки подтвердились при чтении кода.** R061 переоценки нет, но путь оборонительный (P2 адекватен). R063 слегка переоценивает «оба хука почти одинаковы» — thread album/file идёт через `.SendQueue.threadComposeSend`, а отсутствие клока/галочек в `.ThreadSendQueue` — дизайн, не деградация (учтено, P2).
