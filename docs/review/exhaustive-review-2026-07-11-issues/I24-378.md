# Стабилизировать DOM-id у local_time/last_seen и свести дубли avatar/initials-хелперов веб-слоя

**Приоритет:** P2 — поддерживаемость и минорный churn/UX: нестабильный DOM-id в стрим-строках даёт лишний destroy/mount хука при каждом ре-стриме строки (реакция/edit/read-тик/thumbnail), а avatar/initials-хелперы размножены по 4-5 модулям (риск рассинхрона при смене схемы URL/cache-bust). Ни корректности, ни краха — поэтому P2, не P0/P1.
**Лейблы:** `enhancement` `P2: medium` `UI/UX` `architecture` `ready-for-agent`
**Находки ревью:** R074, R075, R078, R185, R191, R200

## Контекст
Веб-слой (`lib/eden_web/`) держит несколько мелких презентационных хелперов, которые расползлись по нескольким LiveView-ам и компонентам. Две тесно связанных проблемы:

1. **Нестабильный DOM-id.** Компоненты `local_time/1` и `last_seen/1` в `chat_live.ex` генерируют id через `System.unique_integer/1` — новый id на каждый рендер. Внутри стрим-контейнера `#messages` (`phx-update="stream"`) это ломает переиспользование узла morphdom: при ре-стриме строки `<time>` получает другой id, старый узел уничтожается, хук `.LocalTime`/`.LastSeen` пересоздаётся. Это ровно тот анти-паттерн «нестабильный id в стриме», о котором предупреждает `AGENTS.md` (R074, R075, R191).

2. **Дублирование avatar/initials-хелперов.** `avatar_src/1`, `initials/1` и родственные почти дословно повторены в `chat_live.ex`, `settings_live.ex`, `admin_live.ex`, `notify_hook.ex` и `shell_components.ex` — единый URL-роут `/users/:id/avatar?v=phash2(key)` и логика инициалов живут в 4-5 копиях, уже с расхождениями в защите от nil (R078, R200).

Плюс одна косметическая мелочь по optimistic-медиа-пину (R185), которую логично закрыть в этом же проходе по `chat_live.ex`.

## Находки и доказательства

### R074 · P2/bug · `chat_live.ex:12170-12181` — рандомный id ломает morphdom при ре-стриме
- **Проблема:** `local_time/1` рендерит `<time>` с id, производным от `System.unique_integer/1`, и вызывается внутри стрим-строк.
- **Доказательство в коде:**
  ```elixir
  defp local_time(assigns) do
    ~H"""
    <time
      class={@class}
      phx-hook=".LocalTime"
      id={"t-#{System.unique_integer([:positive])}"}
      datetime={DateTime.to_iso8601(@at)}
    >
      {Calendar.strftime(@at, "%H:%M")}
    </time>
    """
  end
  ```
  Компонент используется в стрим-строках: `flat_message` (`chat_live.ex:10484` — `<span class="ed-flat__time"><.local_time at={@message.inserted_at} /></span>`) и `msg_meta` (`10761 — <.local_time at={@at} />`, вызываемый из бабл-меты). Строки живут в `#messages` (`phx-update="stream"`). Ре-стрим одной строки идёт через `restream_message_in_place/3` → `stream_insert(:messages, ...)` (`12820-12834`, реакции/thumbnail) и `{:message_edited}` → `stream_insert`. При этом `<time>` получает новый id → morphdom не сопоставляет узлы. Хук `.LocalTime` (`chat_live.ex:5553-5562`: `mounted()/updated() → fmt()`) destroyed+mounted.
- **Влияние:** лишний destroy/mount цикл хука и замена DOM-узла на частых событиях (реакции/правки). Верификатор отметил: «видимое мигание времени» переоценено — `mounted()` зовёт `fmt()` синхронно в том же патч-цикле, откат к серверному UTC обычно не успевает отрисоваться; достоверна именно лишняя работа morphdom + пере-монтаж хука. Confidence medium (verified).

### R075 · P2/perf · `chat_live.ex:12175, 12193` — новый id на каждый рендер (то же и для last_seen)
- **Проблема:** та же нестабильность id распространяется и на `last_seen/1`.
- **Доказательство в коде:**
  ```elixir
  <time
    phx-hook=".LastSeen"
    id={"ls-#{System.unique_integer([:positive])}"}
    datetime={DateTime.to_iso8601(@peer.last_active_at)}
  >
  ```
  (`chat_live.ex:12191-12194`). `AGENTS.md:284` прямо рекомендует data-derived id для стрим-контекста (`id="edit-form-#{message.id}"`) — фикс совпадает.
- **Влияние:** при рестриме строки — повторный `fmt()` + destroy/mount хука. Верификатор уточнил: заявленная частота преувеличена — streamed-строки НЕ ре-рендерятся «на каждый тайпинг», только когда именно эта строка ре-стримится. Confidence high (verified). `last_seen` — в шапке 1:1 (не стрим), но нестабильный id так же вызывает лишний re-mount при перерисовке шапки; тривиально стабилизируется по `peer.id`.

### R191 · P3/quality · `chat_live.ex:12175` — дубликат R074/R075
- **Проблема:** та же находка, что R074/R075, зарегистрированная отдельным ревьюером.
- **Доказательство в коде:** см. R074/R075 — идентичный фрагмент `local_time/1`.
- **Влияние:** churn монтирования/размонтирования `.LocalTime` по видимой ленте при частых ре-стримах. Confidence medium (advisory). Сводится в один фикс с R074/R075.

### R078 · P2/quality · `chat_live.ex:13110, 13116, 13131` — дублирование avatar-хелперов в 4 модулях
- **Проблема:** `avatar_src/1`, `group_avatar_src/1`, `initials/1` определены приватно в нескольких модулях; `shell_components.ex` уже владеет каналовыми аналогами.
- **Доказательство в коде:**
  ```elixir
  # chat_live.ex:13110
  defp avatar_src(%{avatar_key: key, id: id}) when is_binary(key),
    do: ~p"/users/#{id}/avatar?v=#{:erlang.phash2(key)}"
  defp avatar_src(_user), do: nil
  # chat_live.ex:13116
  defp group_avatar_src(%{id: id, avatar_key: key}) when is_binary(key),
    do: ~p"/conversations/#{id}/avatar?v=#{:erlang.phash2(key)}"
  # chat_live.ex:13131
  defp initials(name), do: name |> String.first() |> String.upcase()
  ```
  Копии: `settings_live.ex:1623-1628` (`avatar_src/1` + `initials/1`), `admin_live.ex:729-737` (`avatar_src/1` + `initials/1`, тут `initials` уже с guard `is_binary(name) and name != ""` и веткой `"?"` — расхождение с остальными), `notify_hook.ex:104-107` (2-арная вариация `avatar_src(key, id)` — тот же URL). `shell_components.ex:404-412` держит `me_avatar_src/1` (тот же `/users/:id/avatar` URL) и `me_initials/1` (`"?"`-ветка), а также публичные `channel_avatar_src/1` (`398`) и `channel_initials/1` (`423`, до 2 инициалов). Confidence high (advisory — не верифицирован независимо, но факт дублирования очевиден из grep).
- **Влияние:** любое изменение схемы URL аватара или стратегии cache-bust надо править в 4-5 местах; уже есть расхождение в защите от nil/пустого имени.

### R200 · P3/quality · `settings_live.ex:1623-1628` — дубль avatar_src/initials в трёх+ модулях
- **Проблема:** та же находка, что R078, другим ревьюером; акцент на расхождении правил инициалов.
- **Доказательство в коде:**
  ```elixir
  # settings_live.ex:1623
  defp avatar_src(%{avatar_key: key, id: id}) when is_binary(key),
    do: ~p"/users/#{id}/avatar?v=#{:erlang.phash2(key)}"
  defp avatar_src(_user), do: nil
  defp initials(name), do: name |> String.first() |> String.upcase()
  ```
  Расхождения: `settings_live`/`chat_live` — `initials(name)` без guard (упадёт на `nil`); `admin_live`/`shell_components` — с guard и `"?"`. Confidence medium (advisory).
- **Влияние:** DX/сопровождаемость; риск разойтись при правках. Сводится в один фикс с R078.

### R185 · P3/quality · `chat_live.ex:6995-6997` — pin по img `load` не срабатывает при ошибке загрузки
- **Проблема:** для незавершённых `<img>` в optimistic-медиа-ноде вешается `load`-обработчик пина `{once:true}`, но на `error` пина нет.
- **Доказательство в коде:**
  ```elixir
  pin()
  row.querySelectorAll("img").forEach((img) => {
    if (!img.complete) img.addEventListener("load", pin, { once: true })
  })
  ```
  (`chat_live.ex:6994-6997`). Если превью не декодируется (битый blob/data-URL), `load` не выстрелит и повторного пина к низу не будет.
- **Влияние:** незначительный UX-нюанс — при неудачной локальной отрисовке прокрутка к низу может не догнать рост карточки. Утечки нет (`once` + узел уедет при swap). Confidence low (advisory). Есть подстраховка: `contentRo` из `ScrollBottom` перепинивает по росту `#messages`.

## План реализации

- [ ] **Добавить `local_time/1` обязательный атрибут стабильного ключа и убрать `System.unique_integer`.** В `chat_live.ex:12167-12181` добавить `attr :id, :string, required: true` (или `:key`) и рендерить `id={"t-#{@id}"}`. Прокинуть детерминированный ключ во всех колл-сайтах:
  - стрим-строки (главное): `flat_message` (`10484`) и `msg_meta` (`10761`) — вывести id из `@message.id`/`@at`. `msg_meta/1` получает только `@at`, поэтому либо добавить ему `attr :id`, прокинутый из вызывающих `message_bubble`/media-бабла, либо передавать ключ = `message.id`; убедиться, что в одной строке несколько `<time>` не сталкиваются (при необходимости суффикс, напр. `"t-#{message.id}-meta"`).
  - вне стрима (шапка/поиск/превью): `3432`, `3970`, `4242`, `8758`, `8877`, `8946`, `10525` — id вывести из `message.id`/`conversation.id`/`root.id` соответственно.
- [ ] **Стабилизировать `last_seen/1`** (`chat_live.ex:12187-12200`): `id={"ls-#{@peer.id}"}` вместо `System.unique_integer`.
- [ ] **Свести avatar/initials-хелперы в один презентационный модуль.** Создать `EdenWeb.AvatarHelpers` (или расширить `EdenWeb.CoreComponents`/`ShellComponents`) с публичными:
  - `user_avatar_src/1` (`~p"/users/#{id}/avatar?v=#{phash2(key)}"`, nil-fallback);
  - `group_avatar_src/1` (`~p"/conversations/#{id}/avatar?v=..."`);
  - `initials/1` — с единой защитой (`is_binary(name) and name != "" → String.first |> upcase`, иначе `"?"`; выбрать одно поведение и применить везде).
  Держать каналовые `channel_avatar_src/1`/`channel_initials/1` там же (уже публичны в `shell_components.ex:398/423`).
- [ ] **Удалить приватные копии и переключить импорты:** `chat_live.ex:13110-13119, 13131`, `settings_live.ex:1623-1628`, `admin_live.ex:729-737` (сохранить обёртку-компонент `user_avatar/1`, но на общих хелперах), `shell_components.ex:404-412` (`me_avatar_src`/`me_initials` → делегировать в общий `user_avatar_src`/`initials`). Для `notify_hook.ex:104-107` — либо использовать общий `user_avatar_src/1`, либо оставить тонкую 2-арную обёртку `user_avatar_src(key, id)` в общем модуле (там нет %User{}-структуры, только key+id из payload).
- [ ] **(Опционально, R185) Догрузить pin на ошибку изображения.** В `chat_live.ex:6995-6997` добавить `img.addEventListener("error", pin, { once: true })` рядом с `load`, либо явно положиться на `contentRo` из `ScrollBottom` и оставить как есть (задокументировать выбор комментарием). Мелочь — можно отложить, если фокус на стабилизации id/дедупе.

Порядок: сначала общий модуль avatar/initials + переключение импортов (механически, компилятор ловит промахи), затем `attr :id` для `local_time`/`last_seen` и правка колл-сайтов, затем опциональный R185.

## Критерии приёмки
- `local_time`/`last_seen` рендерят детерминированный, стабильный между патчами DOM-id; `System.unique_integer` из них убран.
- Единственный владелец `user_avatar_src`/`group_avatar_src`/`initials`; приватные копии в `chat_live`/`settings_live`/`admin_live`/`shell_components`/`notify_hook` удалены или делегируют в него.
- Поведение аватаров/инициалов не изменилось визуально (тот же URL/cache-bust, единый nil-fallback).
- `mix check` и `mix dialyzer` зелёные; тесты покрывают стабильность id и общий хелпер.

## Заметки
- R074/R075/R191 — одна и та же проблема (нестабильный id в `local_time`), сведена в один фикс; R078/R200 — одна и та же (дубли avatar/initials), тоже сведены. Все id перечислены для трассируемости.
- Развилка для человека/агента: единое поведение `initials/1` на `nil`/пустом имени — сейчас две версии (`"?"` в admin/shell vs падение в chat/settings). Предлагается защищённая версия с `"?"`; подтвердить, что это ок для всех вызовов (display_name у активных пользователей не бывает пустым, но payload-пути и удалённые аккаунты стоит проверить).
- `msg_meta/1` принимает только `@at` (не сам message) — чтобы дать `<time>` id по `message.id`, нужно либо добавить компоненту `attr :id`, прокинутый из вызывающих, либо передавать ключ явно. Это мелкий, но реальный объём правок в колл-сайтах — учтено в плане.
- R078 и R185 — status `advisory` (не верифицированы независимым проходом), но факт дублирования (R078) очевиден из grep, а R185 — низкой уверенности косметика; оба подтверждены чтением кода выше.
- Верификатор (R074) отметил: формулировка «видимое мигание времени» в исходной находке переоценена — реальный дефект это лишний DOM-churn/re-mount, не откат к UTC. Заголовки/тексты выше это учитывают.
