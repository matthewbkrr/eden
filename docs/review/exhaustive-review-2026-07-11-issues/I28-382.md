# Батч P3-корректностей: гонки владения/аватара, секундная граница unread, cast-конвенция и мелкие UI-рассинхроны

**Приоритет:** P3 — набор мелких корректностей и поддерживаемости: редкие гонки (два owner'а, откат `last_reply_at`), эвристики на секундной границе, нарушения cast-конвенции и захардкоженные magic-числа/незащищённые парсинги в UI. Ни одна находка не является потерей данных или крашем на штатном пути.
**Лейблы:** `bug` `P3: low` `backend` `UI/UX` `ready-for-agent`
**Находки ревью:** R111, R116, R121, R122, R123, R131, R141, R142, R155, R156, R157, R182, R183, R199

## Контекст

Сборный issue из четырнадцати P3-находок код-ревью 2026-07-10, объединённых по признаку «мелкая корректность / гигиена, локализованный механический фикс». Тематически они делятся на несколько групп:

- **Гонки и монотонность в `Eden.Chat`** — два owner'а при конкурентной передаче группы (R123), откат `last_reply_at` назад при конкурентных реплаях (R141/R142 — дубль), недостижимая ветка reclaim аватара из-за отсутствия `stale_error_field` (R122).
- **Эвристики на секундной границе** — секундная точность `last_read_at`/timestamps роняет unread на быстрой переписке (R121).
- **Нарушения cast-конвенции проекта** (AGENTS.md «programmatic fields must not be in cast») — `room_changeset` кастит `:position` (R156), `Membership`/`FolderMembership` кастят id-поля (R157).
- **No-op edit** ставит `edited_at` на медиа-пути, хотя текстовый путь его гасит (R131).
- **UI/JS-рассинхроны** — захардкоженный `maxlength="4000"` вместо `max_body()` (R182), отсутствие `Number.isFinite`-гарда в `GalleryMonths` (R183), дубль window-drop листенеров (R111), возможный двойной flash при rename папки (R199).

Общий корень у большинства — «эвристика/дефолт, который совпадает со строгим вариантом сегодня, но разъедется при рефакторинге»: продублированный лимит, секундное усечение, cast программного поля, комментарий, документирующий несуществующее поведение.

## Находки и доказательства

### R123 · P3/bug · `lib/eden/chat.ex:827` — конкурентная передача владения даёт двух owner'ов

- **Проблема:** `transfer_group_ownership/3` читает роль актёра (`ensure_role(actor_role, ~w(owner))`) **вне** транзакции, а `transfer_group_tx/3` в транзакции проверяет только `{1, _}` на апдейте цели. `update_group_role` проставляет роль безусловно (без `WHERE role = 'owner'`). Два конкурентных `transfer` разным целям оба проходят гейт «actor — owner», оба проставляют цели `owner` — итог два owner'а.
- **Доказательство в коде:**
  ```elixir
  # chat.ex:529 — роль актёра прочитана ДО транзакции
  actor_role when is_binary(actor_role) <- group_role_of(id, actor.id),
  :ok <- ensure_role(actor_role, ~w(owner)),
  ...
  :ok <- transfer_group_tx(id, actor.id, target_id) do

  # chat.ex:827
  defp transfer_group_tx(conversation_id, actor_id, target_id) do
    Repo.transact(fn ->
      case update_group_role(conversation_id, target_id, "owner") do
        {1, _} ->
          update_group_role(conversation_id, actor_id, "admin")   # безусловно
          {:ok, :ok}
        _ -> {:error, :not_found}
      end
    end)

  # chat.ex:815 — set: [role: role] без гварда на текущую роль
  defp update_group_role(conversation_id, user_id, role) do
    Repo.update_all(from(m in Membership, where: ... and is_nil(m.left_at)), set: [role: role])
  end
  ```
  Инвариант «ровно один owner» нигде больше не восстанавливается (`ensure_role`/`transfer` его лишь предполагают).
- **Влияние:** Ломается матрица прав (owner-only операции, «никто не удаляет owner'а», `transfer_group_ownership` требует роль owner — теперь их двое). Гонка узкая (нужны два одновременных `transfer` при исходно одном owner'е — практически двойной клик/дабл-сабмит того же владельца), но состояние неустранимо. Confidence: medium.

### R122 · P3/bug · `lib/eden/chat.ex:682` — ветка «reclaim при GC группы» недостижима (StaleEntryError)

- **Проблема:** Комментарий else-ветки `store_and_swap_group_avatar` обещает удалить записанный блоб, если группа была GC'd mid-flight. Но `conv |> Ecto.Changeset.change(...) |> Repo.update()` на удалённой строке **рейзит** `Ecto.StaleEntryError` (нет `stale_error_field`), а не возвращает `{:error, _}` — `else` не сработает, блоб-сирота останется, пользователь получит 500.
- **Доказательство в коде:**
  ```elixir
  # chat.ex:685
  with :ok <- Storage.put_binary(key, jpeg),
       {:ok, updated} <- conv |> Ecto.Changeset.change(avatar_key: key) |> Repo.update() do
    ...
  else
    error ->
      # "the group was GC'd mid-flight — reclaim the orphan"  ← этот путь мёртв
      Storage.delete(key)
      error
  end
  ```
  Сравни с `update_room/2` (chat.ex:975), который делает именно `Repo.update(stale_error_field: :id)` и разбирает `errors[:id]` — правильный идиом уже применяется в этом же файле.
- **Влияние:** Крайне редкая гонка (удаление группы одновременно с загрузкой аватара), но комментарий документирует поведение, которого нет. При рефакторинге на него положатся; тот же stale-идиом, что во всём проекте. Confidence: medium.

### R141 / R142 · P3/bug · `lib/eden/chat.ex:3030` — `last_reply_at` может откатиться назад при конкурентных реплаях

*(R141 и R142 — одна и та же находка на одной строке; сводятся в один пункт плана.)*

- **Проблема:** `deliver_reply` бампит root'а через `set: [last_reply_at: reply.inserted_at]` — last-writer-wins без монотонности. Два конкурентных реплая: если `update_all` реплая с более поздним `inserted_at` применится **раньше** реплая с более ранним, финальным значением останется более раннее время. `reply_count` при этом корректен (`inc` атомарен).
- **Доказательство в коде:**
  ```elixir

      )
  ```
  Монотонный `GREATEST`-идиом уже есть в `sync_thread_after_delete` для `reply_count` (chat.ex:2735: `fragment("GREATEST(reply_count - 1, 0)")`) — тот же приём применим к `last_reply_at`.
- **Влияние:** Сортировка `list_followed_threads` (`order_by: [desc_nulls_last: r.last_reply_at, ...]`, chat.ex:3000) и время «последнего ответа» в футере треда могут показать чуть устаревшее значение до следующего реплая (самовосстанавливается). Confidence: medium.

### R116 · P3/bug · `lib/eden/accounts/user.ex:214` — regex username с `^…$` пропускает хвостовой `\n`

  (Соседний `normalize_text/1` тримит free-text поля, но к username он не применяется — username проходит только `validate_format`/`validate_length`/`unique_constraint`.)
- **Влияние:** Username — это и логин, и публичный `@tag`. Невидимый `\n` делает его невоспроизводимым при входе (`get_user_by_username_and_password` не тримит) и странным в рендере `@tag`. Попасть можно вставкой из буфера в форму регистрации/переименования. Confidence: high.

## План реализации

Порядок: сперва чистые backend-корректности (гонки/валидация), затем cast-конвенция, затем UI. Изменения независимы — можно и отдельными коммитами.

- [ ] **R123 — устранить гонку двух owner'ов.** В `transfer_group_tx/3` (chat.ex:827): в начале транзакции взять `FOR UPDATE` на строку текущего owner'а (или на все memberships группы), и даунгрейдить актёра **условно** — `update_group_role` варианта с `WHERE role == "owner"` для актёра, проверяя `{1, _}` и на его апдейте; при `{0, _}` — `Repo.rollback(:not_owner)`. Так параллельный `transfer` увидит блокировку/уже-не-owner и откатится.

## Критерии приёмки

- Конкурентная передача владения группой оставляет ровно один owner (R123); reclaim блоба при GC группы не рейзит и удаляет сироту (R122); `last_reply_at` монотонно неубывающий (R141/R142); no-op медиа-edit не ставит `edited_at` и не броадкастит (R131).
- `username` с хвостовым `\n` отклоняется (R116).
- `:position` не кастится из attrs комнаты; id-поля memberships задаются на структуре — конвенция AGENTS.md соблюдена, поведение публичных функций не изменилось (R156, R157).
- Тредовый composer использует `Chat.Message.max_body()` (R182); GalleryMonths не рендерит `Invalid Date` при битом ts (R183).
- `mix check` и `mix dialyzer` зелёные; новые тесты покрывают перечисленные кейсы; регрессий в существующих chat/accounts/settings-тестах нет.

## Заметки

- **R121 — не механический фикс, требует решения человека.** Секундное усечение (`:utc_datetime` + `DateTime.truncate(:second)`) пронизывает всю схему сообщений и множество запросов. Оба варианта фикса (перейти на `:utc_datetime_usec` для `messages.inserted_at`/`memberships.last_read_at`, либо ввести `last_read_message_id` и сравнивать по id) — это миграция + правка всех трёх unread-гвардов (chat.ex:257/279/1284) и, возможно, поведения `{:read, ...}` broadcast. Вынесен в план не как чек-строка, а как развилка — **обсудить объём** перед реализацией (может стать отдельным issue). Impact на пользователя реальный (пропуск сообщения на границе секунды), но confidence medium и цена фикса высокая для P3.
- **R199 — уточнение по формулировке находки.** Находка предполагает «гонку», зависящую от того, успел ли `reload_folders` до прихода blur. Для LiveView это неточно: события одного сокета обрабатываются **строго последовательно** одним процессом, так что `reload_folders` из submit гарантированно отработает до обработки blur. Реальный (узкий) кейс двойного flash — когда blur-значение (`params["value"]`) отличается от submit-значения (напр. хвостовой пробел, тримится по-разному) или когда submit-rename упал. Отсюда — low confidence; фикс опционален.
- **R111** — status advisory, confidence low: не баг. Оставлен в issue для трассируемости; шаг плана помечен «опционально/осознанно оставить».
- **R141 == R142** — одна находка, продублированная в разных зонах (`chat-messages` / `chat-threads`); сведены в один пункт плана и один тест.
- Идиомы, на которые опираются фиксы, уже присутствуют в кодбазе: `stale_error_field: :id` (`update_room/2`, chat.ex:975) для R122; `GREATEST(...)` (`sync_thread_after_delete`, chat.ex:2735) для R141/R142; ветка «Unchanged — don't stamp» (`edit_message`, chat.ex:2541) для R131 — так что фиксы консистентны с существующим кодом.
