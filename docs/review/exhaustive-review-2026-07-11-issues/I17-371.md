# Закрыть тест-дыры в тредах: broadcast-контракты и follow-tracking

**Приоритет:** P2 — все три находки это отсутствующие тесты вокруг core-контрактов live-обновлений тредов (broadcast-кортежи, декремент-гейт, форвард-в-тред); не баг сам по себе, а риск молчаливой регрессии → поддерживаемость/краевые случаи.
**Лейблы:** `enhancement` `P2: medium` `backend` `ready-for-agent`
**Находки ревью:** R105, R106, R209

## Контекст

Подсистема тредов (`#57`, rooms-only) держит три вещи, целиком проверяемых в
`Eden.Chat`, но не покрытых тестами: (1) два PubSub-контракта, на которые
целиком опирается веб-слой — `{:thread_read, conv_id, root_id}` (мультитаб-синк)
и `{:thread_updated, root}` (пере-расчёт футера/списка после удаления реплая);
(2) гейт `last_viewed_at` в декременте `sync_thread_after_delete`, ровно то место,
где может незаметно поехать логика «просмотревшего не декрементим»; (3) форвард-в-тред,
отдельный путь кода (`insert_forward_reply`, не `create_reply`), который тоже
проходит через `deliver_reply → track_reply/notify_new`, но проверяется только по
`root_id`/атрибуции/`reply_count`. Общий корень: follow-tracking и broadcast-фанаут
тредов держатся на честности `describe "thread following (#57)"`, а часть путей туда
не заглядывает — рефактор `deliver_reply`/`track_reply`/`sync_thread_after_delete`
пройдёт зелёным, а бейджи, футеры и мультитаб молча сломаются.

## Находки и доказательства

### R105 · P2/test-gap · `test/eden/chat_test.exs:1363` — broadcast-контракты `{:thread_read}` и `{:thread_updated}` не заассерчены

- **Проблема:** В `describe "thread following (#57)"` (`chat_test.exs:1363`) ни один
  тест не подписывается на топики и не проверяет две PubSub-рассылки, которые
  веб-слой обрабатывает буквально. `mark_thread_read` (тест на строке 1405) вызывается,
  но `{:thread_read, ...}` не ловится; удаление реплая (тест на строке 1467) не проверяет
  `{:thread_updated, ...}`.
- **Доказательство в коде:** `mark_thread_read/2` шлёт кортеж на **user-топик**
  (`user:<id>:chat`, тот же, что у `subscribe_user/1`):

  ```elixir
  # lib/eden/chat.ex:2932
  # Sync the user's other open sessions (multi-tab): zero this thread's badge.
  Phoenix.PubSub.broadcast(
    @pubsub,
    user_topic(user.id),
    {:thread_read, root.conversation_id, root.id}
  )
  ```

  `sync_thread_after_delete/1` шлёт на **conversation-топик** (тот же, что у `subscribe/1`):

  ```elixir
  # lib/eden/chat.ex:2750
  case preloaded_message(root_id) do
    nil -> :ok
    root -> broadcast(conversation_id, {:thread_updated, root})
  end
  ```

  Веб-слой полагается на оба и на их точную форму:

  ```elixir
  # lib/eden_web/live/chat_live.ex:2405
  def handle_info({:thread_updated, root}, socket) do
  # lib/eden_web/live/chat_live.ex:2452
  def handle_info({:thread_read, conversation_id, root_id}, socket) do
  ```

  Ни `assert_receive {:thread_read, ...}`, ни `assert_receive {:thread_updated, ...}`
  в сьюте нет (проверено grep'ом по `chat_test.exs`). Уверенность high, статус advisory
  (независимо верификатором не подтверждалось, но код сверен вручную — цитаты актуальны).
- **Влияние:** Смена shape кортежа или потеря broadcast при рефакторинге
  (`sync_thread_after_delete`, `mark_thread_read`) пройдёт зелёной сборкой, а
  мультитаб-бейджи тредов и футеры/список после удаления реплая молча перестанут
  обновляться у пользователя.

### R106 · P2/test-gap · `test/eden/chat_test.exs:1467` — гейт `last_viewed_at` в декременте `sync_thread_after_delete` не покрыт

- **Проблема:** Тест «deleting an unread reply decrements the follower's count»
  (`chat_test.exs:1467`) покрывает только фолловера с `last_viewed_at = nil` (alice
  как автор рута, тред не открывала). Вторая ветка гейта — «фолловер, просмотревший
  тред ПОСЛЕ удаляемого реплая, НЕ декрементится» — не проверяется.
- **Доказательство в коде:** гейт из двух условий, тестируется только левая часть
  дизъюнкции (`is_nil(...)`):

  ```elixir
  # lib/eden/chat.ex:2743
  from(tm in ThreadMembership,
    where:
      tm.root_id == ^root_id and tm.unread_replies > 0 and
        (is_nil(tm.last_viewed_at) or tm.last_viewed_at < ^reply_at)
  )
  |> Repo.update_all(inc: [unread_replies: -1])
  ```

  Правая ветка (`tm.last_viewed_at < ^reply_at`) означает: если фолловер уже открыл
  тред (`mark_thread_read` проставил `last_viewed_at` и обнулил `unread_replies`), то
  при удалении **старого, уже прочитанного** реплая его счётчик не трогается. Ни один
  тест не проходит через `mark_thread_read` перед удалением реплая. Уверенность high,
  статус advisory.
- **Влияние:** Это ровно граница, где живёт потенциальный баг некорректного декремента
  (сдвиг `<` на `<=`, потеря `unread_replies > 0`, инверсия гейта). Без теста ошибка
  уведёт счётчик в минус/фантом либо задвоит декремент незаметно.

### R209 · P3/test-gap · `test/eden/chat_test.exs:1126` — форвард-в-тред не проверяет follow-tracking

- **Проблема:** Тест «forwarding a message INTO a thread lands as a reply and bumps
  the root» (`chat_test.exs:1126`) проверяет `root_id`, `body`, `forwarded_from_id`,
  `{:thread_reply, ...}` и `reply_count == 1`, но НЕ то, что follow-эффекты отработали:
  автофоллоу форвардера с `unread 0`, втягивание автора рута с `unread 1`.
- **Доказательство в коде:** форвард-в-тред — отдельный путь (`insert_forward_reply`,
  не `create_reply`), но сходится в `deliver_reply`, который и запускает
  `track_reply` + `notify_new`:

  ```elixir
  # lib/eden/chat.ex:3377
  defp do_forward_reply(user, root, source) do
    result =
      Repo.transact(fn ->
        with {:ok, reply} <- insert_forward_reply(user, root, source),
             :ok <- copy_attachments(reply.id, source.attachments) do
          {:ok, reply}
        end
      end)

    case result do
      {:ok, reply} -> {:ok, deliver_reply(root, reply)}  # ← тот же путь, что create_reply
      error -> error
    end
  end
  ```

  ```elixir
  # lib/eden/chat.ex:3037
  # Follow tracking (#57): the replier auto-follows, the root author is pulled
  # in on the first reply, and every other follower gains one unread reply.
  track_reply(root, reply)
  broadcast(root.conversation_id, {:thread_reply, fresh_root, reply})
  notify_new(reply)
  ```

  Соседние follow-тесты (`chat_test.exs:1374`, `1387`) уже используют образец
  `Chat.thread_follow_state(scope(...), root.id)` → `%{following: _, unread: _}`.
  В сетапе `describe "threads and replies"` (строка ~1096) форвардер `alice` —
  автор рута, `bob` — второй участник; для чистого разделения ролей форвардера и
  автора рута тест лучше расширить так, чтобы форвардил не автор рута. Уверенность
  high, статус advisory.
- **Влияние:** Рефактор `deliver_reply`/`track_reply` может потерять follow-эффекты
  именно на форвард-пути (если кто-то переставит `track_reply` в `create_reply`
  вместо `deliver_reply`), и сьют этого не заметит — форвард-в-тред тихо перестанет
  подписывать/уведомлять.

## План реализации

- [ ] **R105 — тест broadcast `{:thread_read}`.** В `describe "thread following (#57)"`
  (`chat_test.exs:1363`) добавить тест: `Chat.subscribe_user(scope(alice))`, создать
  реплай от bob, `Chat.mark_thread_read(scope(alice), root.id)`, затем
  `assert_receive {:thread_read, conv_id, root_id}` с проверкой `conv_id == conv.id`
  и `root_id == root.id`. (В сетапе этого describe есть `conv`, `root`, `carol`.)
- [ ] **R105 — тест broadcast `{:thread_updated}`.** Тот же describe: `Chat.subscribe(conv.id)`,
  создать два реплая (`r1`, `r2`), удалить один через `Chat.delete_message_for_both(scope(bob), r1.id)`,
  затем `assert_receive {:thread_updated, %Message{id: rid, reply_count: 1}}` с `rid == root.id`.
  Проверить, что `%Message{}` доступен как алиас в тест-файле (в `chat_test.exs` он уже
  используется, напр. в паттернах `{:ok, %{reply_count: ...}}`) — при необходимости
  матчить `%{id: ..., reply_count: 1}` без явного `%Message{}`.
- [ ] **R106 — тест гейта `last_viewed_at` (просмотревший не декрементится).** Тот же
  describe, рядом с тестом на строке 1467: `r1` от bob → `Chat.mark_thread_read(scope(alice), root.id)`
  (alice просмотрела, `unread → 0`, `last_viewed_at` проставлен) → `r2` от bob
  (`unread → 1`) → `Chat.delete_message_for_both(scope(bob), r1.id)` → `assert %{unread: 1}
  = Chat.thread_follow_state(scope(alice), root.id)` (удаление старого, уже прочитанного
  реплая НЕ трогает счётчик). Зеркальный ассерт к существующему тесту (`nil`-ветка).
- [ ] **R209 — расширить тест форвард-в-тред follow-эффектами.** В тесте
  `chat_test.exs:1126` после `assert_receive {:thread_reply, ...}` добавить
  `assert %{following: true, unread: 0} = Chat.thread_follow_state(scope(alice), root.id)`
  (форвардер автофолловится, свой реплай прочитан). Для проверки втягивания автора
  рута с unread нужен третий участник-форвардер, отличный от автора рута: либо
  форвардить реплаем от `bob` вместо `alice` и проверить
  `assert %{following: true, unread: 1} = Chat.thread_follow_state(scope(alice), root.id)`
  (alice — автор рута — втянута с 1 unread), либо добавить `carol` в сетап и форвардить
  от неё. Предпочтительно: форвард от `bob`, ассерт unread у `alice`.

## Тесты

  `subscribe_user` + `mark_thread_read` → `assert_receive {:thread_read, ^conv_id, ^root_id}`.

- [ ] `describe "thread following (#57)"` — гейт декремента: просмотревший тред после
  удаляемого реплая НЕ декрементится (`mark_thread_read` до `delete_message_for_both(r1)`,
  `assert %{unread: 1}`); существующий тест 1467 остаётся как `nil`-ветка.
- [ ] `describe "threads and replies"` (`test:1126`) — форвард-в-тред: форвардер
  `following: true, unread: 0`; автор рута `following: true, unread: 1`.
- [ ] (Опционально, если хочется прибить и notify-контракт форварда) подписаться на
  notify-топик автора рута (`user:<id>:notify`) и `assert_receive {:notify, _}` — но это
  тяжелее; минимум follow-state достаточно для трассируемости R209.

## Критерии приёмки

- Новые тесты в `test/eden/chat_test.exs` падают, если убрать/поломать соответствующий
  broadcast (`{:thread_read}`/`{:thread_updated}`), инвертировать гейт `last_viewed_at`,
  или вырезать `track_reply` из форвард-пути (реально проверяют контракт, а не проходят
  вхолостую).
- `mix check` зелёный (format, credo --strict, test), `mix dialyzer` зелёный.
- Никаких изменений в `lib/` — только тесты; поведение неизменно, регрессий нет.

## Заметки

- Все три находки подтверждены чтением актуального кода; номера строк совпадают с
  входом (`chat.ex:2743/2750/2932/3037/3377`, `chat_live.ex:2405/2452`,
  `chat_test.exs:1126/1363/1467`). Ни одна находка не отпала.
- R106 связан с находкой про `follow_thread` из другого issue (гейт `last_viewed_at` —
  «то самое место, где живёт баг некорректного декремента»): если параллельно правится
  сам декремент, зеркальный тест ручного фолловера с до-подписочным реплаем зафиксирует
  и текущий баг — но это уже относится к issue про фикс `follow_thread`, здесь достаточно
  прибить гейт как контракт.
- В сетапе `describe "threads and replies"` форвардер `alice` совпадает с автором рута —
  поэтому для R209 (проверка втягивания автора рута с unread) форвардить нужно от `bob`
  (второй участник комнаты в сетапе), иначе роль форвардера и автора рута не разделить.
