# Довести a11y модальных слоёв: focus-trap, role=dialog и alt для лайтбокса, видео-модалки и предпросмотра вложений

**Приоритет:** P2 — все три находки это a11y/UX-дефекты (WCAG 2.4.3 Focus Order, 4.1.2 Name/Role/Value, 1.1.1 Non-text Content): клавиатурный/скринридерный пользователь теряет контекст и озвучку в модальных слоях. По рубрике проекта это `maintainability… UX`, не корректность логики и не потеря данных — значит P2, а не P1.
**Лейблы:** `enhancement` `P2: medium` `UI/UX` `ready-for-agent`
**Находки ревью:** R067, R068, R073

## Контекст

В `chat_live.ex` есть эталонный `.FocusTrap` colocated-hook (строки 4588-4615): при открытии он сохраняет `document.activeElement`, переносит фокус внутрь, зацикливает Tab в пределах диалога и возвращает фокус на триггер в `destroyed()`. Все серверно-рендеримые модалки навешивают его единообразно (`phx-hook=".FocusTrap"` — 9 мест: 9196, 9342, 9546, 9661, 9748, 9850, 10340, 11428, 11509) плюс `role="dialog" aria-modal="true"`.

Из этой конвенции выпадают три модальных слоя, и все — в одной подсистеме просмотра/отправки медиа:
- два **клиентских оверлея**, которые строятся вручную через `innerHTML`/`createElement` в хуках `.Lightbox` и `.VideoExpand` (лайтбокс фото, полноэкранный видео-плеер) — у них вообще нет ни `role=dialog`, ни управления фокусом;
- одна **серверная модалка** предпросмотра вложений (`compose_overlay`), у которой есть `role`/`aria-modal`/Escape, но забыт `phx-hook=".FocusTrap"` — единственная серверная модалка без него.

Общий корень: клиентские оверлеи писались в обход системы модалок и не получили её a11y-гарантий, а `compose_overlay` — механический пропуск одного хука. Чиним одним issue, чтобы привести все модальные слои к общей норме.

## Находки и доказательства

### R067 · P2/ux · `lib/eden_web/live/chat_live.ex:8337-8353` — Лайтбокс фото без ловушки/возврата фокуса, без role=dialog и с пустым alt

- **Проблема:** оверлей `#ed-lightbox` строится вручную; контейнер получает только `id`/`className` — ни `role="dialog"`, ни `aria-modal="true"`. `openLightbox()` не сохраняет `document.activeElement` и не переносит фокус в оверлей; `close()` не возвращает фокус на триггер; keydown ловит только Escape/стрелки, поэтому Tab выпадает на скрытую под оверлеем страницу. Увеличенный `<img alt="">` пуст для скринридера.
- **Доказательство в коде:** контейнер (8337-8339):
  ```javascript
  box = document.createElement("div")
  box.id = "ed-lightbox"
  box.className = "ed-lightbox"
  ```
  разметка с пустым alt (8350-8353):
  ```javascript
  box.innerHTML =
    `<button class="ed-lightbox__close" aria-label="${lbl.lbClose || "Close"}">${chevron(xmark)}</button>` +
    `<button class="ed-lightbox__nav ed-lightbox__nav--prev" aria-label="${lbl.lbPrev || "Previous"}">${chevron(left)}</button>` +
    '<img class="ed-lightbox__img" alt="">' +
    `<button class="ed-lightbox__nav ed-lightbox__nav--next" aria-label="${lbl.lbNext || "Next"}">${chevron(right)}</button>`
  ```
  открытие без сохранения/переноса фокуса (8326-8331):
  ```javascript
  box.classList.add("ed-lightbox--open")
  box.__swiped = false
  document.body.style.overflow = "hidden"
  document.addEventListener("keydown", box.__onKey)
  ```
  закрытие без возврата фокуса (8355-8359):
  ```javascript
  const close = () => {
    box.classList.remove("ed-lightbox--open")
    document.body.style.overflow = ""
    document.removeEventListener("keydown", box.__onKey)
  }
  ```
  keydown только Escape/стрелки, Tab не перехватывается (8360-8364):
  ```javascript
  box.__onKey = (e) => {
    if (e.key === "Escape") close()
    else if (e.key === "ArrowLeft") box.__step(-1)
    else if (e.key === "ArrowRight") box.__step(1)
  }
  ```
  Для сравнения — эталон `.FocusTrap` (4592-4609) делает всё заявленное: `this._prev = document.activeElement` → `(f[0] || this.el).focus()` → Tab-loop → в `destroyed()` `this._prev.focus()`. Статус находки: **verified**, confidence high.
- **Влияние:** просмотр фото — одна из основных фич, и для клавиатурного/скринридерного пользователя она открывается «в пустоту»: контент не озвучен (`alt=""`), фокус не заперт (Tab уходит на скрытую страницу под скримом), а после закрытия фокус не возвращается на плитку-триггер — теряется место в ленте.

### R068 · P2/ux · `lib/eden_web/live/chat_live.ex:8486-8494` — Видео-модалка без role=dialog/aria-modal и без управления фокусом

- **Проблема:** полноэкранный плеер `#ed-video-modal` строится вручную. Escape и клик по подложке закрывают (ок), но нет `role="dialog"`/`aria-modal="true"`, фокус не переносится в модалку при open и не возвращается на постер-триггер при close.
- **Доказательство в коде:** контейнер без role/aria (8486-8494):
  ```javascript
  box = document.createElement("div")
  box.id = "ed-video-modal"
  box.className = "ed-video-modal"
  const lbl = document.getElementById("message-scroll")?.dataset || {}
  const xmark = "…"
  box.innerHTML =
    `<button class="ed-video-modal__close" aria-label="${lbl.lbClose || "Close"}">…</button>` +
    '<video class="ed-video-modal__player" controls playsinline></video>'
  ```
  open без сохранения/переноса фокуса (8476-8478), close без возврата (8496-8507), keydown только Escape (8508-8510):
  ```javascript
  box.__onKey = (e) => {
    if (e.key === "Escape") close()
  }
  ```
  Статус находки: **advisory** (не проходила независимую верификацию верификатором — `verdict_note` пуст), confidence high; при чтении кода находка подтверждается. Менее критично, чем лайтбокс: внутри есть `<video controls>`, так что фокус попадает на реальный интерактивный элемент, но role/aria-modal и возврат фокуса всё равно отсутствуют.
- **Влияние:** скринридер не узнаёт, что открылся модальный слой (нет role/aria-modal), Tab может уйти на скрытую под оверлеем страницу, а после закрытия фокус не вернётся на постер. Тот же класс проблемы WCAG 2.4.3/4.1.2, что и лайтбокс.

### R073 · P2/ux · `lib/eden_web/live/chat_live.ex:10948-10956` — Модалка предпросмотра вложений без FocusTrap (единственная серверная модалка без него)

- **Проблема:** `compose_overlay` имеет `role="dialog" aria-modal="true"` и Escape, но НЕ навешивает `phx-hook=".FocusTrap"` — единственная серверная модалка без него.
- **Доказательство в коде:** диалог (10948-10956):
  ```elixir
  <div
    class="ed-compose"
    data-upload-preview
    role="dialog"
    aria-modal="true"
    aria-label={gettext("Attachment preview")}
    phx-window-keydown="cancel_all_uploads"
    phx-key="Escape"
  >
  ```
  `.FocusTrap` нигде внутри `compose_overlay` не встречается. Все прочие серверные диалоги его навешивают (grep по файлу — `phx-hook=".FocusTrap"` на 9196, 9342, 9546, 9661, 9748, 9850, 10340, 11428, 11509). Единственный другой `role="dialog"` без FocusTrap — `profile_popover` (11590), но это popover с хуком `.Popover`, не модальный диалог, вне scope. Статус находки: **advisory** (`verdict_note` пуст), confidence high; подтверждается при чтении. Природа компонента усугубляет дефект: композер-бар остаётся отрендеренным позади скрима и переводится в `inert` (см. коммент 10930-10935), так что без ловушки Tab уходит на инертный бар и дальше на страницу под скримом.
- **Влияние:** в модалке предпросмотра фото/файлов (с полем подписи и кнопками «Отмена»/«Добавить»/«Отправить») фокус не запирается и не возвращается на композер после закрытия — клавиатурный пользователь теряет контекст отправки прямо в момент отправки.

## План реализации

- [ ] **R073 (самый простой, механический):** добавить `phx-hook=".FocusTrap"` на диалог-контейнер `compose_overlay` — на `<div class="ed-compose" role="dialog" …>` (`chat_live.ex:10948-10956`). Диалог уже имеет `role`/`aria-modal`, так что хука достаточно. Убедиться, что у контейнера есть `id` (нужен LiveView для colocated-hook-биндинга); если id нет — добавить статический `id="compose-overlay"`. Проверить, что фокус попадает на первую кнопку заголовка (крестик «Отмена»), а не на скрытые контролы инертного бара позади.
- [ ] **R067 — лайтбокс, доступность контейнера:** в `box()` (`chat_live.ex:8337-8339`) после установки `id`/`className` навесить `box.setAttribute("role", "dialog")` и `box.setAttribute("aria-modal", "true")` (+ по желанию `aria-label` из `lbl`).
- [ ] **R067 — лайтбокс, alt увеличенного изображения:** заменить `<img class="ed-lightbox__img" alt="">` (8352) на осмысленный alt. Источник подписи — датасет плитки-триггера (напр. подпись/имя файла); поскольку `<img>` статичен в `innerHTML`, а картинка меняется в `show(n)` (8309-8320), задавать `img.alt` там же из `tiles[i].dataset` (имя файла/caption), с фолбэком на локализованную строку «Photo». Проверить, что нужный dataset прокинут на плитку-триггер; если нет — добавить его в HEEx-разметку плитки.
- [ ] **R067 — лайтбокс, управление фокусом:** сохранять `box.__prevFocus = document.activeElement` в `openLightbox()` перед `box.classList.add("ed-lightbox--open")` (8326); в `close()` (8355-8359) после снятия класса вызывать `box.__prevFocus && box.__prevFocus.focus()`; при открытии переносить фокус на кнопку закрытия (`box.querySelector(".ed-lightbox__close").focus()`).
- [ ] **R067 — лайтбокс, Tab-ловушка:** расширить `box.__onKey` (8360-8364), добавив ветку `e.key === "Tab"`, зацикливающую фокус между `.ed-lightbox__close`/`.ed-lightbox__nav` (переиспользовать логику `_focusables()` из `.FocusTrap`, 4610-4614; либо вынести общий helper). Учитывать, что при `tiles.length <= 1` кнопки nav скрыты (`ed-lightbox--gallery` не выставлен, 8323) — фильтровать по видимости, как эталон (`el.offsetParent !== null`).
- [ ] **R068 — видео-модалка, доступность контейнера:** в `modal()` (`chat_live.ex:8486-8488`) навесить `box.setAttribute("role", "dialog")` + `box.setAttribute("aria-modal", "true")` (+ `aria-label`).
- [ ] **R068 — видео-модалка, управление фокусом:** сохранять `box.__prevFocus = document.activeElement` в `open()` перед `box.classList.add("ed-video-modal--open")` (8476); в `close()` (8496-8507) возвращать фокус на триггер; при открытии переносить фокус на `<video>` или кнопку закрытия. Tab-ловушка опциональна (внутри всего два фокусируемых: крестик + `<video controls>`), но для единообразия желательна — переиспользовать тот же helper, что и лайтбокс.
- [ ] **Дедуп/рефактор (по возможности):** лайтбокс и видео-модалка получают идентичную триаду save-focus / move-in / restore-on-close + Tab-loop. Вынести общий JS-helper (напр. модуль или функцию в общем colocated-скрипте) и переиспользовать в обоих хуках, чтобы не дублировать логику `.FocusTrap` в трёх местах. Сделать это ПОСЛЕ того как оба фикса заработают точечно — сначала корректность, потом дедуп.

## Критерии приёмки

- Все три модальных слоя (`#ed-lightbox`, `#ed-video-modal`, `compose_overlay`) имеют `role="dialog"` + `aria-modal="true"`, переносят фокус внутрь при открытии, зацикливают Tab в пределах диалога и возвращают фокус на триггер при закрытии.
- Увеличенное изображение в лайтбоксе имеет осмысленный, локализованный `alt` (не пустая строка).
- Существующее поведение не регрессирует: Escape/клик-по-скриму/свайпы (лайтбокс #95/#96) и пагинация альбома работают как прежде; композер-бар в `compose_overlay` остаётся `inert` и недостижим Tab-ом.
- `mix check` зелёный (форматирование HEEx/скриптов, credo); e2e-специи проходят на chromium/firefox/webkit.

## Заметки

- Все три находки подтверждены чтением актуального кода; номера строк из входа не сместились (8337-8353, 8486-8494, 10948-10956 совпадают). R067 — verified/high; R068 и R073 — advisory (без независимой верификации), но при чтении кода полностью подтверждаются, confidence high.
- Развилка для человека/дизайна по R067 (alt): нужно решить источник подписи увеличенного фото — имя файла, серверная caption сообщения или общий фолбэк «Photo». Если у плитки-триггера ещё нет подходящего dataset-поля, потребуется прокинуть его из HEEx-разметки медиа-плитки — это уже не чисто-JS правка. Остальные шаги механические и локализованные.
- `profile_popover` (`chat_live.ex:11590`) тоже несёт `role="dialog"` без `.FocusTrap`, но это popover с собственным хуком `.Popover` (не модальный диалог с scrim-inert-паттерном) — сознательно вне scope этого issue; при желании отдельно оценить его a11y.
- Приоритет P1→P2 по R067 понижен верификатором осознанно (см. `verdict_note`): по рубрике проекта P1 — корректность логики/данных, а это a11y/UX-дефект.
