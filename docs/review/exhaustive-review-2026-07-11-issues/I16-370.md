# Треды: починить корректность unread-счётчиков, cap панели, tab-gate и facepile

**Приоритет:** P2 — набор корректностных/UX-дефектов подсистемы тредов: бейдж непрочитанных врёт (в обе стороны), длинный тред обрезается на старейших ответах, фоновая вкладка «читает» тред за пользователя, а facepile показывает призрачных участников. Ни одна находка не даёт краш/потерю данных (латентный CastError в R140 — узкий), поэтому P2, хотя часть находок на границе P1/P2 (корректность бейджа).
**Лейблы:** `bug` `P2: medium` `backend` `ready-for-agent`
**Находки ревью:** R032, R034, R054, R055, R124, R129, R135, R140, R177

## Контекст
Все девять находок относятся к подсистеме тредов (#57 — per-user follow + per-thread unread, Mattermost-style) в `Eden.Chat` и её web-обвязке (`ChatLive` + `ThreadPanel`). Общий корень — **инкрементальное ведение `thread_memberships.unread_replies`** и производных (facepile, панель, бейдж): счётчик двигают дельтами (`+1` в `track_reply`, `-1` в `sync_thread_after_delete`), а не пересчётом, и эвристика «кому засчитан удалённый реплай» использует только `last_viewed_at`, игнорируя момент подписки (`inserted_at`). Отсюда две зеркальные ошибки бейджа (R032/R135 — врёт вниз; R129 — врёт вверх), плюс несколько связанных дефектов открытия/жизненного цикла тред-панели (R034 cap, R054 perf fan-out, R055 tab-gate, R177 facepile) и два гигиенических (R124 leave-cleanup, R140 нормализация id). Собраны вместе, потому что чинятся в одних и тех же функциях (`sync_thread_after_delete`, `list_thread`, `thread_follow_state`, handle_info `{:thread_reply}`/`{:thread_updated}`).

## Находки и доказательства

### R032 · P2/bug · `lib/eden/chat.ex:2743` — декремент unread у фолловера, которому удалённый реплай не был засчитан
- **Проблема:** `sync_thread_after_delete` декрементит `unread_replies` любому фолловеру с `unread>0` и `(last_viewed_at IS NULL OR last_viewed_at < reply_at)`. Строка ручной подписки (`follow_thread`) создаётся с `last_viewed_at=nil` уже ПОСЛЕ существующих реплаев — её `unread` считается только с момента подписки. Удаление до-подписочного реплая матчит гейт `is_nil(last_viewed_at)` и занижает честный счётчик.
- **Доказательство в коде:** `sync_thread_after_delete` (актуально на строках 2743–2748):
  ```elixir
  from(tm in ThreadMembership,
    where:
      tm.root_id == ^root_id and tm.unread_replies > 0 and
        (is_nil(tm.last_viewed_at) or tm.last_viewed_at < ^reply_at)
  )
  |> Repo.update_all(inc: [unread_replies: -1])
  ```
  `follow_thread/2` (2895–2903) → `upsert_thread_membership(user.id, root.id, following: true)` (3083) создаёт строку с `last_viewed_at: nil` (в схеме `field :last_viewed_at, :utc_datetime` без дефолта, `thread_membership.ex:22`). При этом `track_reply/2` (3057–3078) инкрементит `unread_replies` только у уже существующих following-строк, то есть счётчик подписчика растёт лишь с момента подписки. Гейт `is_nil(last_viewed_at)` не отличает «подписан до реплая» от «подписан после». Для авто-фолловеров (автор рута через `ensure_following`, 3099; репаер) гейт `IS NULL` корректен — они подписаны ДО любого реплая.
- **Влияние:** бейдж непрочитанных тредов врёт в меньшую сторону — пользователь пропускает реально непрочитанный ответ. Сценарий: реплаи r1–r3, Кэрол делает `follow_thread` (unread=0), приходит r4 (unread=1), кто-то делает `delete_message_for_both(r1)` — её unread падает в 0, хотя r4 непрочитан. (verified, confidence medium; узкий edge.)

### R135 · P3/bug · `lib/eden/chat.ex:2743` — та же неточная эвристика по `last_viewed_at` (дубль корня R032)
- **Проблема:** Формулировка того же дефекта, что и R032, с зеркального угла: инкрементальный декремент по `last_viewed_at` не гарантирует, что удалённый реплай вообще был засчитан этому фолловеру. Одна и та же строка кода (2743–2748), один и тот же фикс.
- **Доказательство в коде:** см. R032 (тот же фрагмент `sync_thread_after_delete`). Находка предлагает то же: гейтить по `inserted_at` строки подписки, либо перейти на пересчёт `count(replies where inserted_at > coalesce(last_viewed_at, followed_at))`.
- **Влияние:** идентично R032 — занижение бейджа. (advisory, confidence medium.) **Сводится в один пункт плана вместе с R032.**

### R129 · P3/ux · `lib/eden/chat.ex:2505` — delete-for-me непрочитанного реплая не декрементит собственный unread
- **Проблема:** `delete_message_for_me` не имеет тред-ветки: скрытый для себя реплай пропадает из `list_thread` (там `left_join MessageDeletion ... where is_nil(d.id)`), но `unread_replies` скрывшего не уменьшается — `sync_thread_after_delete` вызывается только из `delete_message_for_both`.
- **Доказательство в коде:** `delete_message_for_me/2` (2505–2520) вставляет только `MessageDeletion` и шлёт `{:message_hidden}`, без единого обращения к `ThreadMembership`:
  ```elixir
  def delete_message_for_me(%Scope{user: user} = scope, message_id) do
    with {:ok, message} <- fetch_message(scope, message_id),
         {:ok, _deletion} <-
           %MessageDeletion{}
           |> Ecto.Changeset.change(message_id: message.id, user_id: user.id)
           |> Repo.insert(on_conflict: :nothing, conflict_target: [:message_id, :user_id]) do
      ...
  ```
  `sync_thread_after_delete` дергается только на строке 2678 (внутри `delete_message_for_both`), не здесь. `list_thread/2` (2810–2826) действительно фильтрует скрытые реплаи джойном `MessageDeletion`.
- **Влияние:** бейдж треда считает ответ, который пользователь никогда не увидит (в списке треда его нет); сбрасывается только полным открытием (`mark_thread_read`). Бейдж «врёт вверх» — зеркало R032/R135. (advisory, confidence medium; мелко.)

### R034 · P2/product · `lib/eden/chat.ex:2818` — `list_thread` обрезает тред до 500 СТАРЕЙШИХ реплаев без пагинации
- **Проблема:** Реплаи выбираются `order_by asc: m.id |> limit(@thread_cap)` — при >500 реплаев панель показывает первые 500 по возрасту; свежие ответы (501+) при открытии треда не рендерятся. Живые приходят только через `{:thread_reply}` до перезагрузки панели. Футер честно показывает растущий `reply_count`; permalink на реплай за капом сфокусирует несуществующий DOM-узел.
- **Доказательство в коде:** `list_thread/2` (2810–2826):
  ```elixir
  replies =
    Message
    |> where([m], m.root_id == ^root.id and is_nil(m.deleted_at))
    |> join(:left, [m], d in MessageDeletion, on: d.message_id == m.id and d.user_id == ^user.id)
    |> where([_m, d], is_nil(d.id))
    |> order_by([m], asc: m.id)
    |> limit(@thread_cap)   # @thread_cap = 500 (chat.ex:2763)
    ...
  ```
- **Влияние:** в активном длинном треде диалог «обрывается» на старине — пользователь открывает тред и видит только древние ответы, хотя счётчик говорит о новых. Slack/Mattermost показывают хвост (новейшие). (advisory, confidence high.)

### R055 · P2/ux · `lib/eden_web/live/chat_live.ex:2375` — `mark_thread_read` по `{:thread_reply}` не гейтится `tab_visible`
- **Проблема:** В `handle_info({:thread_reply,...})` `mark_thread_read` вызывается всегда, когда панель открыта (`viewing?`), без проверки `tab_visible`. Main-stream аналог (`stream_new_message`) помечает прочитанным только в фокусной вкладке (#206).
- **Доказательство в коде:** `chat_live.ex:2372–2375`:
  ```elixir
  def handle_info({:thread_reply, root, reply}, socket) do
    viewing? = ThreadPanel.thread_open_for?(socket, root.id)
    # Reading the thread keeps it read on the server (the DB just incremented it).
    if viewing?, do: Chat.mark_thread_read(socket.assigns.current_scope, root.id)
  ```
  Против main-stream (`stream_new_message`, 12787): `if message.sender_id != ... and socket.assigns.tab_visible do Chat.mark_read(...)`. Обработчик `"tab_visible"` (1092–1101) дочитывает открытый чат при возврате, но НЕ открытый тред.
- **Влияние:** пользователь оставил панель треда открытой и свернул браузер — входящие реплаи мгновенно помечаются прочитанными, `{:thread_read}` (chat.ex:2933–2937) обнуляет бейдж во всех вкладках/устройствах. Вернувшись, он не видит новых ответов — та же проблема, что #206 закрыл для основного потока. (advisory, confidence high.)

### R054 · P2/perf · `lib/eden_web/live/chat_live.ex:2372` — `{:thread_reply}` гоняет DB-запрос `thread_follow_state` на каждый ответ
- **Проблема:** `handle_info({:thread_reply, root, reply})` при `open?(root.conversation_id)` всегда зовёт `ThreadPanel.sync_thread_unread(root.id)` → `Chat.thread_follow_state` (SELECT по `thread_memberships`), даже если сессия не следит за тредом и не открыла его.
- **Доказательство в коде:** `chat_live.ex:2377–2388` безусловно вызывает `sync_thread_unread` при открытой комнате; `thread_panel.ex:73–83` внутри всегда делает `Chat.thread_follow_state(...)` (chat.ex:2947–2956, `Repo.one` SELECT). **Важно (independent verify):** fan-out переоценён — подписка на топик комнаты только на ОТКРЫТУЮ комнату (`select_conversation` unsub/sub по единичному id), т.е. запрос идёт у активно-смотрящих сессий, а не у всех членов комнаты. Плюс это одиночный индексируемый SELECT, и комментарий на 2380 объясняет, зачем нужен серверный источник (авто-подписанный автор рута без локального ключа — чисто локальным чеком не решить).
- **Влияние:** мягкая perf-заметка: лишний SELECT на каждый reply у сессий, которые точно не следят и не открыли тред. При масштабе оживлённых тредов — заметный, но не критичный оверхед. (verified, confidence **low**; формулировка находки про «все сессии/всех членов» преувеличена — фиксировать как оптимизацию, не как баг.)

### R177 · P3/ux · `lib/eden_web/live/chat_live.ex:2415` — facepile не очищается при удалении последнего реплая
- **Проблема:** `handle_info({:thread_updated, root})` пере-считывает `thread_participants` и делает `Map.merge` со старым ассайном. Когда удалённый реплай был единственным, `thread_participants` не вернёт ключ `root.id` — `merge` оставляет старых участников: facepile показывает репаера, хотя `reply_count` уже 0.
- **Доказательство в коде:** `chat_live.ex:2408–2416`:
  ```elixir
  participants =
    Chat.thread_participants(socket.assigns.current_scope, root.conversation_id, [root.id])
  socket
  |> ThreadPanel.restream_root_if_loaded(root)
  |> assign(:thread_participants, Map.merge(socket.assigns.thread_participants, participants))
  ```
  `Chat.thread_participants/3` (2857–2888): при отсутствии не-удалённых реплаев `rows` пуст → результат `%{}` (ключа `root.id` нет), поэтому `Map.merge(old, %{})` == `old`, стариков не стирает.
- **Влияние:** визуальная ложь в футере треда до пере-выбора комнаты: аватарки участников без единого ответа. (advisory, confidence medium.)

### R124 · P3/quality · `lib/eden/chat.ex:1181` — `leave_rooms` не чистит `ThreadMembership`
- **Проблема:** Выход из канала/комнаты hard-делитит только `Membership`; строки `thread_memberships` остаются (каскад лишь по руту/пользователю). Фан-аут `track_reply` инкрементит following-строки без проверки членства — счётчики экс-участников растут вечно; при повторном входе всплывает накопленный unread по старым тредам.
- **Доказательство в коде:** `leave_rooms/2` (1181–1191) удаляет только `Membership`:
  ```elixir
  def leave_rooms(channel_id, user_id) do
    Repo.delete_all(
      from m in Membership,
        join: c in Conversation, on: c.id == m.conversation_id,
        where: c.channel_id == ^channel_id and m.user_id == ^user_id
    )
    :ok
  end
  ```
  `track_reply/2` (3069–3074) инкрементит все `following == true` строки без джойна `Membership`. Утечки при этом нет: `thread_unread_counts`/`list_followed_threads` гейтятся `member?`, а нотификации джойнят `Membership` — счётчики остаются мёртвыми до re-join.
- **Влияние:** мёртвые строки + сюрприз-бейдж при повторном входе. Спорно как продукт (реплаи и правда непрочитаны, но пользователь осознанно уходил). (advisory, confidence medium.) **Требует продуктовой развилки — см. Заметки.**

### R140 · P3/quality · `lib/eden/chat.ex:2947` — `thread_follow_state` не нормализует `root_id`
- **Проблема:** Единственная thread-функция без `safe_id`/`fetch_message`: `tm.root_id == ^root_id` с сырым аргументом. Нечисловой строковый id (например из будущего вызова с `params["root_id"]`) даст `Ecto.Query.CastError`; членство не проверяется (утечки нет — читается только своя строка, но паттерн выбивается).
- **Доказательство в коде:** `thread_follow_state/2` (2947–2956):
  ```elixir
  def thread_follow_state(%Scope{user: user}, root_id) do
    case Repo.one(
           from tm in ThreadMembership,
             where: tm.user_id == ^user.id and tm.root_id == ^root_id,
             select: {tm.following, tm.unread_replies}
         ) do
      {following, unread} -> %{following: following, unread: unread}
      nil -> %{following: false, unread: 0}
    end
  end
  ```
  Сравни с `fetch_message/2` (3405): `with id when is_integer(id) <- safe_id(message_id)`. `safe_id/1` == `Eden.Ids.normalize/1` (3988). Все текущие вызовы (`sync_thread_unread`) передают integer из авторизованного рута — латентный, не активный краш.
- **Влияние:** латентный CastError: первый же вызов с `params["root_id"]` уронит LiveView. (advisory, confidence medium; профилактика.)

## План реализации

- [ ] **Гейтить декремент по моменту подписки (R032 + R135, один фикс).** В `sync_thread_after_delete/1` (chat.ex:2743–2748) добавить в `where` условие, что строка подписки существовала на момент удалённого реплая: заменить гейт на
  `(is_nil(tm.last_viewed_at) and tm.inserted_at <= ^reply_at) or tm.last_viewed_at < ^reply_at`.
  Так авто-фолловеры (подписаны до реплая) сохраняют текущее корректное поведение, а ручная подписка после реплая (R032) больше не теряет честный unread. Альтернатива на будущее (не в этом issue): пересчёт `count(replies where inserted_at > coalesce(last_viewed_at, followed_at))` — требует хранения `followed_at`; здесь берём минимальный гейт по `inserted_at`.
- [ ] **Декремент собственного unread при delete-for-me реплая (R129).** В `delete_message_for_me/2` (chat.ex:2505) после успешной вставки `MessageDeletion`, если `message.root_id != nil`, декрементить строку скрывшего: `update_all` по `(user_id == user.id, root_id == message.root_id)` с гейтом `unread_replies > 0 and (is_nil(last_viewed_at) or last_viewed_at < message.inserted_at)` (тот же гейт, что в п.1). Вынести общий гейт в приватный хелпер (напр. `decrement_thread_unread(query, reply_at)`), чтобы не дублировать логику с `sync_thread_after_delete`. Порядок: делать ПОСЛЕ п.1, чтобы хелпер уже был исправлен.
- [ ] **Показывать хвост треда, а не старейшие (R034).** В `list_thread/2` (chat.ex:2810–2826) заменить `order_by asc |> limit` на «новейшие `@thread_cap` в хронологическом порядке»: `order_by desc: m.id |> limit(@thread_cap)` в подзапросе, затем `Enum.reverse` результата (или обернуть в `from(r in subquery(...), order_by: [asc: r.id])`). Открытие треда всегда показывает актуальный хвост. Полноценная пагинация «load older» (before-курсор, как `list_messages/3`) — отдельный follow-up, отметить в Заметках.
- [ ] **Гейтить `mark_thread_read` по `tab_visible` (R055).** В `handle_info({:thread_reply,...})` (chat_live.ex:2375) заменить `if viewing?, do: ...` на `if viewing? and socket.assigns.tab_visible, do: Chat.mark_thread_read(...)`. В `handle_event("tab_visible", ...)` (chat_live.ex:1092–1101) при возврате дочитывать открытый тред: если `thread_root` присутствует, вызвать `Chat.mark_thread_read(scope, thread_root.id)` (по аналогии с `mark_read` открытого чата).
- [ ] **Явно класть facepile для затронутого рута (R177).** В `handle_info({:thread_updated, root})` (chat_live.ex:2413–2416) заменить `Map.merge(...)` на явную установку значения именно для `root.id`:
  `Map.put(socket.assigns.thread_participants, root.id, Map.get(participants, root.id, []))`.
  Тогда пустой результат (последний реплай удалён) очищает facepile, а не оставляет стариков.
- [ ] **Нормализовать `root_id` в `thread_follow_state` (R140).** В `thread_follow_state/2` (chat.ex:2947) обернуть: `with id when is_integer(id) <- safe_id(root_id), ...` и возвращать дефолт `%{following: false, unread: 0}` для не-integer — приводит функцию к общему паттерну контекста, устраняет латентный `Ecto.Query.CastError`.
- [ ] **(perf, опционально) Ленивый `sync_thread_unread` (R054).** В `handle_info({:thread_reply,...})` (chat_live.ex:2382) пропускать `sync_thread_unread`, когда `root.id` заведомо не отслеживается: если `root.id` нет в `socket.assigns.thread_unreads` И тред не открыт (`not viewing?`) И пользователь не автор рута (`reply.sender` / `root.sender_id != current user`, чтобы покрыть кейс из комментария на 2380) — обновлять счётчик локально без похода в БД. Реализовать аккуратно, сохранив авто-подписку автора рута; при сомнении оставить как есть и закрыть находку как «by design» (см. Заметки — низкая уверенность). **Не блокирует остальные пункты.**
- [ ] **(product, требует решения) `leave_rooms` и `ThreadMembership` (R124).** ЛИБО: в `leave_rooms/2` (chat.ex:1181) добавить `Repo.delete_all` по `thread_memberships`, где `root_id` принадлежит покидаемым комнатам (join `Message` по `conversation_id`/`channel_id`). ЛИБО: зафиксировать текущее поведение в `CONTEXT.md` как осознанное и добавить членский джойн в фан-аут `track_reply/2` (3069), чтобы экс-участники не копили unread. Выбор — за человеком (см. Заметки).

## Критерии приёмки
- R032/R135: удаление до-подписочного реплая не занижает unread ручного подписчика; авто-фолловеры сохраняют прежнее корректное поведение (существующий тест 1467 зелёный).
- R129: delete-for-me реплая декрементит собственный unread скрывшего, не трогая других фолловеров.
- R034: открытие треда с >500 реплаями показывает новейший хвост; свежий ответ виден сразу при открытии.
- R055: реплай в свёрнутую/невидимую вкладку не помечает тред прочитанным на сервере; возврат во вкладку дочитывает открытый тред.
- R177: facepile очищается, когда удалён последний реплай треда.
- R140: `thread_follow_state` с мусорным id возвращает дефолт, не роняет процесс.
- R054 (если сделан) / R124 (если сделан): подтверждено тестом; при отказе — решение задокументировано в CONTEXT.md.
- `mix check` зелёный (format / warnings-as-errors / credo --strict / sobelow / deps.audit / test), `mix dialyzer` зелёный; регрессий в тред-тестах нет.

## Заметки
- **R032 и R135 — одна и та же строка/дефект** (chat.ex:2743), сведены в один пункт плана и один тест-кейс; оба id перечислены для трассируемости. R129 — зеркальный симптом того же инкрементального подхода (врёт вверх), чинится тем же общим хелпером-гейтом.

Пункт помечен опциональным; допустимо закрыть находку как «acceptable / by design» без изменений.
