# Убрать мёртвый Phoenix-скаффолд и починить страницу ошибки, индикатор связи и рассинхрон комментариев/доков

**Приоритет:** P2 — поддерживаемость и визуальная согласованность: мёртвый Phoenix-скаффолд (`Layouts.app`) — ловушка для будущих агентов (AGENTS.md обязывает начинать шаблоны с `<Layouts.app>`), страница ошибки игнорирует светлую тему и мажет мимо фирменного токена, а несколько комментариев/строк CLAUDE.md рассинхронены с кодом. Ни одна находка не является крашем/потерей данных.
**Лейблы:** `enhancement` `P2: medium` `UI/UX` `backend` `ready-for-agent`
**Находки ревью:** R043, R044, R110, R125, R161, R162, R163, R172, R173

## Контекст
Пучок находок из код-ревью по «санитарной гигиене» веб-слоя и документации. Общий корень у большинства — **дефолтный скелет `phx.new`, который так и не подчистили**: `EdenWeb.Layouts.app/1` (Phoenix-брендинг, daisyUI, ссылки на phoenixframework.org), сопутствующие `flash_group/1` и `theme_toggle/1` — мёртвый код, а приложение оборачивает каждый LiveView в собственный `ed-root` и рисует `root.html.heex` через `{@inner_content}` напрямую. Из этого же корня растут два UX-пробела: **баннер переподключения** живёт только внутри неиспользуемого `flash_group`, поэтому page-level индикатора состояния связи нет (при том что «честность про связь» — заявленная ценность), и **страница ошибки** (`ErrorHTML`) захардкожена под тёмную тему мимо `--ed-*` токенов. Остальные находки (R110/R125/R172/R173) — точечные расхождения комментариев/CLAUDE.md с фактическим поведением кода, сгруппированные сюда как единый проход по «правде в документации».

- **Влияние:** PRODUCT.md принцип 5 — «UI честно сообщает о состоянии связи (reconnecting, sending, sent)». На flaky cross-border линке пользователь видит «зависший» интерфейс без сигнала о потере соединения. ### R044 · P2/quality · `lib/eden_web/controllers/error_html.ex:36-50` — страница ошибки всегда тёмная + цвет мимо токена
- **Проблема:** инлайн-стили 404/500 захардкожены под тёмную тему (`color-scheme: dark`, dark-значения `--ed-bg`/`--ed-ink`), а цвет ссылки-кнопки/кода — `oklch(0.62 0.16 261)`, тогда как фирменный `--ed-primary = oklch(0.55 0.16 261)`.
- **Доказательство в коде:**
  ```elixir
  # lib/eden_web/controllers/error_html.ex:36
  :root { color-scheme: dark; }
  ...
  background: oklch(0.18 0.012 261); color: oklch(0.96 0.005 261);   /* = dark --ed-bg / --ed-ink */
  ...
  .code { ... color: oklch(0.62 0.16 261); }                          /* ≠ --ed-primary */
  a { ... background: oklch(0.62 0.16 261); color: oklch(0.99 0.01 261); }
  ```
  Сверка с токенами (`assets/css/app.css`): светлая тема `--ed-bg: oklch(0.995 0.001 261)`, `--ed-ink: oklch(0.24 0.012 261)`; `--ed-primary: oklch(0.55 0.16 261)` (одинаков в light/dark), `--ed-on-primary: oklch(0.99 0.01 261)`. То есть страница жёстко тёмная и красит кнопку/код в **более светлый** синий, чем бренд. Confidence **high**.
- **Влияние:** пользователь на светлой теме получает резко-тёмную страницу ошибки — разрыв визуальной согласованности («Три темы, одно ощущение»), плюс оттенок кнопки не фирменный кобальт. localStorage-тему тут не прочитать (страница может рендериться при недоступных ассетах), поэтому опора — на OS через `prefers-color-scheme`.

### R110 · P3/quality · `CLAUDE.md:58-59` — «the UI signs them back in» после смены пароля не соответствует коду
- **Проблема:** CLAUDE.md утверждает, что после `change_password` «the UI signs them back in», но SettingsLive после успеха уводит на `/login` с флешем «Please sign in again» — авто-перелогина нет (и UI-копия формы честно обещает «signs you out of every device»).
- **Доказательство в коде:**
  ```elixir
  # lib/eden_web/live/settings_live.ex:1344
  def handle_event("change_password", %{"password" => %{"current" => cur, "new" => new}}, socket) do
    case Accounts.change_password(socket.assigns.profile_user, cur, new) do
  ```

  UI-копия формы: `settings_live.ex:503` — `gettext("Changing your password signs you out of every device.")`. Против CLAUDE.md:58-59: «`change_password/3` … then **revokes every session** — the UI signs them back in». Confidence **medium/advisory**.
- **Влияние:** карта архитектуры — вход для агентов/ревьюеров; расхождение уже стоило неверных предсказаний (урок «verify-main-before-predicting»).

- **Проблема:** комментарий в `folder_unread_counts` утверждает, что DM может нести треды, но инвариант rooms-only фиксирован в коде и CLAUDE.md.

### R172 · P3/quality · `lib/eden_web/live/chat_live.ex:44` — комментарий называет chunk_size 512KB, фактически 1MB
- **Проблема:** внутреннее противоречие в шапке модуля: комментарий `@upload_chunk_timeout` (37-41) корректно оперирует «1MB», а комментарий ниже (44) говорит «512KB».
- **Доказательство в коде:**

  Источник истины: `lib/eden/chat.ex:94` — `@upload_chunk_size 1_000_000`; `upload_chunk_size/0` (112), `upload_max_frame_size/0 = @upload_chunk_size * 4` (115). Значение бампалось (512KB → 1MB), комментарий на 44 не обновили. Confidence **high**.
- **Влияние:** вводит в заблуждение при подборе `chunk_timeout`/`max_frame_size` (они выведены из chunk_size); «512KB» искажает оценку RTT-бюджета и таймингов столлов.

### R173 · P3/quality · `lib/eden_web/live/chat_live.ex:109-110` — устаревший комментарий про форму `sel_delete` (`all_mine` vs `for_all`)
- **Проблема:** комментарий инициализации описывает `sel_delete` как `%{count, all_mine}`, но хендлер кладёт `%{count, for_all}`, и шаблон читает `.for_all`. Ключа `all_mine` нет — поведение корректно, устарел лишь комментарий.
