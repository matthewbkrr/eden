# Channels-контекст: закрыть тест-пробелы (ensure_member, transfer_ownership) и выровнять rescue в join_by_token

**Приоритет:** P2 — поддерживаемость/краевые случаи: две центральные ветки авторизации не зафиксированы тестами (регрессия прошла бы молча), плюс асимметрия обработки гонок, способная в редком окне уронить контроллер вместо аккуратного flash. Ни потери данных, ни доказанного live-бага (см. Заметки по R023).
**Лейблы:** `bug` `P2: medium` `backend` `ready-for-agent`
**Находки ревью:** R020, R022, R023

## Контекст
Все три находки — в `lib/eden/channels.ex`, корпоративном слое (эпик #26). Две из них (R020, R022) — тест-пробелы на путях, реализующих ключевые инварианты авторизации канала: без-приглашения auto-join (#41) и снятие owner-блокировки выхода (`transfer_ownership`). Третья (R023) — хардненинг: путь редемпции по инвайт-токену `join_by_token` не защищён `rescue Postgrex.Error`, в отличие от структурно идентичного `approve_pending`. Общий корень — тонкая транзакционная логика Channels (count-check от ownerless-состояния, идемпотентные `insert_all … on_conflict: :nothing`, FK-гонки при параллельном удалении канала/комнаты) недостаточно закреплена: одна ветка защищена и покрыта, симметричная — нет. Issue сводит эти пробелы в один заход по контексту.

## Находки и доказательства

### R020 · P2/test-gap · `lib/eden/channels.ex:187` — нет прямого теста контракта `ensure_member/2`
- **Проблема:** `ensure_member/2` — единственная точка входа в корпоративный слой без приглашения (#41: любой аутентифицированный пользователь вступает, посетив ссылку). У функции нет собственного `describe` в `test/eden/channels_test.exs` (в списке describe’ов её нет — есть `list/get scoping`, `members`, `invite links` и т.д.); в тестах она фигурирует лишь как фикстура-хелпер в `chat_test.exs`. Не зафиксированы: (а) non-member реально вступает и материализуется в `general`, (б) идемпотентность (повторный вызов — тот же канал, без нового членства и без лишних броадкастов), (в) `:not_found` на несуществующем/мусорном `channel_id` (existence не утекает), (г) сами броадкасты `announce_join`.
- **Доказательство в коде:**
  ```elixir
  def ensure_member(%Scope{} = scope, channel_id) do
    case get_channel(scope, channel_id) do
      {:ok, channel} -> {:ok, channel}
      {:error, :not_found} -> auto_join(scope, channel_id)
    end
  end

  defp auto_join(%Scope{user: user} = scope, channel_id) do
    with id when is_integer(id) <- Ids.normalize(channel_id),
         true <- Repo.exists?(from c in Channel, where: c.id == ^id) do
      {:ok, joined} = Repo.transact(fn -> {:ok, join_channel_tx(id, user.id)} end)
      if joined, do: announce_join(id, user.id)
      get_channel(scope, id)
    else
      _ -> {:error, :not_found}
    end
  end

  defp announce_join(channel_id, user_id) do
    broadcast_user(user_id, :channels_changed)
    broadcast_channel(channel_id, {:members_changed, channel_id})
  end
  ```
  `join_channel_tx/2` (`channels.ex:642-665`) возвращает `false` для уже-члена (insert пропущен), поэтому `announce_join` вызывается только при реальном вступлении — это и есть непокрытый инвариант «повторный вызов не шлёт лишних броадкастов». `Ids.normalize/1` на мусорном id даёт `:error`, что уходит в `else -> {:error, :not_found}`.
- **Влияние:** регрессия на единственном без-приглашения входе (случайно перестало вступать; или начало утекать существование канала возвратом чего-то отличного от `:not_found`; или лишний броадкаст на no-op) прошла бы незамеченной — контракт нигде напрямую не зафиксирован. `confidence=high`, `status=advisory` (найдено ревьюером, независимо не переподтверждалось верификатором).

### R022 · P2/test-gap · `lib/eden/channels.ex:758` — `transfer_ownership` слабо покрыт краевыми случаями
- **Проблема:** на `transfer_ownership/3` есть ровно один тест — `channels_test.exs:551`, happy-path внутри общего сценария `describe "members"`. Не покрыты: передача самому себе (`{:error, :self}`), передача не-члену/мусорному id (`:not_found`), попытка не-владельцем (`:forbidden`), и — главное — гонка в `transfer_tx` (строки 778-793), где цель ушла между чтением роли и записью: count-check `{1, _}` должен провалиться, транзакция откатывается, канал НЕ остаётся без owner. Именно этот откат — критичная защита от ownerless-состояния — не проверен.
- **Доказательство в коде:**
  ```elixir
  def transfer_ownership(%Scope{user: actor} = scope, channel_id, user_id) do
    with {:ok, channel} <- get_channel(scope, channel_id),
         :ok <- ensure_role(channel.role, ~w(owner)),
         target_id when is_integer(target_id) <- Ids.normalize(user_id),
         false <- target_id == actor.id,
         target_role when is_binary(target_role) <- role_of(channel.id, target_id),
         :ok <- transfer_tx(channel.id, actor.id, target_id) do
      broadcast_channel(channel.id, {:members_changed, channel.id})
      broadcast_user(target_id, :channels_changed)
      :ok
    else
      true -> {:error, :self}
      nil -> {:error, :not_found}
      :error -> {:error, :not_found}
      {:error, _} = error -> error
    end
  end

  # The target's promotion is count-checked: if they left between the read and
  # the write, rolling back keeps the channel from ending up ownerless.
  defp transfer_tx(channel_id, actor_id, target_id) do
    Repo.transact(fn ->
      case update_role(channel_id, target_id, "owner") do
        {1, _} ->
          update_role(channel_id, actor_id, "admin")
          {:ok, :ok}

        _ ->
          {:error, :not_found}
      end
    end)
    |> case do
      {:ok, :ok} -> :ok
      error -> error
    end
  end
  ```
  Разбор веток `else`: `false <- target_id == actor.id` при self даёт `true` → `{:error, :self}`; `role_of/2` для не-члена возвращает `nil` → `{:error, :not_found}`; `Ids.normalize/1` на мусоре → `:error` → `{:error, :not_found}`; `ensure_role` при не-owner → `{:error, :forbidden}` (через `{:error, _} = error`). Гонка: `role_of` вернул роль (цель ещё член), но к моменту `update_role(…, "owner")` членство исчезло → `{0, _}` ≠ `{1, _}` → `{:error, :not_found}`, actor остаётся owner.
- **Влияние:** `transfer_ownership` — единственный способ снять owner-блокировку выхода (owner уходит только после передачи или удаления канала). Тонкая транзакция с count-check — защита от потери владельца; регрессия здесь оставила бы канал ownerless и прошла бы молча. `confidence=medium`, `status=advisory`.

### R023 · P2/bug · `lib/eden/channels.ex:930` — `join_by_token` не оборачивает `Postgrex.Error` в rescue, в отличие от `approve_pending`
- **Проблема:** `approve_pending` (строки 473-498) специально защищён `rescue Postgrex.Error -> {:error, :not_found}` на случай, когда комната/канал удалены между чтением и записью join-ов (`insert_all` даёт FK-violation). `join_by_token` (строки 926-949) выполняет по сути ту же редемпцию (`redeem_invite` → `join_channel_tx` + `redeem_room` → `Chat.join_room`, оба через `Repo.insert_all … on_conflict: :nothing`) внутри `Repo.transact`, но БЕЗ rescue. `on_conflict: :nothing` ловит только unique-конфликты, но НЕ FK-violation при удалённом канале/комнате — та всплывёт как непойманный `Postgrex.Error`.
- **Доказательство в коде:**
  ```elixir
  # approve_pending — ЗАЩИЩЁН:
  end)
  |> case do
    {:ok, :approved} -> ...
    {:error, _} -> {:error, :not_found}
  end
  rescue
    Postgrex.Error -> {:error, :not_found}
  end
  ```
  ```elixir
  # join_by_token — БЕЗ rescue:
  result =
    Repo.transact(fn ->
      case lock_invite(hashed) do
        nil -> {:error, :invalid}
        invite -> redeem_invite(invite, user.id)   # -> join_channel_tx + redeem_room->Chat.join_room
      end
    end)
  ```
  `join_channel_tx` (`channels.ex:642-665`) и `Chat.join_room` (`chat.ex:1036-1043`) — оба `Repo.insert_all(…, on_conflict: :nothing)`. `channel_join_controller.ex:24` имеет catch-all `{:error, reason}` → аккуратный flash, но непойманный `Postgrex.Error` уронит контроллер (500) вместо «мёртвая ссылка».
- **Влияние:** асимметрия обработки одного класса гонок: один путь редемпции устойчив, симметричный может уронить процесс LiveView/контроллера. Verdict верификатора (`status=verified`, но `confidence=low`): окно УЖЕ, чем у `approve_pending` — FK `channel_invites.channel_id/room_id` = `on_delete: delete_all` (миграции `20260611080636` / `20260612085821`), а `lock_invite` берёт инвайт `FOR UPDATE` (`channels.ex:975-977`), так что параллельный DELETE канала/комнаты каскадит на `channel_invites` и блокируется нашим локом до коммита — во время транзакции родительская строка ещё жива, insert членства FK-violation не ловит; практически окно близко к нулю. Это защитная симметрия/хардненинг, не доказанный live-баг → P2 адекватен (не P1: корректность не нарушена благодаря FOR UPDATE; не P3: потенциальный 500 вместо flash).

## План реализации
- [ ] **Выровнять rescue в `join_by_token`** (`channels.ex:926-949`): обернуть тело функции (или как минимум `Repo.transact`-редемпцию) `rescue Postgrex.Error -> {:error, :invalid}` — тегированная ошибка `:invalid` соответствует остальному контракту `join_by_token` («мёртвая ссылка»), тогда как `approve_pending` мапит в `:not_found`. Рассмотреть общий приватный хелпер `rescue_fk_race/1`, оборачивающий редемпцию, чтобы `approve_pending` и `join_by_token` делили одну точку обработки FK-гонки (устраняет будущую асимметрию); если хелпер усложняет — достаточно локального `rescue` по образцу строк 492-498. Проверить, что `channel_join_controller` уже имеет catch-all `{:error, reason}` (строка 24) — новый `:invalid` в него попадёт.
- [ ] **Добавить `describe "ensure_member/2"`** в `test/eden/channels_test.exs` (R020) — см. раздел «Тесты».
- [ ] **Расширить покрытие `transfer_ownership/3`** (R022) — новый `describe` или доп. кейсы в `describe "members"`; см. «Тесты». Порядок не важен: изменения тестов независимы; правку `join_by_token` (код) делать первой, чтобы новый тест на гонку сразу проверял желаемое поведение.

## Тесты
- [ ] `describe "ensure_member/2"` в `test/eden/channels_test.exs` (R020): (а) non-member → `{:ok, channel}`, и он реально член + материализован в `general` (проверить `Chat.room_member?` для general-комнаты или `list_channels` содержит канал); (б) идемпотентность: повторный вызов существующим членом → тот же канал, число `Membership`-строк не выросло; (в) `:not_found` на несуществующем integer-id И на мусорной строке (existence не утекает); (г) броадкасты: подписаться на `user:<id>:channels` и на топик канала, убедиться, что `:channels_changed` + `{:members_changed, id}` приходят при первом вступлении и НЕ приходят при повторном (no-op) вызове.
- [ ] Кейсы `transfer_ownership/3` (R022): передача самому себе → `{:error, :self}`; не-члену → `{:error, :not_found}`; мусорный `user_id` → `{:error, :not_found}`; вызов не-владельцем (admin/member) → `{:error, :forbidden}`; happy-path проверить смену ролей (цель → owner, бывший owner → admin) явно, а не только `:ok`. Отдельный тест на гонку `transfer_tx`: удалить членство цели ПОСЛЕ прохождения `role_of`, но до записи — практично сымитировать, удалив членство цели прямо перед вызовом (тогда `role_of` вернёт `nil` → `:not_found` до транзакции) ЛИБО, для точного покрытия count-check ветки, замокать/подставить сценарий, где `update_role` для цели даёт `{0, _}`; минимально — тест «цель ушла из канала → `transfer_ownership` возвращает `:not_found`, actor остаётся owner (роль не изменилась)».
- [ ] Кейс гонки для `join_by_token` (R023): напрямую сложно сымитировать FK-violation под `FOR UPDATE`; достаточно unit-теста на то, что новый `rescue` возвращает `{:error, :invalid}` (например, через сценарий, где родитель удалён так, что insert ловит FK), либо явно задокументировать, что окно близко к нулю и покрытие — на уровне единообразия ветки rescue. Существующие invite-тесты (`channels_test.exs:558-628`, revoked/expired/exhausted/invalid/garbage) не трогать.

## Критерии приёмки
- Новый `rescue Postgrex.Error -> {:error, :invalid}` в `join_by_token` (или общий хелпер с `approve_pending`); параллельное удаление канала/комнаты во время редемпции даёт тегированную ошибку, а не всплывающий `Postgrex.Error` / 500 в `channel_join_controller`.
- `describe "ensure_member/2"` покрывает вступление, идемпотентность (без лишних броадкастов), `:not_found` (existence не утекает) и оба броадкаста.
- Покрытие `transfer_ownership/3`: ветки `:self` / `:not_found` (не-член + мусор) / `:forbidden` + сценарий «цель ушла → канал не остаётся ownerless».
- `mix check` зелёный (format → compile --warnings-as-errors → credo --strict → sobelow → deps.audit → test); никаких регрессий в существующих `describe "invite links"` / `describe "members"`.

## Заметки
- R023 верифицирован (`status=verified`), но `confidence=low`: это защитная симметрия/хардненинг, а не доказанный live-баг — благодаря `lock_invite` `FOR UPDATE` (строки 975-977) + FK `on_delete: delete_all` на `channel_invites` окно FK-violation практически близко к нулю. Лейбл `bug` поставлен из-за категории находки (потенциальный 500 вместо аккуратного flash — некорректное поведение на грани), но приоритет остаётся P2. Если предпочтителен `enhancement` (чистый хардненинг без доказанного дефекта) — это развилка для триажа человека; я оставил `bug`, следуя правилу «bug если хоть одна находка category=bug».
- R020 и R022 — `status=advisory` (найдены ревьюером, независимо не переподтверждались верификатором), но при чтении кода полностью подтверждаются: собственного `describe "ensure_member/2"` в `channels_test.exs` нет, а `transfer_ownership` встречается в тестах лишь на `channels_test.exs:551`.
- Номера строк во входных находках актуальны (сверено с текущим `lib/eden/channels.ex`): `ensure_member` — 187, `transfer_ownership` — 758, `transfer_tx` — 778-793, `join_by_token` — 926-949, `approve_pending`-rescue — 492-498.
- Развилка по R023: локальный `rescue` (минимум, быстро) против общего хелпера `rescue_fk_race/1` для обоих колл-сайтов (чище, но небольшой рефактор `approve_pending`). Рекомендация — хелпер, если он выходит компактным; иначе локальный rescue.
