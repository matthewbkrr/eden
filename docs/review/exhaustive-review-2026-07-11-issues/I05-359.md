# Закрыть незакрытый класс гонок #258 в Chat: FK-constraint на message-changeset'ах, stale_error_field/delete_all вместо raise, блокировки для GC и 1:1-дедупа

**Приоритет:** P1 — необработанные исключения (`Ecto.ConstraintError` / `Ecto.StaleEntryError`) в путях send/edit/delete роняют LiveView вместо чистого `{:error, _}`; плюс write-skew в GC ведёт к вечной утечке строк/блобов, а гонка 1:1 расщепляет историю. Это дефекты корректности в конкурентных окнах (P1 по рубрике), не потеря видимых данных и не auth-bypass, поэтому не P0.
**Лейблы:** `bug` `P1: critical` `backend`
**Находки ревью:** R007, R008, R027, R028, R030, R033, R035, R036, R037, R038, R042, R107, R130, R134

## Контекст

`Eden.Chat` — единственный контекст мессенджера. В эпике #258 уже начали чинить один класс крашей: bare-struct-insert и `Repo.delete`/`Repo.update` по конкурентно исчезнувшей строке поднимают исключение внутри транзакции/пайплайна вместо тегированной ошибки, а `case`/`with` в контексте обрабатывают только `{:ok}`/`{:error}` — исключение долетает до LiveView-хендлера (`try/rescue` там нет) и валит процесс. Правку применили **точечно** — только к `create_system_message` (`chat.ex:1063-1075`, `foreign_key_constraint`) и `resolve_join_request`/`update_room` (`stale_error_field: :id`), плюс `do_toggle_reaction`/`do_toggle_folder`/`toggle_folder_mute` (`Repo.delete_all`/`update_all`).

Все остальные пути жизненного цикла сообщения остались с тем же незакрытым паттерном: обычная отправка/реплай/форвард/альбом (insert без FK-constraint), soft-delete и media-edit (`Enum.each(&Repo.delete/1)`), edit текста (`{:ok,_} = Repo.update()`), rename/delete папки, ThumbnailWorker (`Repo.update` без stale-guard). Сюда же примыкают три родственные конкурентные проблемы того же корня («check-then-act без блокировки/констрейнта»): write-skew при одновременном выходе двух последних участников (беседа никогда не GC-ится → утечка), гонка 1:1 (два дубликата на пару), и гонка удаления рута против реплая (tombstone-рут с осиротевшими реплаями). Один issue — потому что корень и рецепт лечения общие: либо FK-constraint/`stale_error_field`/`delete_all` (raise → tagged), либо блокировка/условный `update_all` (check-then-act → атомарно).

## Находки и доказательства

### R008 · P1/bug · `lib/eden/chat/message.ex:88-110`, `lib/eden/chat.ex:2025` — обычная отправка без `foreign_key_constraint(:conversation_id)`
### R042 · P2/bug · `lib/eden/chat/message.ex:88`, `102` — тот же дефект, полный список путей вставки

- **Проблема:** Ни `changeset/2`, ни `photo_changeset/2` не несут `foreign_key_constraint` на `:conversation_id`/`:root_id`/`:reply_to_id`. Все пользовательские пути вставки гвардят `has_access?`/`fetch_message` **до** insert; если беседу/комнату конкурентно удалили в TOCTOU-окне между гвардом и `Repo.insert()`, вставка поднимает `Ecto.ConstraintError` вместо `{:error, changeset}`.
- **Доказательство в коде:** `message.ex:88-110` — оба changeset'а заканчиваются только `dedup_constraint()`:
  ```elixir
  def changeset(message, attrs) do
    message
    |> cast(attrs, [:body, :client_id])
    |> update_change(:body, &sanitize/1)
    |> validate_required([:body])
    |> validate_length(:body, max: @max_body, count: :codepoints)
    |> validate_length(:client_id, max: @max_client_id)
    |> dedup_constraint()
  end
  ```
  `create_message` (`chat.ex:2030-2036`) обрабатывает только dedup:
  ```elixir
  %Message{conversation_id: conversation_id, sender_id: user.id, reply_to_id: reply_to_id}
  |> Message.changeset(attrs)
  |> Repo.insert()
  |> case do
    {:ok, message} -> {:ok, deliver(conversation_id, message)}
    {:error, changeset} -> resolve_duplicate(changeset, user.id)
  end
  ```
  Контраст с `create_system_message` (`chat.ex:1067-1074`), где #258 уже добавил `|> Ecto.Changeset.foreign_key_constraint(:conversation_id)`. FK `messages.conversation_id references(:conversations, on_delete: :delete_all), null: false` (миграция `20260606130148_create_chat.exs`). Пути с той же дырой: `create_message` (2030-2032), `create_reply` (2792-2793), `insert_album_message` (4041-4042), `insert_forward` (3511-3512), `insert_forward_reply` (3399-3400).
- **Влияние:** Пользователь отправляет сообщение ровно когда админ удаляет комнату/канал (или GC 1:1 — см. ниже) → краш и ремоунт LiveView с потерей композера, вместо «Thread/Chat not found».

### R008 (ветка потери данных) · `lib/eden/chat.ex:2025` vs `401-418` — insert 1:1 против GC без блокировки

- **Проблема:** В 1:1, где оба «удалили» чат, отправитель B пишет снова (`has_access?` для 1:1 намеренно пермиссивен). GC (`leave_and_maybe_gc`) не лочит строку беседы. Если insert B закоммитился до `delete_all` GC — сообщение каскадится (`on_delete: :delete_all`) после того, как B уже увидел «отправлено» (deliver/broadcast отработали по структуре в памяти) → тихая потеря. Если `delete_all` раньше — сырой `ConstraintError` (см. R008/R042 выше).
- **Доказательство в коде:** `create_message` вставляет без транзакции и без блокировки беседы; GC-транзакция (`chat.ex:402-411`) лочит только СВОЮ membership-строку:
  ```elixir
  Repo.transact(fn ->
    Repo.update_all(from(m in Membership, where: ...), set: [left_at: now()])
    {:ok, if(all_members_left?(conversation_id), do: gc_collect(conversation_id), else: [])}
  end)
  ```
  Комментарий выше (`397-399`) заявляет атомарность против конкурентного deliver, но `SELECT ... FOR UPDATE` строки `conversations` тут нет.
- **Влияние:** Узкое, но легитимное окно: «написать человеку ровно когда он удаляет пустой для себя чат». FK-constraint из R008/R042 превращает ветку-краш в `{:error, :not_found}`; ветку тихой потери закрывает блокировка беседы в GC (см. R028).

### R033 · P2/bug · `lib/eden/chat.ex:2792`, `3400`, `4042` — reply/forward-reply/album без FK-constraint на `:root_id`

- **Проблема:** Те же пути, но с дополнительным FK на `root_id`: если админ удалил комнату (каскад сносит рут) между `fetch_message` рута и insert реплая, `Ecto.ConstraintError` вместо `{:error, :not_found}`.
- **Доказательство в коде:** `create_reply` (`chat.ex:2786-2797`) вставляет `%Message{root_id: root.id}` через `Message.changeset` без FK-constraint; `insert_forward_reply` (`3392-3401`) и `insert_album_message` (`4030-4047`) — через `photo_changeset`. FK `messages.root_id` — `on_delete: :delete_all` (миграция `20260611085902`). Аналогичное окно у `Repo.insert!` в `upsert_thread_membership` (`chat.ex:3084`), но оно уже прикрыто 0-row-гвардом в `deliver_reply` (`3035`) — см. заметку.
- **Влияние:** Отправка реплая в момент удаления комнаты → краш вместо чистой ошибки. (Один и тот же changeset-фикс R008/R042 + добавка `foreign_key_constraint(:root_id)` закрывает это.)

### R007 · P1/bug · `lib/eden/chat.ex:2617` — `apply_media_edit`: `Enum.each(removed, &Repo.delete/1)`
### R036 · P2/bug · `lib/eden/chat.ex:3443` — `soft_delete`: `Enum.each(attachments, &Repo.delete/1)`

- **Проблема:** Оба удаляют строки вложений поштучно через `Repo.delete/1` по структуре. `Repo.delete` шлёт `DELETE WHERE id=?` и ждёт 1 строку; если строку уже удалил конкурентный второй вызов (два устройства автора: edit+delete или delete+delete) — затронуто 0 строк → `Ecto.StaleEntryError` (raise) внутри `Repo.transact` → краш LiveView.
- **Доказательство в коде:** `apply_media_edit` (`chat.ex:2616-2617`):
  ```elixir
  Repo.transact(fn ->
    Enum.each(removed, &Repo.delete/1)
  ```
  `soft_delete` (`chat.ex:3440-3443`):
  ```elixir
  Repo.transact(fn ->
    message = Repo.preload(message, :attachments)
    orphan_keys = unshared_blob_keys(message.attachments)
    Enum.each(message.attachments, &Repo.delete/1)
  ```
  `orphan_keys`/`unshared_blob_keys` считаются **до** транзакции, `delete_unreferenced_blobs` перепроверяет ссылки пост-коммитно (`chat.ex:2641`, `3675`), поэтому переход на `delete_all` (0 строк = чистый no-op) безопасен для логики позиций/блобов. Прецедент — `do_toggle_reaction` (`chat.ex:3446` использует `Repo.delete_all` для реакций именно от StaleEntryError) и тест `chat_test.exs:3606-3614` «delete_all must not raise (Repo.delete would)». `delete_message_for_both` (`2665-2682`) НЕ содержит `ensure_not_deleted` в `with`-цепочке — второй вызов проходит в `soft_delete`.
- **Влияние:** delete-for-both / media-edit — частые операции с двух устройств; конкурентный дубль роняет процесс. R007 подтверждён как medium (та же уязвимость, что R036), приоритет поднят до P1 в связке.

### R130 · P3/bug · `lib/eden/chat.ex:2546` — `edit_message`: `{:ok, edited} = Repo.update()` без stale-guard

- **Проблема:** Успешный путь редактирования текста делает жёсткий паттерн-матч на `{:ok, _}`. При конкурентном hard-delete строки (GC комнаты между `fetch_message` и update) `Repo.update` поднимет `Ecto.StaleEntryError`; при любом `{:error, _}` — `MatchError`.
- **Доказательство в коде:** `chat.ex:2546-2547` (status=advisory, confidence=medium — не проверялось верификатором, но код подтверждает):
  ```elixir
  {:ok, edited} =
    changeset |> Ecto.Changeset.put_change(:edited_at, now()) |> Repo.update()
  ```
  Прецедент — `resolve_join_request` (`chat.ex:1108`) и `update_room` (`chat.ex:975`) уже используют `Repo.update(stale_error_field: :id)` с маппингом в `{:error, :not_found}`.
- **Влияние:** Краш LiveView в узкой гонке редактирования против GC. P3 в исходной находке; включён сюда как одно-строчный однотипный фикс.

### R028 · P2/bug · `lib/eden/chat.ex:410-418` — write-skew в `leave_and_maybe_gc`: беседа без GC навсегда

- **Проблема:** Транзакция обновляет только СВОЮ membership-строку и делает обычный `SELECT` в `all_members_left?/1` (без `FOR UPDATE`). При двух конкурентных выходах последних участников в READ COMMITTED каждая транзакция видит незакоммиченный `left_at` другой как `nil` → обе решают «кто-то остался» → GC не запускается ни у кого.
- **Доказательство в коде:** `chat.ex:414-418`:
  ```elixir
  defp all_members_left?(conversation_id) do
    not Repo.exists?(
      from m in Membership, where: m.conversation_id == ^conversation_id and is_nil(m.left_at)
    )
  end
  ```
  Ни `lock: "FOR UPDATE"`, ни `isolation_level` (Repo без override → READ COMMITTED). Re-триггера нет: осиротевшая беседа невидима обоим, `resurface_direct` (`chat.ex:4009`) срабатывает лишь на новое сообщение, а его некому послать. `FOR UPDATE` — конвенция репо (`accounts.ex:99/298/845`, `channels.ex:976`).
- **Влияние:** Строки conversation/messages/attachments + блобы утекают навсегда. Узкое окно (два ПОСЛЕДНИХ участника выходят почти одновременно), но необратимая утечка. confidence=medium.

### R037 · P2/bug · `lib/eden/chat.ex:3926-3931` — гонка `find_or_create_direct`: два дубликата 1:1

- **Проблема:** Классический check-then-insert без блокировки и без уникального ограничения БД на пару 1:1. Два одновременных «написать» получают `nil` из `existing_direct/2` и оба вставляют новую беседу.
- **Доказательство в коде:** `chat.ex:3926-3931`:
  ```elixir
  defp find_or_create_direct(creator, other_id) do
    case existing_direct(creator.id, other_id) do
      nil -> insert_conversation(creator, [other_id], %{is_group: false})
      conversation -> {:ok, Repo.preload(conversation, memberships: :user)}
  ```

  Единственный релевантный unique-индекс — `unique_index(:memberships, [:conversation_id, :user_id])` (миграция `20260606130148`) — защищает от дублей membership ВНУТРИ беседы, но не от двух 1:1 на пару. `direct_key`/пары-констрейнта нет.
- **Влияние:** Два параллельных 1:1 на одну пару, история расщепляется, `existing_direct` с `limit: 1` без `order_by` дальше выбирает недетерминированно. Чинится только ручным слиянием. confidence=medium. **Требует продуктового решения по механике фикса** (см. Заметки) — этот пункт единственный, который не является механическим.

### R035 · P2/bug · `lib/eden/chat.ex:3027-3031` — гонка delete-for-both рута против `create_reply` → tombstone-рут с живыми реплаями

## План реализации

- [ ] В `describe "concurrent-delete hardening (#258)"` (`chat_test.exs:1684`) — для каждого пути: удалить строку из-под операции напрямую (`Repo.delete_all`) и заассертить **тегированную** ошибку, а не raise:

  - [ ] `edit_message` по hard-deleted строке → `{:error, :not_found}` (R130).

## Критерии приёмки

- Все перечисленные пути возвращают тегированную ошибку (`{:error, :not_found | :has_replies | ...}`) вместо `Ecto.ConstraintError`/`StaleEntryError`/`ArgumentError`/`MatchError` при конкурентном удалении.
- GC двух последних участников выполняется ровно один раз (нет утечки строк/блобов); одна пара 1:1 = одна беседа.
- `mix check` зелёный (format, `compile --warnings-as-errors`, credo --strict, sobelow, deps.audit, test); `mix dialyzer` зелёный.
- Миграция (если выбран `direct_key`-путь) обратима; нет регрессий в существующих тестах send/edit/delete/folder/thread/GC.

## Заметки

- **Продуктовая/архитектурная развилка (единственный не-механический пункт) — фикс 1:1-гонки (R037):** `direct_key` unique-column (нужна миграция + backfill существующих 1:1, риск конфликта на уже-задвоенных парах, если такие есть в проде) против `pg_advisory_xact_lock` на упорядоченную пару id (без миграции, но лочит find-or-create глобально по паре). Требует решения человека — остальной issue готов к агенту. Если 1:1-фикс выделить в отдельный issue, R037 переносится туда, а этот остаётся полностью `ready-for-agent`.
- **Приоритеты уточнены при верификации:** R007/R008 подняты P2→P1 (необработанное исключение в hot-path send/edit + узкое окно потери данных). R035/R134 — реальный нарушенный инвариант, но мягкая деградация без краха; R134 confidence=low. R107/R130/R134 — status=advisory (verdict_note пуст), подтверждены чтением кода, но не проходили независимую верификацию; включены для трассируемости.
- **Все находки подтвердились при чтении кода.** Номера строк совпали с входными (R008 — точнее анкор `message.ex:95` в `changeset/2`; R035 бамп — `chat.ex:3028` внутри `deliver_reply`; R036 soft_delete — `chat.ex:3443`).
- `upsert_thread_membership` `Repo.insert!` (`chat.ex:3084`) в `track_reply` уже прикрыт 0-row-гвардом `bumped == 1` в `deliver_reply` (`chat.ex:3035`) — отдельного фикса не требует (комментарий на `3020-3025` это документирует). Условный бамп (шаг R035) сохраняет этот гвард.
- Связь: продолжение эпика #258 (концентрированного как «hardening»); закрывает остаточные пути того же класса. `find_or_create_direct` и GC-write-skew — родственные, но новые (не часть #258).
