# Не оставлять канал осиротевшим при удалении/деактивации его единственного owner-а

**Приоритет:** P1 — жизненный цикл аккаунта оставляет канал в неустранимом состоянии (нельзя ни удалить, ни передать владение, ни повысить участника), терминальная операция удаления/офбординг застревает без пути восстановления — корректностный дефект ядрового потока (краха/потери данных нет, поэтому не P0; граница мягкая, но отсутствие любого override и помеха офбордингу удерживают P1).
**Лейблы:** `bug` `P1: critical` `backend`
**Находки ревью:** R003, R004, R117

## Контекст
Подсистема — жизненный цикл аккаунта (`Eden.Accounts`) на стыке с корпоративным слоем каналов (`Eden.Channels`). Права канала целиком строятся на per-user членстве с ролью `owner`: удалить канал, передать владение или повысить участника может только сам owner, и у каждого канала ровно один owner (`transfer_ownership` демотит старого в admin). При этом два терминальных потока аккаунта — необратимое удаление (`delete_user_permanently/2`, #303) и деактивация (`deactivate_user/2`, #251) — анонимизируют/блокируют пользователя, **но не трогают его `channel_memberships` и не снимают/переназначают роль owner**. R003 и R004 — это два ракурса одного корня (осиротевший owner канала); R117 (лимит каналов по `creator_id`, а не по владению) — смежная семантическая мелочь из той же зоны `channels`, приложенная для трассируемости.

## Находки и доказательства

### R003 · P1/product · `lib/eden/accounts.ex:262` — удаление/деактивация owner-а не передаёт владение
- **Проблема:** `delete_user_permanently/2` (анонимизация, строка сохраняется) и `deactivate_user/2` (`active=false`) не касаются членства в каналах и роли owner; веб-слой при удалении вызывает только `Chat.scrub_deleted_user_content/1` + `Channels.revoke_invites_by/1`, при деактивации — только `revoke_invites_by`. Ownership не передаётся.
- **Доказательство в коде:** `anonymize_changeset/2` меняет только поля самой `users`-строки — ни одного упоминания каналов:
  ```elixir
  # lib/eden/accounts.ex:329
  defp anonymize_changeset(%User{} = user, dead_hash) do
    Ecto.Changeset.change(user,
      username: "deleted-#{user.id}",
      display_name: @deleted_display_name,
      # ... role: "member", active: false, deleted_at: now(), totp_* нулятся ...
    )
  end
  ```
  Деактивация так же локальна — `deactivate_writes/1` (`lib/eden/accounts.ex:176`) только `set_active(false)` + удаление токенов/инвайтов. FK `channel_memberships.user_id` объявлен `on_delete: :delete_all` (`priv/repo/migrations/20260610151652_create_channels.exs:21`), но удаление здесь — **анонимизация, не `DELETE` строки**, поэтому каскад НЕ срабатывает и мёртвая owner-строка реально остаётся.
- **Влияние:** если удалённый/деактивированный пользователь был единственным owner-ом канала, канал становится неуправляемым: его нельзя удалить и нельзя передать владение (см. R004). Неустранимое состояние для терминальной операции; для деактивации — помеха офбордингу владельца команды. **Verified**, confidence medium (край P1/P2 — канал остаётся рабочим).

### R004 · P1/bug · `lib/eden/channels.ex:691` — канал застревает без живого владельца, без платформенного override
- **Проблема:** все административные пути канала требуют, чтобы **действующим лицом** был owner, и запрещают трогать owner-строку; платформенного super_admin-override в `Channels` нет вовсе (grep по `channels.ex`: `super_admin` не встречается). Мёртвый owner неустраним изнутри канала.
- **Доказательство в коде:** удаление и передача владения — owner-only:
  ```elixir
  # lib/eden/channels.ex:305  delete_channel — owner-only
  :ok <- ensure_role(channel.role, ~w(owner)) do
  # lib/eden/channels.ex:760  transfer_ownership — actor должен быть owner
  :ok <- ensure_role(channel.role, ~w(owner)),
  # lib/eden/channels.ex:733  set_member_role — actor должен быть owner
  :ok <- ensure_role(channel.role, ~w(owner)),
  ```
  Никто не может удалить owner-строку, а owner не может уйти:
  ```elixir
  # lib/eden/channels.ex:690
  defp ensure_removable("owner", target) when target in ~w(admin member), do: :ok
  defp ensure_removable("admin", "member"), do: :ok
  defp ensure_removable(_actor, _target), do: {:error, :forbidden}
  # lib/eden/channels.ex:701  leave_channel — owner не может выйти
  if channel.role == "owner", do: {:error, :owner}, else: ...
  ```
  Дополнительно `list_members/2` (`lib/eden/channels.ex:591`) НЕ фильтрует удалённых — джойнит `User` без `is_nil(deleted_at)`, поэтому в списке участников владельцем висит «Удалённый аккаунт».
- **Влияние:** канал в мёртвом неуправляемом состоянии — ни transfer, ни delete, ни promote; ни channel-admin, ни платформенный super_admin ничего не могут. Каналом можно пользоваться (чат в комнатах жив, admin'ы делают room-CRUD), но администрирование и офбординг владельца заблокированы навсегда. **Verified**, confidence medium.

### R117 · P3/product · `lib/eden/channels.ex:47` — лимит каналов считается по `creator_id`, а не по владению
- **Проблема:** кап `@max_channels` (20) проверяется по числу СОЗДАННЫХ каналов (`creator_id`), который иммутабелен, тогда как владельцем канала может стать другой человек через `transfer_ownership`.
- **Доказательство в коде:**
  ```elixir
  # lib/eden/channels.ex:46
  def create_channel(%Scope{user: user}, attrs) do
    created = Repo.aggregate(from(c in Channel, where: c.creator_id == ^user.id), :count)
    if created >= @max_channels do
      {:error, :limit}
  ```
  `creator_id` задаётся только при вставке (`%Channel{creator_id: user.id}`, строка 66) и никогда не меняется; `transfer_ownership` двигает лишь роль в `channel_memberships`, не `creator_id`.
- **Влияние:** (а) создавший 20 и раздавший все каналы всё равно заблокирован (счётчик = 20, хотя не владеет ничем); (б) получивший 20 по передаче может создать ещё 20 своих. Мелкий контринтуитивный нюанс. **status=advisory, confidence=low** — не верифицировано независимо (`verdict_note` пуст); это продуктовая развилка семантики капа, а не баг фикса из R003/R004.

## План реализации
- [ ] Решить продуктовую политику для осиротевшего owner-а (авто-передача vs явный запрет) — см. «Заметки». Реализация ниже исходит из **авто-передачи с fallback**, как предложено в находках.
- [ ] Добавить в `Eden.Channels` публичный API реассигнации владения на стыке жизненного цикла аккаунта, напр. `reassign_orphaned_ownerships(user_id)` (без `%Scope{}` — это системная операция офбординга, не действие актора; поэтому она обходит owner-only гварды намеренно и внутри контекста Channels, а не reach-in). Для каждого канала, где `user_id` — owner: выбрать преемника детерминированно (старший admin по `list_members`-порядку owner→admin→member→display_name; при отсутствии admin — старейший участник), сделать `update_role(преемник → "owner")` и `update_role(старый → "member")` в одной транзакции; если преемника нет вовсе (owner — единственный участник), удалить канал через внутренний путь (переиспользовать логику `delete_channel` без owner-check: reclaim blob-ов + broadcast `{:channel_deleted, id}`). Каждой затронутой команде разослать `{:members_changed, id}` / `:channels_changed`.
- [ ] Вызвать новый API из веб-слоя на обоих терминальных путях — там же, где уже оркестрируется кросс-контекстная очистка (контексты не reach-in друг в друга):
  - `lib/eden_web/live/admin_live.ex:196-206` (`delete_account`) — рядом с `Chat.scrub_deleted_user_content/1` + `Channels.revoke_invites_by/1`.
  - `lib/eden_web/live/admin_live.ex:158-165` (`deactivate`) — рядом с `Channels.revoke_invites_by/1`.
- [ ] Отфильтровать удалённых из `list_members/2` (`lib/eden/channels.ex:591`) — добавить `where: is_nil(u.deleted_at)` в джойн, чтобы «Удалённый аккаунт» не висел в списке участников (R004). Свериться, что это не ломает атрибуцию сообщений в комнатах (там имя берётся из другого пути) — фильтр только для панели участников.
- [ ] (R117, опционально, после решения человека) Либо задокументировать в moduledoc/`max_channels`, что кап — на число СОЗДАННЫХ каналов (анти-флуд создания), либо сменить семантику на подсчёт по членству-владению (`role == "owner"`). Не блокирует R003/R004.

## Критерии приёмки
- Удаление или деактивация единственного owner-а канала НИКОГДА не оставляет канал с мёртвым owner-ом: владение либо передано живому участнику, либо канал корректно удалён (blob-ы reclaimed).
- Ни один путь `Channels` больше не возвращает канал в неустранимое состояние после офбординга владельца (transfer/delete/promote снова достижимы через нового owner-а).
- `list_members/2` не показывает анонимизированные аккаунты.
- Реассигнация — внутри контекста `Channels`, оркеструется веб-слоем (нет cross-context reach-in из `Accounts` в `Channels`).
- `mix check` и `mix dialyzer` зелёные; существующие тесты каналов/аккаунтов не регрессируют.

## Заметки
- **Развилка для человека (R003/R004):** авто-передача владения vs явный запрет удаления/деактивации последнего owner-а с понятным сообщением. Авто-передача не блокирует офбординг (плюс для #251/#303), но негласно назначает владельцем человека, который об этом не просил; запрет — предсказуемее, но заставляет админа сначала вручную передать владение во всех затронутых каналах. План выше исходит из авто-передачи как основного варианта из находок; финальное решение — за владельцем продукта.
- **Порядок:** сначала API в `Channels` + фильтр `list_members`, потом врезка вызовов в `admin_live.ex` (чтобы веб-слой звал уже готовую функцию).
- **R117 — advisory/не верифицировано** (`verdict_note` пуст, confidence=low): это семантическая развилка капа, а не часть фикса осиротевшего owner-а; вынесена в отдельный опциональный шаг и может уехать в самостоятельный `enhancement`-issue, если владелец продукта решит менять семантику. При чтении кода подтверждается фактологически (`creator_id` иммутабелен, лимит по нему), но «проблема» ли это — вопрос продуктового замысла.
- Все три находки verified/advisory сверены с актуальным кодом; номера строк во входе совпали с текущими (accounts.ex:262/176/329, channels.ex:47/307/691/701/733/760, admin_live.ex:158/196).
