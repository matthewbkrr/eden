# Довести раздачу файлов до готовности к S3/R2: Range на remote-пути, атомарная запись Local, гигиена заголовков ошибок и покрытие тестами

**Приоритет:** P2 — все находки о поддерживаемости, краевых случаях и производительности S3-пути (задокументированный escape hatch), а не активного прода на Local-диске; краха/потери данных/нарушения корректности на текущем адаптере нет.
**Лейблы:** `enhancement` `P2: medium` `backend` `ready-for-agent`
**Находки ревью:** R045, R046, R159, R160, R167, R168, R169, R170, R171

## Контекст

Подсистема — раздача загруженных вложений и аватаров: `EdenWeb.FileController` (оригиналы + тамбнейлы вложений, с Range/206), три аватар-контроллера (`AvatarController`, `ChannelAvatarController`, `GroupAvatarController`) и facade `Eden.Storage` над адаптерами `Local` (dev/текущий прод) и `S3` (`config/runtime.exs` — «одна строка конфига», задокументированный переезд на R2/S3). Общий корень большинства находок один: **весь remote-путь раздачи (`send_remote/send_remote_bytes`) не рассчитан на S3 и не покрыт тестами** — он читает объект целиком в память и режет Range из памяти, дублируя local-ветку без её гарантий (sendfile) и без тестов. Плюс несколько мелочей гигиены заголовков ошибок и один пробел покрытия аватар-контроллера. Сведены в один issue, потому что все относятся к одной поверхности `Storage` + раздача и правятся вместе.

## Находки и доказательства

### R045 · P2/perf · `lib/eden_web/controllers/file_controller.ex:90` — S3-раздача читает весь объект в память на каждый Range-запрос
- **Проблема:** для remote-адаптера `send_object` уходит в `send_remote`, который читает ВЕСЬ объект (до ~50 МБ видео) в память и режет `binary_part` под каждый Range. Каждый seek в `<video>` = новый Range-запрос браузера = полный GET из S3 + буфер в памяти BEAM. Вдобавок `deliver` делает `Storage.exists?` (подписанный HEAD на S3) перед каждой раздачей — лишний round-trip к HEAD поверх GET.
- **Доказательство в коде:**
  ```elixir
  # file_controller.ex:54-59
  defp send_object(conn, key) do
    case Storage.local_path(key) do
      {:ok, path} -> send_local(conn, path)   # sendfile + офсеты — ок
      :error -> send_remote(conn, key)        # S3-путь
    end
  end

  # file_controller.ex:90-95, 98-113
  defp send_remote(conn, key) do
    case Storage.read(key) do
      {:ok, bytes} -> send_remote_bytes(conn, bytes)   # весь объект в памяти
      {:error, _} -> not_found(conn)
    end
  end
  defp send_remote_bytes(conn, bytes) do
    total = byte_size(bytes)
    case parse_range(get_req_header(conn, "range"), total) do
      {:ok, first, last} ->
        conn
        |> put_resp_header("content-range", "bytes #{first}-#{last}/#{total}")
        |> send_resp(206, binary_part(bytes, first, last - first + 1))  # режем из памяти
      ...
  ```
  `Eden.Storage.S3.read/1` (s3.ex:52-58) делает GET **без Range** и возвращает весь `body`; moduledoc (s3.ex:30-33) — до ~50 МБ. Behaviour `Eden.Storage` (storage.ex:10-22) не имеет `read_range/2`. `deliver` (file_controller.ex:38) зовёт `Storage.exists?` → `Eden.Storage.S3.exists?/1` = подписанный HEAD (s3.ex:71). Критично: `Eden.Storage.SigV4.authorization/5` подписывает **только** переданные ей заголовки (sigv4.ex:28: `signed = headers |> Map.keys()`), значит `Range` можно добавить в Req-запрос **вне** SignedHeaders — S3/R2/MinIO принимают неподписанный `Range` на GET и отдают 206. Фикс реализуем.
- **Влияние:** после переезда медиа на R2/S3 видео-перемотка станет мультисекундной и раздует память процесса (N параллельных зрителей × ≤50 МБ); двойной HEAD+GET удваивает латентность и стоимость каждой раздачи. На активном Local-проде проблемы нет.

### R168 · P3/test-gap · `lib/eden_web/controllers/file_controller.ex:98` — remote-путь раздачи не покрыт тестами
- **Проблема:** вся Range-логика remote-ветки (нарезка `binary_part`, 206/416, suffix-range из памяти) дублирует local-ветку, но **все** тесты `FileController` гоняются на Local-адаптере (`config/config.exs:27` — `adapter: Eden.Storage.Local`). Нет ни одного теста, где `Storage.local_path → :error` (S3-путь контроллера).
- **Доказательство в коде:** `file_controller_test.exs:92-141` покрывает Range только через Local (`create_attachment_message` → блоб на диске → `send_local`). Ветки `send_remote`/`send_remote_bytes` (file_controller.ex:90-113) не исполняются ни разу. Off-by-one в `binary_part` или расхождение заголовков с local-веткой ничем не поймать.
- **Влияние:** при включении S3 в проде раздача пойдёт по непроверенному коду. (Тесно связано с R045 — фикс R045 меняет именно этот код, поэтому тесты писать вместе.)

### R159 · P3/quality · `lib/eden/storage.ex:34` — `function_exported?` без `Code.ensure_loaded?`
- **Проблема:** `local_path/1` facade проверяет `function_exported?(adapter, :local_path, 1)`, которая возвращает `false` для ещё **не загруженного** модуля. В dev/интерактиве первый вызов после рестарта (F5 чата с существующими картинками до первого `put`) может увидеть `false` и уйти в `send_remote` — чтение в память вместо sendfile.
- **Доказательство в коде:**
  ```elixir
  # storage.ex:31-35
  def local_path(key) do
    adapter = adapter()
    if function_exported?(adapter, :local_path, 1), do: adapter.local_path(key), else: :error
  end
  ```
  Классическая ловушка `function_exported?`: `false` до загрузки BEAM-файла модуля. confidence=medium, status=advisory. В release (embedded) всё предзагружено — эффект dev-only и самоисправляющийся (байты корректны), но контракт facade «Local ⇒ `{:ok, path}`» тихо нарушается в зависимости от порядка загрузки.
- **Влияние:** редкая dev-only деградация до in-memory чтения; байты верны, диагностики нет.

### R169 · P3/quality · `lib/eden_web/controllers/file_controller.ex:120` — Range-парсер принимает только строчное `bytes=`
- **Проблема:** regex `~r/^bytes=(\d*)-(\d*)$/` регистрозависим, а по RFC 9110 range-unit регистронезависим (`Bytes=0-99` — валиден). Такой заголовок уйдёт в `:none` → 200 с полным телом.
- **Доказательство в коде:**
  ```elixir
  # file_controller.ex:119-124
  defp parse_range([value | _], total) do
    case Regex.run(~r/^bytes=(\d*)-(\d*)$/, value) do
      [_, first, last] -> resolve_range(first, last, total)
      _ -> :none
    end
  end
  ```
  confidence=low, status=advisory. Все основные браузеры шлют строчное `bytes`, отдать 200 вместо 206 легально — практический эффект ~нулевой; нестандартные клиенты/прокси получат полный объект вместо диапазона.
- **Влияние:** нестандартный клиент качает весь 50-МБ объект вместо запрошенного диапазона.

### R167 · P3/quality · `lib/eden_web/controllers/file_controller.ex:82` — 416 не очищает `cache-control: immutable`
- **Проблема:** ветки `:unsatisfiable` (строки 82 и 108) отвечают 416, **сохраняя** выставленный в `deliver` заголовок `cache-control: private, max-age=31536000, immutable`. Для 404 очистка есть (`not_found/1`, строка 148), для 416 — нет.
- **Доказательство в коде:**
  ```elixir
  # file_controller.ex:81-82 (и зеркально 107-108 в send_remote_bytes)
  :unsatisfiable ->
    conn |> put_resp_header("content-range", "bytes */#{total}") |> send_resp(416, "")
  ```
  vs. `not_found/1` (file_controller.ex:146-151) с `delete_resp_header("cache-control")` и комментарием «never cached as immutable for a year». confidence=medium, status=advisory. Практический вред низкий (браузеры редко кэшируют 416), но собственное правило проекта «ошибки не кэшируем immutable» нарушено.
- **Влияние:** ошибочный 416-ответ теоретически кэшируется браузером на год по тому же URL.

### R170 · P3/ux · `lib/eden_web/controllers/file_controller.ex:146` — 404 после установленных заголовков наследует `content-disposition: attachment`
- **Проблема:** `deliver` ставит `content-type`, `content-disposition`, `accept-ranges` **до** `send_object`; если блоб исчез между `Storage.exists?` и `File.stat`/S3-`read`, `not_found/1` чистит только `cache-control`. `Phoenix.Controller.text/2` (`ensure_resp_content_type`) НЕ перетирает уже установленный content-type, так что 404 уходит с content-type исходника и `content-disposition: attachment; filename="…"`.
- **Доказательство в коде:**
  ```elixir
  # file_controller.ex:37-49 — заголовки ставятся ДО send_object
  defp deliver(conn, key, content_type, disposition) do
    if Storage.exists?(key) do
      conn
      |> put_resp_content_type(content_type, nil)
      |> put_resp_header("content-disposition", disposition)
      |> put_resp_header("accept-ranges", "bytes")
      |> ...
      |> send_object(key)                # send_local/send_remote могут вернуть not_found
    ...

  # file_controller.ex:146-151 — чистит ТОЛЬКО cache-control
  defp not_found(conn) do
    conn
    |> delete_resp_header("cache-control")
    |> put_status(:not_found)
    |> text("Not found")
  end
  ```
  `send_local` при `File.stat → {:error, _}` (file_controller.ex:69) и `send_remote` при `{:error, _}` (file_controller.ex:93) зовут `not_found/1` уже после установки заголовков. confidence=medium, status=advisory.
- **Влияние:** браузер, получив 404 с `disposition: attachment`, скачает файл с оригинальным именем и содержимым «Not found» вместо показа ошибки — сбивает с толку при гонке удаления блоба или флейке S3.

### R171 · P3/ux · `lib/eden_web/controllers/file_controller.ex:173` — inline-медиа отдаются без `filename`
- **Проблема:** `disposition/1` для image/video/audio — голое `"inline"` без параметра `filename`, хотя санитизированное `attachment.filename` в БД есть. «Сохранить как…» подставит имя из URL (`123`, `thumb`).
- **Доказательство в коде:**
  ```elixir
  # file_controller.ex:161-173
  defp disposition(%{kind: "file", filename: name}) when is_binary(name) do
    fallback = String.replace(name, ~r/[^\x20-\x7e]|["\\;]/, "_")
    encoded = URI.encode(name, &URI.char_unreserved?/1)
    ~s(attachment; filename="#{fallback}"; filename*=UTF-8''#{encoded})
  end
  defp disposition(%{kind: "file"}), do: "attachment"
  defp disposition(_attachment), do: "inline"    # ← media: имя не пробрасывается
  ```
  confidence=medium, status=advisory. Логика построения `filename="…"; filename*=…` уже есть в file-ветке — её надо вынести в общий helper и применить к inline-медиа.
- **Влияние:** сохранённые из чата фото/видео получают бессмысленные имена, в отличие от Telegram/Mattermost.

### R160 · P3/quality · `lib/eden/storage/local.ex:24` — Local-адаптер пишет неатомарно (без temp+rename)
- **Проблема:** `put/2` (`File.cp`) и `put_binary/2` (`File.write`) пишут напрямую в целевой путь. Краш VM/диска посреди записи оставит обрезанный блоб под финальным ключом; `exists?` вернёт `true`, раздача отдаст обрезанные байты без диагностики.
- **Доказательство в коде:**
  ```elixir
  # local.ex:14-30
  def put(key, source_path) do
    dest = path(key)
    with :ok <- File.mkdir_p(Path.dirname(dest)) do
      File.cp(source_path, dest)          # прямо в финальный путь
    end
  end
  def put_binary(key, binary) do
    dest = path(key)
    with :ok <- File.mkdir_p(Path.dirname(dest)) do
      File.write(dest, binary)            # прямо в финальный путь
    end
  end
  ```
  confidence=medium, status=advisory. Гонок чтения нет (ключ случайный, публикуется в БД только после успешной записи), но повреждённый файл может пережить рестарт.
- **Влияние:** частичная запись неотличима от целой; редкий, но молчаливый источник битых блобов.

### R046 · P2/test-gap · `lib/eden_web/controllers/group_avatar_controller.ex:14` — `GroupAvatarController` не покрыт controller-тестами
- **Проблема:** для `AvatarController` и `ChannelAvatarController` есть HTTP-тесты (`avatar_controller_test.exs`, `channel_avatar_controller_test.exs`), а для `GroupAvatarController` — ни одного (grep по `test/` находит только контекстный `group_avatar_key` в `chat_test.exs:207`).
- **Доказательство в коде:** `group_avatar_controller.ex:14-25` реализует ровно ту же авторизационную поверхность, что и `channel_avatar_controller.ex:14-27` (404 для не-члена без утечки существования, 404 без аватара, `image/jpeg` + `nosniff`, immutable cache-control) — но без HTTP-теста. confidence=high, status=advisory.
- **Влияние:** регрессия на этой авторизационной поверхности (например, при рефакторинге пайплайна раздачи) пройдёт незамеченной.

## Критерии приёмки

- На S3/R2-адаптере видео-seek выполняет один Range-GET нужного диапазона (не полный GET), без предшествующего HEAD; раздача не буферизует весь объект в памяти под Range-запрос. [R045]
- Remote-путь раздачи покрыт тестами (200/206/suffix/416/404), гоняющимися на S3-stub или тест-адаптере без `local_path/1`. [R168]
- Ответы 404 и 416 не несут `cache-control: immutable`, `content-disposition: attachment` и отдаются как `text/plain`. [R167, R170]
- Inline-фото/видео скачиваются с осмысленным именем из БД. [R171]
- Local-адаптер пишет атомарно (temp+rename): нет промежуточного состояния, где `exists? == true`, но байты обрезаны. [R160]
- `Storage.local_path/1` (и новый `read_range/2`) корректны независимо от порядка загрузки модулей (`Code.ensure_loaded?`). [R159]

- `mix check` и `mix dialyzer` зелёные; поведение Local-раздачи (текущий прод) не регрессирует — существующие тесты `file_controller_test.exs` проходят без изменений семантики.

## Заметки

- Все находки перепроверены по коду — строки актуальны (line-номера во входе совпадают с текущим файлом). Ни одна не опровергнута.
- **Развилка для теста R168/R045:** глобальная подмена `Eden.Storage` `:adapter` на S3 в `async: true`-тесте небезопасна (гонка с другими тестами, читающими Local). Два варианта, требуют решения: (а) сделать этот describe `async: false` с `Application.put_env` + `on_exit`; (б) написать лёгкий тест-адаптер (Agent/ETS-backed, без `local_path/1`, с `read`/`read_range`) и подменять только на время describe. Вариант (б) чище и не трогает S3-стек — рекомендую его.
- Приоритеты находок в issue сохранены как во входе; наивысший — P2 (R045/R046), поэтому лейбл приоритета `P2: medium`. Тип `enhancement` (нет ни одной находки category=bug — все perf/test-gap/quality/ux).
- Фиксы независимы, но behaviour-изменение (`read_range/2`) и `Code.ensure_loaded?` (R045/R159) должны идти **до** правки `send_remote` и тестов remote-пути — план это отражает.
- Связь: escape hatch S3/R2 задокументирован в `CLAUDE.md` (Storage / Deploy) и `s3.ex:9-22`; переезд медиа на объект-хранилище — предпосылка, при которой R045/R160/R168 из «спящих» становятся реальными. Пока прод на Local — issue уместно закрыть до включения S3.
