# Доставлять уведомление о reply в followed-треде, когда комната открыта, а панель треда закрыта

**Приоритет:** P1 — корректность: молчаливая потеря обещанного уведомления (chime/баннер) о новом ответе в отслеживаемом треде; пользователь не видит реплай (он не попадает в основной стрим) и при этом не получает сигнала.
**Лейблы:** `bug` `P1: critical` `backend` `UI/UX`
**Находки ревью:** R011

## Контекст

Подсистема уведомлений (`EdenWeb.NotifyHook` + гейтинг в `Eden.Chat`). Сервер выбирает получателей и строит locale-нейтральный payload (`Chat.notify_payload/2`), а единственный оставшийся в веб-слое гейт — **фокус**: «смотрю ли я прямо сейчас именно этот чат» (`NotifyHook.focused?/2`). Для reply в треде этот гейт слишком груб: он сравнивает только `selected.id == payload.conversation_id`, а `conversation_id` у reply — это id **комнаты**, тогда как `root_id` (какой именно тред) игнорируется. В итоге, когда пользователь держит комнату открытой, но НЕ открыл панель конкретного треда, за которым следит, фокус ошибочно считается активным и уведомление подавляется — хотя ответа он не видит: реплаи не попадают в основной стрим.

## Находки и доказательства

### R011 · P1/bug · `lib/eden_web/notify_hook.ex:77` — фокус треда путается с открытой комнатой

- **Проблема:** `focused?/2` подавляет уведомление по одному только совпадению id открытой комнаты с `payload.conversation_id`, не учитывая `payload.root_id` и состояние панели треда. Для reply в треде это приводит к ложному «в фокусе».

- **Доказательство в коде.** Гейт фокуса (актуально, notify_hook.ex:77-80):

  ```elixir
  defp focused?(payload, assigns) do
    !!(assigns[:tab_visible] && assigns[:selected] &&
         assigns[:selected].id == payload.conversation_id)
  end
  ```

  Payload reply несёт id комнаты в `conversation_id` и отдельно `root_id` (`Chat.notify_payload/2`, chat.ex:3872-3886):

  ```elixir
  defp notify_payload(%{sender: %User{} = sender} = message, conv) do
    %{
      conversation_id: message.conversation_id,
      message_id: message.id,
      root_id: message.root_id,
      ...
  ```

  Reply доставляется именно фолловерам треда — `recipient_base_query/1` для сообщения с `root_id` фильтрует по `ThreadMembership.following == true` (chat.ex:3802-3813), т.е. получатель осознанно подписан на тред.

  При открытой комнате `select_conversation` ставит `selected: conversation` и `thread_root: nil` (chat_live.ex:12240 и 12256) — панель треда закрыта. Значит `selected.id == payload.conversation_id` истинно, `focused?` → `true`, уведомление подавляется.

  При этом reply НЕ попадает в основной стрим: в `handle_info({:thread_reply, root, reply}, …)` вставка в стрим `:thread` происходит только при `viewing? = ThreadPanel.thread_open_for?(socket, root.id)` (chat_live.ex:2373, 2390-2397); иначе обновляется лишь футер root (счётчик/facepile/unread), но не появляется само сообщение. То есть в закрытой панели ответ буквально не показан.

  Нужный предикат уже есть — `ThreadPanel.thread_open_for?/2` (thread_panel.ex:49-51), но он принимает `socket`, а `focused?/2` оперирует `assigns`, так что проверять надо `assigns[:thread_root]` напрямую:

  ```elixir
  def thread_open_for?(socket, root_id) do
    match?(%{id: ^root_id}, socket.assigns.thread_root)
  end
  ```

  Тестом это не покрыто: в `notify_hook_test.exs` есть кейсы DM на `/settings`, DM на `/app`, room-сообщение на `/settings` и «sidebar-chatter не долетает», но ни одного про reply в треде при открытой комнате (файл целиком, 83 строки).

  Статус находки — **verified**, confidence **medium**; остаточный индикатор (пилюля «N unread» в футере root + бейдж непрочитанных тредов на тулбаре) частично смягчает потерю, но обещанные chime/баннер теряются.

- **Влияние:** Пользователь, который явно подписался на тред (first-class фича) и держит комнату открытой без панели этого треда, не услышит звонок и не увидит баннер о новом ответе — и в самом стриме ответа тоже нет. Тихая потеря уведомления о followed-треде.

## План реализации

- [ ] **Уточнить гейт фокуса для reply в треде** в `EdenWeb.NotifyHook.focused?/2` (notify_hook.ex:77-80). Развести два случая по `payload.root_id`:
  - для payload **без** `root_id` (обычное сообщение) — сохранить текущее поведение: focused, когда `tab_visible` и `selected.id == payload.conversation_id`;
  - для payload **с** `root_id` (reply в треде) — считать focused только когда открыта **именно панель этого треда**: `tab_visible` и `assigns[:thread_root]` присутствует и `assigns[:thread_root].id == payload.root_id` (совпадение `selected.id == conversation_id` для reply само по себе фокусом не считать). Реализовать через две клаузы `focused?/2` с матчем на `%{root_id: root_id}` / `%{root_id: nil}` (или guard `is_nil`), не размазывая условие по `cond`.
  - `thread_root` — ChatLive-only assign (как `selected`/`tab_visible`); на `/settings` и `/admin` его нет, поэтому доступ через `assigns[:thread_root]` даёт `nil` и обе новые ветки корректно вырождаются в «не focused ⇒ доставить».
- [ ] Обновить moduledoc `EdenWeb.NotifyHook` (notify_hook.ex:17-24 — блок про единственный оставшийся гейт «focus»): отметить, что для reply в треде фокус привязан к открытой панели конкретного треда (`thread_root`), а не к открытой комнате.

- [ ] Добавить кейс-антипод (регрессия): когда панель именно этого треда открыта (`thread_root.id == root_id`) — reply НЕ уведомляет (`refute_push_event`).
- [ ] Регрессия для обычного пути: обычное room-сообщение (без `root_id`) при открытой комнате по-прежнему подавляется (focused), а на `/settings` — доставляется (существующие кейсы должны остаться зелёными).

## Критерии приёмки

- Reply в отслеживаемом треде доставляет `notify` (chime/баннер), если панель именно этого треда не открыта — включая случай, когда сама комната открыта.
- Reply НЕ уведомляет, когда открыта панель этого конкретного треда; переключение фокуса корректно для нескольких тредов одной комнаты.
- Обычные сообщения (без `root_id`) сохраняют прежнее поведение фокуса; кейсы на `/settings`/`/admin`/`/app` не регрессируют.
- `mix check` и `mix dialyzer` зелёные; новые тесты покрывают оба направления (доставка при закрытой панели, подавление при открытой).

## Заметки

- Фикс локализован в веб-слое (`NotifyHook`), контекст `Eden.Chat` менять не нужно: payload уже несёт `root_id`, а выбор получателей (фолловеры треда) уже серверный (chat.ex:3802-3813) — теряется только per-session focus-подавление.
- Развилки для человека нет: поведение однозначно (панель треда открыта ⇒ фокус ⇒ подавить; закрыта ⇒ доставить). `ready-for-agent`.
- `ThreadPanel.thread_open_for?/2` принимает `socket`, а не `assigns`, поэтому переиспользовать его в `NotifyHook` напрямую нельзя — используем эквивалентный `match?(%{id: ^root_id}, assigns[:thread_root])` в самом гейте (не выносить общий helper ради одной строки).
