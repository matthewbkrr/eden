# Довести локализацию до конца: auth-флеши, errors.po, оставшиеся UI-строки, единый глоссарий «чат»

**Приоритет:** P2 — это i18n/UX-полировка (сообщения на неверном языке, рассинхрон терминов): формы и авторизация работают корректно, нет краха/потери данных/регрессии, поэтому не P1; но брешь заметна на самых частых экранах RU-аудитории, поэтому не мелочь P3. Наивысший приоритет среди находок — P2.
**Лейблы:** `enhancement` `P2: medium` `UI/UX` `backend` `ready-for-agent`
**Находки ревью:** R099, R100, R101, R204, R206, R207, R208

## Контекст

ihichat — RU-продукт: `EdenWeb.Locale` (lib/eden_web/locale.ex:53-73) негоциирует `ru` из `Accept-Language` (строка 69 — `"ru" <> _ -> "ru"`), а `priv/gettext/ru/LC_MESSAGES/default.po` уже содержит 516 переводов. На этом фоне остались системные бреши локализации: (1) `errors.po` почти пустой — все changeset-ошибки форм показываются по-английски; (2) шесть auth-флешей в `user_auth.ex` вообще не обёрнуты в gettext; (3) один UI-msgid до сих пор без перевода; (4) термин один и тот же объект называется то «чат», то «беседа» и в переводах, и в исходных msgid; (5) пара шероховатых RU-формулировок; (6) на страницах `/welcome/two-factor` и `/reset/:token` NotifyHook смонтирован, но `<.notifier>` не отрендерен — событие уведомления уходит в никуда. Все находки — про одно свойство продукта («единый, отточенный, свой язык интерфейса»), поэтому сведены в один issue.

## Находки и доказательства

### R099 · P2/ux · `lib/eden_web/user_auth.ex:78,120,135,143,154,221` — шесть auth-флешей жёстко на английском

- **Проблема:** Модуль `EdenWeb.UserAuth` не подключает Gettext (`use EdenWeb, :verified_routes` на строке 7 — и всё), а шесть пользовательских флешей заданы сырыми английскими литералами. При RU-локали они всё равно английские.
- **Доказательство в коде:**
  ```elixir
  # user_auth.ex:78  (плаг require_authenticated_user)
  |> put_flash(:error, "You must log in to access this page.")
  # :120  (on_mount :require_authenticated)
  |> Phoenix.LiveView.put_flash(:error, "You must log in to access this page.")
  # :135  (on_mount :require_admin, is_nil(user))
  |> Phoenix.LiveView.put_flash(:error, "You must log in to access this page.")
  # :143  (не-админ)
  |> Phoenix.LiveView.put_flash(:error, "You don't have access to that page.")
  # :152-155  (админ без TOTP)
  |> Phoenix.LiveView.put_flash(:error, "Turn on two-factor authentication to use the admin panel.")
  # :221  (on_sessions_revoked — отзыв сессии)
  |> Phoenix.LiveView.put_flash(:error, "Your session ended. Please sign in again.")
  ```
  Соседние модули делают это правильно: `user_session_controller.ex:20` — `put_flash(:error, gettext("Invalid username or password."))`; `notify_hook.ex:30` — `use Gettext, backend: EdenWeb.Gettext`. `user_auth.ex` — единственный, кто выбивается. Порядок on_mount в каждой live_session ставит `EdenWeb.Locale` ПЕРВЫМ (router.ex:69, 86, 93 — `on_mount: [EdenWeb.Locale, {EdenWeb.UserAuth, ...}, ...]`), значит `Gettext.put_locale` уже вызван к моменту флеша; для плаг-пути `EdenWeb.Locale.call/2` в `:browser`-пайплайне тоже отрабатывает раньше.
- **Влияние:** Русскоязычный пользователь на частых сценариях — редирект на логин, отзыв сессии при деактивации/смене пароля/«выйти везде», форс-2FA для админа, попытка админ-панели без прав — всегда видит английский текст. Ломает единство языка и «trustworthy» позиционирование. Статус находки — advisory (не верифицировано отдельно), но код читается однозначно; confidence high.

### R101 · P2 (снижен с P1)/quality · `priv/gettext/ru/LC_MESSAGES/errors.po:14-114` — errors.po почти не переведён

- **Проблема:** В `errors.po` переведён ровно один msgid; у всех остальных changeset-ошибок `msgstr` пуст. При пустом msgstr Gettext отдаёт исходный английский текст.
- **Доказательство в коде:**
  ```
  errors.po:14  msgid "can't be blank"          msgstr ""      # 15
  errors.po:17  msgid "has already been taken"   msgstr ""      # 18
  errors.po:20  msgid "is invalid"               msgstr ""      # 21
  errors.po:26  msgid "has invalid format"       msgstr ""      # 27
  errors.po:38  msgid "does not match the password"  msgstr "Пароли не совпадают"   # 39 — ЕДИНСТВЕННЫЙ перевод
  errors.po:71  "should be at least %{count} character(s)"  msgstr[0..2] ""   # 73-75
  # …все plural-формы 47-99 и числовые границы 101-114 — пусты
  ```
  Механизм подтверждён верификатором: `core_components.ex:481-495` `translate_error/1` → `Gettext.dngettext(EdenWeb.Gettext, "errors", …)` / `Gettext.dgettext(EdenWeb.Gettext, "errors", …)`; ошибки рендерятся через `ed_field_errors/1` (core_components.ex:614-624). Реальные поверхности: `invite_live.ex` (пустой username/пароль), `settings_live.ex` (bio/username, ~367/451), `reset_live.ex:67`, папки (settings_live ~1598), managed-поля в админке. Тесты `accounts_test.exs` читают сырой changeset через `errors_on`, а не Gettext — так что конфликтующих гвардов нет.
- **Влияние:** Валидационные ошибки — самые частые сообщения на формах; вся RU-аудитория видит их по-английски. Верифицировано (status=verified, confidence high). Приоритет обоснованно снижен P1→P2: корректность валидации не нарушена, это язык сообщения.

### R100 · P2/quality · `priv/gettext/ru/LC_MESSAGES/default.po:2768` — одна оставшаяся непереведённая UI-строка

- **Проблема:** Из трёх строк, отмеченных в находке, ДВЕ уже переведены к моменту чтения (строки сместились): `settings_live.ex:880` «Double-clicking a message reacts…» → default.po:1902-1905 «Двойной клик по сообщению ставит эту реакцию…» и `settings_live.ex:913` «Alerts for new messages…» → default.po:1943-1946 «Оповещения о новых сообщениях…». Осталась ОДНА непереведённая.
- **Доказательство в коде:**
  ```
  # default.po:2767-2768
  msgid "Your admin access was removed."
  msgstr ""
  ```
  Строка используется реально: `admin_live.ex:257` — `put_flash(:error, gettext("Your admin access was removed."))` (флеш при само-демоции админа в открытой сессии, push_navigate → `/settings`). Комментарий-локатор в PO указывает `admin_live.ex:149` — устаревший (реальный вызов на 257), это лишь подтверждает лёгкий template-drift.
- **Влияние:** RU-админ, у которого сняли доступ прямо в открытой вкладке, видит английский флеш. Мелко, но на видном событии. status=advisory, но перепроверено в коде.

### R204 · P3/ux · `lib/eden_web/router.ex:80,98` — WelcomeTotpLive и ResetLive в NotifyHook-сессиях без `<.notifier>`

- **Проблема:** `EdenWeb.NotifyHook` смонтирован на всех трёх live_session (router.ex:69, 86, 93) и на connected mount подписывается + пушит клиентское событие `notify` (notify_hook.ex:57,65 — `attach_hook(:notify, …)` → `push_event(socket, "notify", …)`). Но хост `<.notifier>` рендерят только `settings_live.ex:201`, `admin_live.ex:355`, `chat_live.ex:2867`. `/welcome/two-factor` (WelcomeTotpLive, router.ex:80, live_session `:authenticated`) и `/reset/:token` (ResetLive, router.ex:98, live_session `:default`) — не рендерят.
- **Доказательство в коде:**
  ```
  $ grep -rn "notifier" lib/eden_web/live/welcome_totp_live.ex lib/eden_web/live/reset_live.ex
  (пусто — 0 вхождений)
  ```
  То есть залогиненный пользователь на этих страницах подписан (`Notifications.subscribe`), событие `push_event "notify"` уходит, но JS-хука-приёмника (`<.notifier>` → `EdenWeb.Notifier`) на странице нет — событие теряется.
- **Влияние:** Мелкое нарушение обещания #272 «alerts while a browser tab is open»: на этих экранах сообщение не прозвучит. Практический эффект мал (welcome — краткий онбординг нового юзера, у него ещё нет чатов; reset обычно открыт разлогиненным — тогда `notify_prefs=nil` и подписки нет, notify_hook.ex:40-43). Непоследовательность + впустую подписка/событие для залогиненного случая. status=advisory, confidence high.

### R206 / R208 · P3/quality · `priv/gettext/ru/LC_MESSAGES/default.po` + `lib/eden_web/live/chat_live.ex` — глоссарий «чат» vs «беседа»

- **Проблема:** Один и тот же объект (DM/группа) назван двумя терминами и в переводах, и в исходных msgid.
- **Доказательство в коде:**
  ```
  # default.po — «беседа»:
  :298  msgid "New conversation"        msgstr "Новая беседа"
  :303  msgid "No conversation selected" msgstr "Беседа не выбрана"
  # default.po — «чат» (рядом, тот же UI):
  :231  msgid "Chats"        msgstr "Чаты"
  :634  msgid "No chats yet" msgstr "Пока нет чатов"
  :630  «Удалить этот чат? Он исчезнет из вашего списка.»
  ```
  Дрейф и на уровне ИСХОДНЫХ строк:
  ```elixir
  # chat_live.ex:1133
  gettext("Couldn't delete that chat.")
  # chat_live.ex:2296 и :2316
  gettext("Couldn't start the conversation.")
  ```
  Кнопка «Новая беседа» ведёт в список «Чаты» — рассинхрон в соседних элементах. CONTEXT.md/PRODUCT.md требуют не дрейфовать к синонимам.
- **Влияние:** Непоследовательная терминология повышает когнитивную нагрузку и снижает ощущение «precise, consistent». status=advisory, confidence medium — есть развилка (какой термин выбрать), см. «Заметки».

### R207 · P3/quality · `priv/gettext/ru/LC_MESSAGES/default.po:2728,2582` — калькированные/неестественные формулировки

- **Проблема:** Порядок слов калькирован с английского, звучит неестественно.
- **Доказательство в коде:**
  ```
  # default.po:2727-2728  (welcome_totp_live.ex:108,181 — онбординг 2FA)
  msgid "Skip for now"   msgstr "Пропустить пока"
  # default.po:2581-2582  (admin_live.ex:337)
  msgid "Reactivate account"  msgstr "Реактивировать аккаунт"
  ```
- **Влияние:** «Пропустить пока» на онбординг-экране 2FA (где важно доверие/аккуратность) и «Реактивировать» (канцелярит) — заметные шероховатости копирайта. status=advisory, confidence medium.

## Критерии приёмки

- Под локалью `ru` все шесть auth-флешей, ошибки форм и «Your admin access was removed.» отображаются по-русски; ни одна из этих поверхностей не показывает английский.
- В `errors.po` нет пустых `msgstr` (кроме служебного заголовка), plural-формы заполнены для всех трёх nplurals.
- Термин DM/группы единообразен в переводах и исходных msgid (после решения по глоссарию); .pot/.po синхронны (`mix gettext.extract --merge` без непредвиденного diff).
- `/welcome/two-factor` и `/reset/:token` либо рендерят `<.notifier>`, либо явно исключены комментарием в NotifyHook.
- `mix check` зелёный; регрессий в существующих тестах локали/форм нет.

## Заметки

- **Развилка (R206/R208, требует решения человека):** какой единый термин — «чат» или «беседа». Ревьюер рекомендует «чат» (уже в «Чаты»/«Пока нет чатов» и в исходном `chat_live.ex:1133` "chat"), но это продуктовое/копирайтинговое решение; замена ИСХОДНЫХ msgid (`chat_live.ex:2296,2316`) — видимое изменение английского UI. До решения шаг глоссария не выполнять механически. Из-за этой развилки триаж по строгости — на грани `ready-for-human`; общий триаж оставлен `ready-for-agent`, т.к. остальные 5 шагов (R099/R100/R101/R204/R207) полностью механические, а глоссарный шаг вынесен отдельно и явно ждёт продуктового «ок».
- **R100 частично устарела:** две из трёх строк находки (`settings_live.ex:880`, `settings_live.ex:913`) уже переведены в текущем `default.po` (строки 1902-1905, 1943-1946) — вероятно, дозаполнены после снятия находки. Актуальна только `"Your admin access was removed."` (default.po:2768). Локатор-комментарий в PO для неё указывает `admin_live.ex:149`, реальный вызов — `admin_live.ex:257` (лёгкий template-drift; extract-merge его выправит).
- **R204 нюанс:** для `/reset/:token` разлогиненный кейс безопасен — `notify_hook.ex:40-43` при отсутствии `current_scope.user` не подписывается и ставит `notify_prefs=nil`, так что `:if={@notify_prefs}` не отрендерит хост. Проблема касается только залогиненного пользователя на этих страницах. Поэтому для reset вариант «оставить без notifier + комментарий» вполне допустим; для welcome (страница всегда залогинена) — либо добавить notifier, либо явно задокументировать пропуск.
- **Связь:** R101 верифицирована (единственная verified в связке); R099/R100/R204 — advisory, но перепроверены по коду и подтверждены; R206/R207/R208 — advisory, confidence medium (копирайт/термин).
