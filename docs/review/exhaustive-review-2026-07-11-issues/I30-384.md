# Продуктовые бреши мессенджера: @упоминания/пины/черновики, поиск в админ-панели, деактивированные в пикерах

**Приоритет:** P2 — наивысший приоритет среди находок; всё это поддерживаемость/UX/масштабируемость и продуктовые бреши, а не краши или потеря данных (P0/P1 нет). Часть пунктов — крупные фичи, требующие продуктового решения.
**Лейблы:** `enhancement` `P2: medium` `backend` `UI/UX` `ready-for-human`
**Находки ревью:** R049, R112, R180

## Контекст
Три находки касаются продуктовой полноты пользовательских потоков поверх уже реализованного ядра. Общего технического корня нет, но все три — «то, чего в продукте нет/недоделано» на фоне заявленного Mattermost/Slack-паритета корпоративного слоя и аудитории «built to grow»:
- **R049** — админ-панель `/admin` не масштабируется на сотни аккаунтов (плоский список без поиска).
- **R112** — деактивированные пользователи попадают в пикеры создания диалога/добавления в комнату без пометки (расхождение с админкой, где бейдж «Deactivated» есть).
- **R180** — отсутствуют три ожидаемые фичи: @упоминания, закреплённые сообщения, черновики композера.

R049 и R112 — локализованные фиксы, готовые к агенту; R180 — крупная фича-развилка (нужна приоритезация и, вероятно, отдельные issue). Собраны вместе как единый «продуктовый бэклог/gap-аудит» для трассируемости; при исполнении R180 стоит выделить в отдельный epic.

## Находки и доказательства

### R049 · P2/product · `lib/eden_web/live/admin_live.ex:451` — плоский список людей без поиска/фильтра
- **Проблема:** Секция «People» рендерит `<ul>` по всем `@users` (из `Accounts.list_users()`, `admin_live.ex:42`) без поля поиска, фильтра по роли/статусу или пагинации. Единственный ориентир по объёму — счётчик-заголовок.
- **Доказательство в коде:** список (актуально на `admin_live.ex:451`):
  ```elixir
  <ul class="divide-y" style="border-color: var(--ed-border);">
    <li :for={u <- @users}>
      <button type="button" phx-click="select" phx-value-id={u.id} ...>
  ```
  И счётчик (`admin_live.ex:363`):
  ```elixir
  {ngettext("%{count} person", "%{count} people", length(@users))}
  ```
  Загрузка — целиком в mount (`admin_live.ex:42`): `users: Accounts.list_users()`; `list_users/0` (`accounts.ex:59-61`) отдаёт всех неудалённых без лимита/фильтра. Никакого `handle_event` для фильтрации в модуле нет. Находка `advisory`/`confidence=medium` — это UX-масштабирование, не баг: сейчас список полностью функционален.
- **Влияние:** При росте до сотен аккаунтов админ вынужден скроллить весь список, чтобы найти человека для reset-ссылки/деактивации/выдачи роли. Ключевые административные действия «погребены» на большом объёме — плохо масштабируется.

### R112 · P3/product · `lib/eden/accounts.ex:45` — деактивированные попадают в пикеры без пометки
- **Проблема:** `list_other_users/1` исключает только `deleted_at`, но не `active == false`. Деактивированные (#251) возвращаются как обычные люди в пикеры создания DM/группы и добавления в комнату/канал — без бейджа «Deactivated», хотя в админке такой бейдж есть.
- **Доказательство в коде:** запрос (`accounts.ex:45-52`):
  ```elixir
  def list_other_users(%Scope{user: user}) do
    from(u in User,
      where: u.id != ^user.id and is_nil(u.deleted_at),
      order_by: [asc: u.display_name]
    )
    |> Repo.all()
  end
  ```
  Фильтр по `active` отсутствует. Функция используется в **четырёх** пикерах ChatLive (находка называла два — фикс общий для всех):
  - новый диалог/группа — `chat_live.ex:986` (`toggle_new`);
  - добавить в комнату — `chat_live.ex:1486` (`open_room_add`);
  - добавить в канал — `chat_live.ex:1667` (`open_add_members`);
  - добавить в группу — `chat_live.ex:1697` (`open_group_add_members`).

  Рендер пикера нового диалога (`chat_live.ex:11462-11470`) показывает только имя и `@username`, без флага активности:
  ```elixir
  <.avatar name={u.display_name} src={avatar_src(u)} size={:sm} />
  <span class="flex-1 min-w-0">
    <span class="block" ...>{u.display_name}</span>
    <span class="block" ...>@{u.username}</span>
  </span>
  ```
  Для сравнения — в `admin_live.ex:472-474` бейдж есть:
  ```elixir
  <span :if={!u.active} style="color: var(--ed-danger);">
    · {gettext("Deactivated")}
  </span>
  ```
  Находка `advisory`/`confidence=medium`; данные (`u.active`) уже в структуре, так что оба варианта фикса дёшевы.
- **Влияние:** Коллега пишет в новый DM деактивированному сотруднику и ждёт ответа, которого не будет до (возможно, не наступающей) реактивации; в комнаты/группы добавляются «мёртвые» участники. Поведение к тому же не закреплено тестом.

### R180 · P3/product · `lib/eden_web/live/chat_live.ex:3486` — нет @упоминаний, закреплённых сообщений и черновиков
- **Проблема:** Отсутствуют три ожидаемые для корпоративного мессенджера фичи: (1) @mentions с автодополнением и уведомлениями по упоминанию; (2) закреплённые сообщения (pinned) на комнату/чат; (3) per-conversation drafts композера (сохранение недописанного текста при переключении чата).
- **Доказательство в коде:** грепы по `mention|pinned|pin_message|draft|закреп|черновик` по `lib/eden`/`lib/eden_web` дают только совпадения-омонимы: «pinned» — это исключительно термины скролл-якорения/раскладки (`chat_live.ex:4882` `this.pinned = true`, `shell_components.ex:38` «pinned outside it», `chat.ex:1552` «root is always pinned at the top of the panel»), не фича закрепления сообщений. Схема `Eden.Chat.Message` (`lib/eden/chat/message.ex:19-71`) не содержит `pinned_at`, полей упоминаний или черновиков; миграций с этими понятиями нет. Точка гейтинга уведомлений, которую переиспользует @mentions, существует: `notify_recipient_ids/2` (`chat.ex:3792`). Находка `advisory`/`confidence=medium` — это подтверждённое отсутствие фич (продуктовая брешь), а не баг.
- **Влияние:** Для корпоративного слоя с комнатами и тредами @упоминания и pinned-сообщения — две самые ожидаемые функции (Mattermost/Slack-паритет, на который равняется продукт). Черновики особенно ценны на flaky-сети (кросс-граничный RTT ~160ms), где легко переключить чат, не отправив. Отсутствие снижает воспринимаемую полноту продукта.

## План реализации
Порядок: сперва дешёвые локализованные фиксы (R112, R049), затем крупная фича-развилка (R180 — вероятно, в отдельный epic после продуктового решения).

- [ ] **R112 — определиться и реализовать политику деактивированных в пикерах.** Продуктовая развилка (см. «Заметки»):
  - Вариант A (фильтровать): добавить `and u.active` в `where` `list_other_users/1` (`accounts.ex:45-52`) — симметрично `is_nil(u.deleted_at)`. Правит все 4 колл-сайта разом (`chat_live.ex:986,1486,1667,1697`), кода в шаблонах не трогает.
  - Вариант B (показывать с бейджем): оставить запрос, добавить в рендер пикеров бейдж «Deactivated» по `!u.active`, переиспользовав стиль из `admin_live.ex:472-474`. Затрагивает `new_conversation_modal/1` (`chat_live.ex:11462-11470`) и `add_members_modal/1` (`chat_live.ex:9640+`, `attr :addable`). Данные `u.active` уже есть.
  - Рекомендация: Вариант A (фильтрация) — проще, устраняет «мёртвых» участников из результата целиком и совпадает с уже принятым принципом «deleted/inactive login refused» (#251).
- [ ] **R049 — поиск/фильтр в списке людей `/admin`.** Добавить над `<ul>` (`admin_live.ex:451`) поле поиска по `display_name`/`@username`:
  - Серверный вариант (предпочтителен при масштабе): новый `Accounts.list_users(opts)` или `search_users/1` с `ILIKE`/трайграм-фильтром (переиспользовать паттерн `body_match/1` из Chat — уже есть `gin_trgm_ops`); хранить строку поиска в assign, обновлять `@users` в `handle_event("filter_users", ...)`. Клиентский фильтр допустим только как временный шаг — не масштабируется на сотни строк в DOM.
  - Опционально: тумблеры «только деактивированные» / «только админы» (данные `u.active`/`u.role` уже загружены).
  - Счётчик `admin_live.ex:363` показывать как «N найдено из M».
- [ ] **R180 — приоритезировать и распланировать (крупная фича, вероятно отдельный epic).** Не для механического исполнения; требует продуктового решения по объёму и порядку:
  1. **@mentions** (приоритет для корпоративного слоя): парсинг `@username` в теле, автодополнение по участникам комнаты/чата в композере, подсветка в рендере (переиспользовать `EdenWeb.Markup`), гейтинг уведомлений по упоминанию через `notify_recipient_ids/2` (`chat.ex:3792`) — упомянутый получает уведомление даже при mute (Mattermost-семантика; согласовать с «no badge past mute»-инвариантом).
  2. **Pinned messages** на комнату/чат: новое поле/таблица + admin/owner-гейтинг закрепления, панель закреплённых.
  3. **Drafts**: per-conversation сохранение текста композера при переключении (клиентский `sessionStorage`-хук по образцу `.ForwardCarry`, либо серверно).

- Все три находки — `advisory`/`confidence=medium` (не верифицированы отдельным верификатором), но подтверждены чтением кода: список без фильтра (R049), отсутствие `active`-фильтра/бейджа в пикерах (R112), полное отсутствие полей/фич pin/mention/draft (R180).
