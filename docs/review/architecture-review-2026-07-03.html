eden — архитектурный разбор

Phase 2 многотрекового ревью · 2026-07-03 · метод: deep modules (глубина интерфейсов, размещение швов, deletion test) · зоны: Accounts/auth · Chat · Channels · Storage/media · LiveView-слой · классификация: Strong — уверенный вывод с доказательствами · Worth exploring — правдоподобно, требует обсуждения · Speculative — идея, низкая определённость

TL;DR
  1. Контекстный слой — сильный. Accounts, Chat, Channels, Storage — глубокие модули с чистыми швами: веб-слой ни разу не обращается к Repo напрямую (проверено: 0 вхождений), авторизация через %Scope{} повсеместна, транзакционность инвариантов на месте.
  2. Весь архитектурный долг сконцентрирован в одном файле: chat_live.ex — 13 052 строки, 54% кода приложения, 134 обработчика событий, render на ~5 000 строк, 61 встроенный компонент, 32 JS-хука. Это не «плохой код» — это несколько несепарированных модулей в одном намespace.
  3. Storage — образцовый шов: настоящая двухадаптерная граница (Local + S3), SigV4 изолирован, обходов фасада нет, свап на R2 через одну env-переменную безопасен (все Local-only допущения обработаны фолбэками).
  4. Два системных риска внутри Chat: фильтрация видимости сообщений не полностью централизована (поиск пересобирает фильтры вручную) и PubSub-рассылка размазана по 21+ точкам с двумя параллельными реализациями «кого уведомлять».
  5. Org-слой ляжет без перестройки: все три контекста впитывают org_id расширением Scope; сопротивления форма кода не оказывает.

Findings — Strong

Strong S1. ChatLive — концентрация всего архитектурного долга

chat_live.ex: 134 × handle_event, 33 × handle_info, render ≈ 5 049 строк (2510–7559), 61 компонент, 32 хука; роуты router.ex:57-65

Один LiveView обслуживает шесть URL-форм и два режима рендера. Ветвление @selected.channel_id (flat vs bubble) встречается 13+ раз, включая контракт хука (data-layout, стр. 3266) и целые ветки рендера (3168–3192). Цена уже проявилась: оба недавних P0 (stream-ordering, композер) жили именно здесь; любые параллельные PR сталкиваются в одном файле; каждое изменение перекомпилирует все 13k строк. При этом дисциплина внутри хорошая (см. S2) — проблема именно в несепарированности, а не в качестве кода.

Strong S2. Граница веб → контекст соблюдена безупречно

grep Repo\. / from( по lib/eden_web — 0 вхождений; UI-проверки ролей дублируют, но не заменяют контекстные (chat_live.ex:952 ↔ channels.ex ensure_role)

Веб-слой не строит Ecto-запросов, не переопределяет доменные правила и не вызывает приватные функции контекстов. Проверки прав в UI — только для показа/скрытия кнопок; настоящая авторизация всегда в контексте. Это редкая дисциплина для кодовой базы такого размера.

Strong S3. Storage: настоящий двухадаптерный шов, готовый к R2-свапу

storage.ex:25-44 (фасад), s3.ex + local.ex (оба реализуют behaviour), config/runtime.exs:95-104 (свап одной env), sigv4_test.exs:6-39 (сверка с образцом AWS)

Прямых обращений к адаптерам мимо фасада — ноль. Все четыре потенциально Local-only допущения проверены и закрыты: sendfile → фолбэк на байтовую нарезку (file_controller.ex:91), ffmpeg-у нужен файл → with_local_source/2 скачивает во временный (chat.ex:3974-3991), в БД хранятся только опаковые ключи, размер ограничен капами до чтения в память. Деплойный «аварийный выход» на R2 при нехватке диска — реально одна строка конфига.

Strong S4. Фильтрация видимости не полностью централизована — единственный найденный «протекающий» инвариант

chat.ex:1790-1799 (visible_messages — 4 вызывающих), но поиск (1412-1436) и поиск в тредах (1461-1475) пересобирают те же фильтры вручную

«Пользователь видит сообщение X» — самый дорогой инвариант мессенджера (членство + left_at + delete-for-me + tombstone). Сейчас он живёт в одном билдере запроса, но не все пути через него ходят. Новый message-запрос, забывший один фильтр, = утечка данных. Тесты это сегодня ловят, но структурной защиты нет. Рекомендация: сделать visible_messages (или параметризованный apply_visibility/3) единственным допустимым источником фильтров и провести через него оба поисковых пути.

Strong S5. PubSub-контракт когерентен, но рассылка размазана

14 типов сообщений, 21+ точка broadcast в chat.ex; два параллельных «кого уведомлять»: direct_recipient_ids/1 (3491-3510) и thread_recipient_ids/1 (3515-3528)

Топики именуются последовательно (conversation:id, user:id:chat), но контракт «топик + форма payload» нигде не записан — вызывающие восстанавливают его чтением кода. Фильтрация уведомлений (mute напрямую / через папку / DND / фолловеры треда) реализована дважды с разной логикой. Любой новый тип события требует правок в 2+ местах. Это ровно тот функционал, который ADR-0001 уже планирует вынести в контекст Eden.Notifications — централизацию стоит сделать его первым шагом (Web-адаптер), а не отдельной работой.

Strong S6. Скрытые инварианты интерфейса Chat задокументированы только прозой (или вообще никак)

broadcast как скрытый side-effect create_message (deliver/2, chat.ex:3646); гарантии preload'ов (@message_preloads:40-48) не указаны в @doc; «вышедший из группы может форвардить» (ensure_member:3193-3195) — намеренно, но нигде не записано

Интерфейс — это всё, что должен знать вызывающий, а не только сигнатуры. Три обнаруженных неочевидности безопасны сегодня, но каждая — будущий баг при появлении нового вызывающего (например, Notifications-контекста или REST API). Рекомендация: зафиксировать в moduledoc Chat: side-эффекты broadcast'ов, гарантированные preload'ы, семантику left-членства.

Strong S7. Org-слой впитывается без перестройки

Scope пронизывает 117 функций Chat + 39 Channels; visible_messages — единая точка для org-фильтра; Channels не трогает чужие таблицы (28 вызовов через API Chat)

Проверка «что будет сопротивляться org_id» дала пустой список: точки добавления фильтра механические (existing_direct, unread-агрегаты, поиск), шов уже проложен через Scope. Единственное открытое моделирование — org-уровневые роли (OrgMembership как отдельная таблица, ChannelMembership не трогается). Совпадает с выводом auth-сессии (ADR-0002: глобальный аккаунт + членства).

Strong S8. Сильные стороны, которые стоит беречь как паттерны

accounts.ex:236-282, user.ex:128-137, chat.ex:3220-3287, channels.ex:118-851

  • Токены (сессии, инвайты) — только SHA-256-хеши в БД; сырой токен живёт один показ.
  • Timing-safe вход: Bcrypt.no_user_verify() на несуществующем юзере — перечисление имён невозможно.
  • Валидность инвайта — вычисляемая, не хранимая (expires_at + used_count + revoked_at): рассинхрон исключён конструкцией.
  • Forward-safe GC блобов: форвард переиспользует ключ, delete щадит разделяемые блобы — редкая аккуратность.
  • Классификация вложений по магическим байтам, никогда — по client content-type; двухслойная защита от decompression bomb.
  • Channels: гонки закрыты системно (FOR UPDATE на последнем использовании инвайта, атомарный transfer_ownership, stale_error_field).
  • Извлечённые album_layout.ex и rail_hook.ex — доказательство, что экстракция из ChatLive в этой кодовой базе работает.

Findings — Worth exploring

Worth exploring W1. Первая экстракция из ChatLive: панель тредов

события 1679–1927, рендер 3618–3738, хелперы 11551–12051; ~11 assigns полностью своих

Из трёх кандидатов (треды / мультиселект / композер) у тредов лучшее соотношение выигрыша к риску: состояние самодостаточно (@thread_* не пересекается с главным стримом), события и broadcasts очерчены, рендер уже собран в один компонент. Мультиселект (@selection, @select_surface, SelectSync) — второй кандидат, тоже ортогонален. Композер — самый ценный и самый рискованный (сцеплён с edit/reply/forward), его — третьим, вместе со снятием дублирования send_dispatch ↔ send_reply_dispatch (12304 ↔ 12370 — зеркальные копии, что уже потребовало «зеркального» P0-фикса).

Worth exploring W2. Третий токен-флоу — момент для общего token-модуля

Accounts.Invite ↔ Channels.Invite: параллельные валидации (~50 строк каждая), общие только build_token/hash_token (channels.ex:779,790)

Сегодня дублирование приемлемое (два разных контекста, свои поля). Но ADR-0002 добавляет третий флоу — reset-ссылки — с той же механикой (хеш, expiry, одноразовость, FOR UPDATE). Правило «два адаптера = настоящий шов» здесь превращается в «три использования = пора извлекать»: при постройке reset-ссылок стоит вынести общий Eden.Tokens-паттерн (генерация/хеш/валидация/lock), оставив каждому контексту свою схему.

Worth exploring W3. Chat.ex: 117 функций — созревает выделение Threads

треды: ~9 публичных функций + ThreadMembership + денормализованные счётчики, слабая связь с остальным (только deliver_reply)

В отличие от ChatLive, chat.ex не горит: домены внутри когезивны, тесты идут через публичный интерфейс (38 describe-блоков по фичам). Но если файл продолжит расти, первым выделяется Eden.Chat.Threads — он уже почти автономен. Не делать превентивно; делать при следующей крупной тредовой фиче.

Worth exploring W4. blob_shared_outside? — O(n) запросов при массовом удалении

chat.ex:3230-3235 — один запрос на каждый ключ; GC разговора с 1000 сообщений = 1000+ запросов

Корректно, но не масштабно. Лечится одним батч-запросом (WHERE storage_key IN (...) AND id NOT IN (...)). Не срочно — станет заметно только на больших GC; зафиксировать как оптимизацию с известным рецептом.

Worth exploring W5. Подготовка Scope к org_id — дёшево сделать заранее

scope.ex — единственный конструктор for_user/1; расширение до %Scope{user, org} затрагивает только новые вызовы

Слот org_id в Scope (пусть пока nil) + привычка прописывать его в тестовых фикстурах снимет половину механики org-фазы. Согласуется с рекомендацией RFC §1 (Repo.prepare_query-бэкстоп — на самой org-фазе, не сейчас).

Worth exploring W6. Мелочи гигиены

accounts_test.exs:14-17 (инлайн-фикстура мимо контекста, дублирует support-фикстуру); контракты hooks (data-*) — только в render, без единого справочника

Обе — не production-риски: выровнять фикстуру на register_user_with_invite; контракты нагруженных хуков (SendQueue, SelectSync) описать одним комментарием-шапкой при ближайшем касании.

Findings — Speculative

Speculative P1. ChannelSurface: канальный режим как отдельная поверхность

~29 событий CRUD каналов/румов/участников/инвайтов (949–1228) — отдельный домен внутри ChatLive. Экстракция возможна, но режимы делят стрим и композер — риск выше, чем у тредов/селекта. Рассматривать только если канальные фичи продолжат расти (админ-панель эпика #172 может, наоборот, забрать часть этого в собственный LiveView).

Speculative P2. Компile-time защита Scope

Идея @enforce_scope-макроса (или Credo-чека), требующего Scope-аргумент у публичных функций контекстов. Прозаичная альтернатива — пункт в чеклист PR-ревью. Ценность вырастет вместе с org_id, когда забытый фильтр станет кросс-тенантной утечкой.

Speculative P3. Presence как переиспользуемый модуль

Presence/typing-кластер (7 assigns, ~10 обработчиков) ортогонален и мог бы стать attachable-хуком в духе RailHook — но пока он нужен только ChatLive, второго потребителя нет. Один адаптер = гипотетический шов; ждать реального второго потребителя.

Рекомендуемый порядок работ

  1. Централизовать видимость (S4) — маленький PR, закрывает единственный найденный путь к утечке данных.
  2. Панель тредов из ChatLive (W1) — первая экстракция, по лекалам album_layout/rail_hook; следом мультиселект.
  3. Reset-ссылки (ADR-0002) вместе с общим token-паттерном (W2) — фича и рефакторинг совпадают по месту.
  4. Eden.Notifications: Web-адаптер + централизация fan-out (S5) — первый шаг уже принятого ADR-0001, попутно снимает двойную логику получателей.
  5. Композер (W1, третий) — после того как треды/селект отработают механику экстракции.
  6. Слот org_id в Scope (W5) — перед стартом org-фазы эпика #172.

Пункты 1–3 самостоятельны и небольшие; 4–6 привязаны к уже запланированным фичам, чтобы рефакторинг ехал на фичевой работе, а не отдельным «стоп-миром».
